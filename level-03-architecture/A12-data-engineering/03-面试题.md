### 7. P 值与置信区间有什么区别？分别在 AB 实验中如何应用？

**参考答案**：

P 值的定义：在原假设为真的前提下，观察到当前结果或更极端结果的概率。P 值越小，拒绝原假设的证据越强。通常取阈值 0.05，P < 0.05 认为结果统计显著。

置信区间的定义：对总体参数的一个区间估计，表示在重复抽样中有 95% 的区间会包含真实参数值。例如实验组比对照组转化率提升 2%，95% CI 为 [1.2%, 2.8%]。

对比：
| 维度 | P 值 | 置信区间 |
|------|------|----------|
| 回答的问题 | 差异是否显著？ | 差异有多大？ |
| 提供的信息 | 一个概率值 | 效果范围 + 方向 |
| 业务决策价值 | 低（仅知有无差异） | 高（知差异大小和精度） |
| 对样本量敏感度 | 高（大样本下微小差异也显著） | 中等 |

```javascript
function decideExperiment(controlData, experimentData, alpha) {
    const pValue = calculatePValue(controlData, experimentData);
    const ci = calculateConfidenceInterval(controlData, experimentData, 0.95);
    if (pValue < alpha) {
        if (ci.lower > 0) return { decision: '全量上线', effect: ci.mean };
        else if (ci.upper < 0) return { decision: '回滚', effect: ci.mean };
    }
    return { decision: '继续实验或放弃' };
}
```

**常见错误**：
- 将 P 值理解为实验组比对照组好的概率——P 值不是效应概率。
- 只看 P 值忽略效应量，大样本下 0.1% 的提升也可能 P < 0.05，但业务无意义。
- 将置信区间误解为参数有 95% 概率落在区间内——实际是 95% 的置信区间会包含真值。
- 多重比较不做校正，对多个指标分别算 P 值导致假阳性膨胀。

**扩展追问**：

- 贝叶斯方法与频率学派方法在 AB 实验决策上有什么本质差异？什么时候该用贝叶斯？

**评分维度**：

### 8. 什么是数据管道（Data Pipeline）？包含哪些核心组件？

**参考答案**：

数据管道是从数据采集到分析应用的全链路架构，将原始数据经过采集、传输、处理、存储，最终转化为可用的分析结果。

核心组件：
1. **数据采集层**：前端埋点 SDK、服务端日志采集、DB binlog 监听、外部 API 接入。
2. **消息队列**：Kafka/Pulsar，高吞吐削峰填谷，解耦生产者和消费者。
3. **数据处理层**：实时处理（Flink/Spark Streaming）进行秒级计算，离线处理（Hive/Spark）进行 T+1 批量计算。
4. **数据存储层**：按分层模型组织——ODS（原始数据）、DWD（清洗明细）、DWS（汇总）、ADS（应用）。
5. **OLAP 引擎**：ClickHouse/Druid/Doris/StarRocks，支撑秒级多维查询。
6. **数据应用**：BI 看板（Superset/Grafana）、实时 API 服务、数据产品。
7. **元数据管理**：事件字典、指标口径、数据血缘追踪。
8. **数据质量监控**：完整性校验、延迟告警、异常检测。

**常见错误**：
- 直接让前端查询数仓——延迟高、压力大，应通过 API 网关封装。
- 忽略消息队列的缓冲作用，高峰期直接将数据写入数据库导致写入瓶颈。
- 缺乏元数据管理和数据血缘，数据问题排查困难。

**扩展追问**：
- Lambda 架构和 Kappa 架构的区别？实时数仓场景下如何选择？
- 如何设计数据回溯方案以应对埋点逻辑变更？

**评分维度**：
- 组件完整性（35%）
- 各组件职责（35%）
- 技术选型（30%）

### 9. 如何设计一套前端埋点 SDK 的完整架构？需要考虑哪些模块？

**参考答案**：

一个生产级前端埋点 SDK 需要从数据采集、处理、存储到上报全链路设计，兼顾性能、可靠性和可扩展性。

**1. 统一事件模型**：
```typescript
interface TrackEvent {
    eventType: string;
    timestamp: number;
    properties: Record<string, any>;
    pageUrl: string;
    eventId: string;
    userId?: string;
    anonymousId: string;
}
```

**2. 缓冲区与批量发送**：
采用环形缓冲区设计，设置阈值（如 10 条或 5 秒），避免高频事件阻塞主线程。

```javascript
class RingBuffer {
    constructor(capacity) {
        this.buffer = new Array(capacity);
        this.head = 0;
        this.tail = 0;
        this.size = capacity;
    }
    push(item) {
        this.buffer[this.tail] = item;
        this.tail = (this.tail + 1) % this.size;
        if (this.tail === this.head) {
            this.head = (this.head + 1) % this.size;
            return false;
        }
        return true;
    }
    drain() {
        const items = [];
        while (this.head !== this.tail) {
            items.push(this.buffer[this.head]);
            this.buffer[this.head] = null;
            this.head = (this.head + 1) % this.size;
        }
        return items;
    }
}
```

**3. 重试与容错**：指数退避策略（1s -> 2s -> 4s -> 8s -> 16s），最大重试 5 次。失败后写入 IndexedDB 做持久化。

**4. 离线支持**：通过 navigator.onLine 检测网络状态，离线时事件存入 IndexedDB，上线后批量回放。

**5. 采样策略**：按用户 ID 哈希做固定比例采样、按事件类型差异化采样（关键事件全采，非关键事件 1% 采样）。

**6. 性能保障**：使用 requestIdleCallback 做低优先级上报；sendBeacon 在页面卸载时保证数据不丢。

**常见错误**：
- 缓冲区无上限导致内存溢出——必须设置最大容量并支持溢出策略。
- 离线恢复时瞬间发送大量积压事件打爆服务端——需要限速回放。
- 忽视 SPA 路由切换时的页面上下文更新——导致所有事件 pageUrl 始终为首屏地址。

**扩展追问**：
- 如果要支持事件的实时性（如实时大屏），缓冲区设计需要做什么调整？
- 如何处理跨域 iframe 中的事件采集？postMessage 通信的序列化和安全性如何保证？

**评分维度**：
- 架构完整性（30%）
- 缓冲区与容错设计（25%）
- 离线与采样策略（25%）
- 性能优化（20%）

### 10. AB 实验中如何计算样本量？MDE 和统计功效是什么？

**参考答案**：

**样本量公式（两比例 Z 检验）**：
n = (Z_a/2 + Z_b)^2 x [p1(1-p1) + p2(1-p2)] / (p2 - p1)^2

其中 Z_a/2 对应显著性水平（a=0.05 时为 1.96），Z_b 对应统计功效（b=0.2 即功效 0.8 时为 0.84），p1 和 p2 分别为对照组和实验组的预期转化率。

```javascript
function calculateSampleSize(baselineRate, mde, alpha, power) {
    const zAlpha = normSInv(1 - alpha / 2);
    const zBeta = normSInv(power);
    const p1 = baselineRate;
    const p2 = baselineRate * (1 + mde);
    const pooledVariance = (p1 * (1 - p1) + p2 * (1 - p2));
    return Math.ceil(Math.pow(zAlpha + zBeta, 2) * pooledVariance / Math.pow(p2 - p1, 2));
}
// 示例：基线 10%，MDE=5%，alpha=0.05，power=0.8 -> 约 15000 人/组
```

**MDE（Minimum Detectable Effect）**：在给定样本量和显著性水平下，实验能够检测到的最小效应量。样本量越大，MDE 越小。MDE 不是想要检测的效果，而是受样本量限制能检测到的效果下限。

**统计功效（Statistical Power）**：当真实效应存在时，实验正确拒绝原假设的概率。通常要求 Power >= 0.8，即真实效应存在时有 80% 的概率能检测到。

**三者关系**：
- 固定 alpha 和 Power，效应越小需要的样本量越大。
- 固定 alpha 和样本量，效应越小 Power 越低（越容易漏报）。
- 实践中先确定 MDE，再反算所需样本量，最后验证能否在合理时长内收集到。

**常见错误**：
- 实验前不做样本量估算，跑了两天发现 P < 0.05 就直接全量——可能假阳性。
- 把 MDE 设得过小（如 0.1%），导致所需样本量远超实际能收集的量。
- 忽视新奇效应——实验前期效果异常好，稳定后回落；计算样本量时未考虑取稳周期。
- 没有考虑多个实验之间的流量分割——每组分到的样本不足。

**扩展追问**：
- 如果实验资源有限，只能在 10 万用户上跑 3 天，如何合理设定 MDE？
- 什么是序贯检验（Sequential Testing）？它如何解决提前偷看带来的假阳性问题？

**评分维度**：
- 样本量公式理解（30%）
- MDE 与 Power 概念（30%）
- 三者关系与应用（40%）

### 11. 如何设计一套分层指标体系？北极星、护栏、核心和探索指标的关系？

**参考答案**：

分层指标体系是将业务目标从战略层到执行层逐级拆解的结构化方法。

**四层体系**：
1. **北极星指标（North Star Metric）**：唯一、聚焦长期价值的顶层指标。例如 Airbnb 的预订过夜数、Spotify 的总收听时长。回答产品存在的终极价值是什么。
2. **护栏指标（Guardrail Metrics）**：防止策略负向影响核心体验。例如页面加载时间、报错率、用户投诉率。实验即使核心指标显著正向，护栏指标显著负向也不能全量。
3. **核心指标（Core Metrics / AARRR）**：获取、激活、留存、收入、推荐。
4. **探索指标（Exploration Metrics）**：深度诊断的细分指标，如按钮点击率、页面滚动深度。不参与实验决策，但帮助定位根因。

**设计原则**：
```
检查清单：
- 北极星指标是否可拆解为多个核心指标？
- 每个核心指标是否有明确的埋点？
- 护栏指标是否覆盖用户体验底线？
- 指标之间是否存在冲突（如转化率 vs 时长）？
```

**常见错误**：
- 指标层级太多，团队无法聚焦，每天看几十个指标无所适从。
- 将北极星指标设成不可直接衡量的概念（如用户幸福感），无法做实验评估。
- 缺少护栏指标，核心指标提升但用户体验严重下降（如强制弹窗提升点击率但降低满意度）。
- 指标高度相关但未做冗余清理，实验评估时多重共线性问题。

**扩展追问**：
- 如何验证北极星指标与商业结果之间存在因果关系？
- 在 OKR 体系中，指标体系如何与 KR 对齐？

**评分维度**：
- 指标分层逻辑（30%）
- 各层定义准确性（30%）
- 设计原则与实践（40%）

### 12. 如何设计和实现漏斗分析？涉及哪些核心概念？

**参考答案**：

漏斗分析是分析用户在多步骤流程中转化率的常用方法，用于注册、购物、订阅等场景。

**核心概念**：
1. **步骤定义**：将流程拆解为有序事件序列。如电商漏斗：访问首页 -> 搜索 -> 查看详情 -> 加购 -> 提交订单 -> 支付。
2. **转化率**：每一步相对于上一步的完成比例。stepN_CR = stepN_user / stepN-1_user x 100%。
3. **整体转化率**：最终步骤相对于第一步的转化比例。
4. **流失分析**：识别流失率最高的步骤，定位优化方向。
5. **转化时长**：用户从第一步到最后一步的平均耗时。

```typescript
interface FunnelStep {
    stepId: number;
    name: string;
    eventType: string;
    filters?: Record<string, any>;
}
interface FunnelResult {
    stepId: number;
    userCount: number;
    stepConversionRate: number;
    overallConversionRate: number;
    avgTimeFromPrevious: number;
}
```

```javascript
function calculateFunnel(events, funnelSteps, timeWindowMinutes) {
    const windowMs = timeWindowMinutes * 60 * 1000;
    const userEvents = groupBy(events, 'userId');
    const results = funnelSteps.map(s => ({
        ...s, userCount: 0, stepConversionRate: 0,
        overallConversionRate: 0, avgTimeFromPrevious: 0,
    }));
    for (const [userId, eventList] of Object.entries(userEvents)) {
        const sorted = eventList.sort((a, b) => a.timestamp - b.timestamp);
        let currentStep = 0;
        let stepTimestamps = [];
        for (const event of sorted) {
            if (currentStep < funnelSteps.length &&
                event.eventType === funnelSteps[currentStep].eventType) {
                if (currentStep === 0 ||
                    (event.timestamp - stepTimestamps[currentStep-1] <= windowMs)) {
                    stepTimestamps[currentStep] = event.timestamp;
                    currentStep++;
                }
            }
        }
        for (let i = 0; i < currentStep; i++) {
            results[i].userCount++;
            if (i > 0) results[i].avgTimeFromPrevious +=
                stepTimestamps[i] - stepTimestamps[i-1];
        }
    }
    // Calculate conversion rates...
    return results;
}
```

**常见错误**：
- 不同时间窗口（同一天 vs 同一次会话）的数据混在一起分析，结论失真。
- 忽略用户可能跨多个会话完成流程，时间窗口设置不合理。
- 漏斗步骤定义粒度过粗或过细。
- 只看整体转化率不逐层分析。

**扩展追问**：
- 如果用户可以在流程中回退（如修改购物车商品数量后再支付），漏斗模型如何处理这种非线性行为？
- 对比两个不同时期的漏斗时，如何判断某一步转化率差异是否统计显著？

**评分维度**：
- 概念完整性（25%）
- 计算方法正确性（30%）
- 实现逻辑（25%）
- 实际应用理解（20%）

### 13. 如何设计会话回放（Session Replay）系统？涉及哪些关键技术？

**参考答案**：

会话回放系统录制用户在页面上的所有交互行为，用于定性分析用户行为、发现可用性问题。

**系统架构**：
1. **DOM 录制层**：使用 MutationObserver 监听 DOM 变化（节点增删、属性变更、文本变更），同时监听鼠标移动、点击、滚动、输入。将所有变更序列化为操作日志。
2. **数据结构**（类似 RRR 协议）：{ type, timestamp, data: { adds, removes, attributes, text, inputValue, scrollX, scrollY } }
3. **序列化与压缩**：对 DOM 节点做增量快照（full + incremental），使用 lz-string 或 pako gzip 压缩。典型压缩率：原始 50-100KB 到 gzip 后 5-15KB。
4. **回放引擎**：解析事件流按时间戳重建 DOM 状态，使用 iframe + sandbox 隔离避免执行恶意脚本，支持快进、慢放、跳过空闲时间。
5. **隐私脱敏**：选择器级别脱敏（CSS 选择器配置屏蔽字段如 input[type=password]），文本节点替换为星号，img src 替换为占位图。

```javascript
class ReplayRecorder {
    start() {
        const fullSnapshot = this.captureFullSnapshot(document.body);
        this.events.push({ type: 'full_snapshot', timestamp: Date.now(), data: fullSnapshot });
        this.observer = new MutationObserver((mutations) => {
            const changes = this.processMutations(mutations);
            if (changes) this.events.push({ type: 'mutation', timestamp: Date.now(), data: changes });
        });
        this.observer.observe(document.body, { childList: true, subtree: true, attributes: true, characterData: true });
        setInterval(() => this.flush(), 5000);
    }
    flush() {
        if (this.events.length === 0) return;
        const compressed = pako.gzip(JSON.stringify(this.events));
        navigator.sendBeacon('/api/replay', compressed);
        this.events = [];
    }
}
```

**常见错误**：
- 未做隐私脱敏——回放中泄露用户密码、身份证号等敏感信息。
- 全量快照过于频繁，数据量爆炸（应全量 + 增量结合）。
- 未处理 shadow DOM，组件库内部行为无法回放。
- 回放引擎与录制时的 DOM 结构不一致——前端版本更新后旧回放异常。

**扩展追问**：
- 如何控制采样率以确保系统成本可控？
- 如何将会话回放与点击热图、注意力图结合构建立体用户洞察？

**评分维度**：
- 录制层设计（30%）
- 序列化与压缩（25%）
- 回放引擎（20%）
- 隐私与合规（25%）

### 14. 如何设计实时数据管道架构？涉及哪些核心组件？

**参考答案**：

实时数据管道在秒级延迟内将前端行为数据传输到存储和分析系统，支撑实时大屏、实时推荐、异常监控。

**端到端架构**：前端 SDK -> Edge/CDN -> 消息队列 -> 流计算引擎 -> OLAP/存储 -> 应用层

**各组件**：
1. **前端 SDK 发送层**：sendBeacon 发送，批量合并（5 秒或 10 条）；关键事件走即时通道，非关键走批量通道。
2. **Edge/CDN 接入层**：全球就近接入，做请求校验、速率限制和动态采样。
3. **消息队列**：Kafka/Pulsar，高吞吐百万条/秒，按 user_id hash 分区保证顺序，设置合理 TTL。
4. **流计算引擎**：Flink/Spark Streaming，常用 filter/map/window/join 算子。
   ```sql
   -- Flink SQL：5 分钟窗口计算 UV
   INSERT INTO uv_report SELECT page_url,
       COUNT(DISTINCT user_id) AS uv,
       TUMBLE_END(event_time, INTERVAL '5' MINUTE) AS window_end
   FROM user_behavior WHERE event_type = 'page_view'
   GROUP BY page_url, TUMBLE(event_time, INTERVAL '5' MINUTE);
   ```
5. **OLAP 存储**：ClickHouse/Doris/StarRocks 支持高并发实时导入；Redis 做实时计数器。
6. **应用层**：WebSocket 推送大屏、实时告警（错误率超 5% 自动告警）、实时个性化画像更新。

**常见错误**：
- 忽略消费端反压（backpressure）——高峰期延迟从秒级退化到分钟级。
- 消息队列 partition 数太少或太多。
- 未做实时管道延迟监控——管道挂了数小时无人知晓。
- OLAP 选型失误——高频写入场景用了不擅长的引擎。

**扩展追问**：
- Lambda 架构与 Kappa 架构的优缺点？当前推荐哪种？
- 如何保证实时管道 Exactly-Once 语义？哪些环节可能产生重复数据？

**评分维度**：
- 架构完整性（30%）
- 各组件选型与设计（30%）
- 流计算原理（20%）
- 端到端延迟与可靠性（20%）

### 15. 多触点归因模型有哪些？如何选择和应用？

**参考答案**：

多触点归因模型解决用户在转化前经过多个渠道时，如何合理分配转化功劳的问题。

**常见模型**：
1. **首次触点归因**：100% 功劳归第一个渠道。适合评估品牌曝光和拉新效果。
2. **末次触点归因**：100% 功劳归最后一个渠道。最简单，适合评估转化收割效果。
3. **线性归因**：功劳平均分配给所有触达渠道。
4. **时间衰减归因**：越接近转化权重越高（如 7 天半衰期）。适合短期营销活动。
5. **Shapley 值归因**：基于合作博弈论，考虑渠道在不同组合下的边际贡献。最公平但计算复杂度高。

| 业务目标 | 推荐模型 | 原因 |
|----------|----------|------|
| 评估拉新渠道 | 首次触点 | 看重获客能力 |
| 评估转化渠道 | 末次触点 | 看重收口能力 |
| 公平分配预算 | 线性或 Shapley | 全链路贡献 |
| 短期营销 | 时间衰减 | 近期效果为主 |

**常见错误**：
- 归因窗口设置不合理——长周期业务窗口太短遗漏早期渠道。
- 多渠道用户身份未统一——跨设备视为不同人，归因断裂。
- 使用多触点归因却不做 AB 实验验证——归因模型本质是假设。

**扩展追问**：
- MMM（媒体混合建模）与多触点归因有什么互补关系？
- 如何解决跨设备归因中的用户 ID 打通问题？

**评分维度**：
- 归因模型理解（30%）
- 模型对比与应用（30%）
- 实现与计算（20%）
- 业务场景匹配（20%）

### 16. 除了 AB 实验，还有哪些因果推断方法？DID、RDD 和合成控制法的应用场景？

**参考答案**：

当 AB 实验不可行时（伦理不允许、无法随机化、成本过高），使用观察数据的因果推断方法。

**1. 双重差分法（DID）**：
原理：比较处理组和对照组在政策前后的变化差异。核心假设是平行趋势假设——干预前两组趋势一致。
DID 估计量 = (处理组后均值 - 处理组前均值) - (对照组后均值 - 对照组前均值)
适用场景：某地区上线新功能对比其他地区。局限性：平行趋势不可直接验证。

**2. 断点回归设计（RDD）**：
原理：利用阈值附近个体被近似随机分配到处理组和对照组。
适用场景：排名前 1000 用户获得权益、18 岁解锁功能。
局限性：需要大量数据集中在阈值附近，只能推断阈值附近效应（LATE）。

**3. 合成控制法（Synthetic Control）**：
原理：用多个对照单元的加权组合构建合成处理单元，模拟反事实。
适用场景：只有一个处理单元（一个国家、一个产品线），如评估版本更新对留存的因果效应。

**常见错误**：
- 使用 DID 未检验平行趋势假设。
- RDD 带宽选择主观导致结论相反。
- 合成控制对照单元太少导致合成效果差。
- 将所有方法视为 AB 实验的备胎——每种方法有自己的识别假设。

**扩展追问**：
- 工具变量法（IV）在什么场景下比 DID 更适用？
- 什么是辛普森悖论？在数据分析中如何发现和处理？

**评分维度**：
- DID 原理与应用（30%）
- RDD 原理与应用（30%）
- 合成控制法原理（20%）
- 方法对比与选型（20%）

### 17. 如何设计大规模实时实验平台？如何处理重叠实验和网络效应？

**参考答案**：

大规模实验平台需要支撑数百个实验同时运行，且实验之间可能产生相互干扰。

**分层流量模型**：将流量划分为多个正交层（Layer），每层独立抽取实验组，通过哈希正交算法保证互不干扰。

```
层 1（推荐算法）：实验 A（20%）、实验 B（20%）、对照（60%）
层 2（UI 样式）：实验 C（50%）、对照（50%）
层 3（文案）：实验 D（33%）、实验 E（33%）、对照（34%）
```

**重叠实验处理**：不同层视为正交。同层实验需检测交互效应——用 Two-way ANOVA 看 A x B 交互项是否显著。

**网络效应处理**：社交产品中用户相互影响，违反 SUTVA 假设。
解决方案：a）群体随机化——以社交群体为单位随机分流；b）网络分簇——用 Louvain 算法划分社交图谱，以簇为单位实验；c）概率化隔离——实验组用户只与实验组交互。

**统计校正**：多重比较用 FDR（Benjamini-Hochberg）控制假阳性。序贯检验允许随时观察而不增加假阳性率。

**平台监控**：实验前检查样本量和护栏指标；运行中监控分流均匀性和 SRM（Sample Ratio Mismatch）；结束自动决策。

**常见错误**：
- 不同层共享分桶逻辑导致层间干扰。
- 未检测交互效应就将多个实验全量上线——单独正向，合在一起负向。
- 社交产品用个体随机化——实验组给对照组点赞导致效果稀释。
- 网络效应场景只聚类标准误但不做群体随机化。

**扩展追问**：
- 如何量化网络效应强度？实验前如何预估偏误大小？
- 双边市场（如广告排序）中实验设计有什么特殊考虑？

**评分维度**：
- 分层流量模型（30%）
- 重叠实验处理（25%）

### 18. 如何设计数据质量保障体系？有哪些关键环节和监控手段？

**参考答案**：

数据质量保障（DQA）确保数据采集、传输、加工、应用全链路准确、完整、及时。

**六层模型**：
1. **采集层**：Schema 校验（字段缺失、类型错误、枚举非法时记录异常），端到端测试（Playwright 模拟操作验证埋点），埋点覆盖率检查。
2. **传输层**：数据延迟监控（p50/p95/p99），数据丢失率监控（对比 SDK 发包量与服务端收到量），重复率监控（事件 ID 去重）。
3. **存储层**：按日期分区，完整性校验（上游条数 = 下游条数 + 异常数 + 重复数），Schema Evolution 管理。
4. **加工层**：ETL 后行数、空值率、分布对比校验，指标一致性校验（Single Source of Truth），回归测试。
5. **应用层**：报表数据突变告警，用户反馈闭环（数据报错按钮自动生成工单），核心指标与财务系统对账。
6. **治理流程层**：埋点审批流程（明确 owner 和定义），DQ Score = (1 - 问题事件数/总事件数) x 100，目标 > 99.9%，季度审计。

**常见错误**：
- 只关注采集层准确，忽略传输和加工层的数据丢失。
- 无自动化监控，全靠人工核对。
- 埋点文档与实际脱节——代码改了但文档没更新。
- Schema 变更破坏下游 ETL。

**扩展追问**：
- 如何平衡数据准确与及时？核心指标高精度可延迟，实时指标低延迟容忍偏差，如何分层？
- 数据湖和数据仓库在质量治理上有什么不同挑战？

**评分维度**：
- 分层保障体系完整性（30%）
- 各层级具体手段（30%）
- 自动化与监控（20%）
- 治理流程（20%）

### 19. 在观测数据中如何进行因果推断？倾向得分匹配和工具变量法怎么用？

**参考答案**：

当无法随机化实验时，观测数据存在混淆偏差——哪些用户进入处理组并非随机。

**倾向得分匹配（PSM）**：
原理：倾向得分 e(X) = P(T=1|X)，通过将倾向得分相近的个体匹配模拟随机化。

步骤：
1. 训练 Logistic 回归预测倾向得分。
2. 为每个样本预测得分。
3. 最近邻匹配（1:1，caliper=0.05）。
4. 计算 ATT（处理组平均处理效应）。

注意事项：不能包含处理变量本身或其结果；需要共同支撑域检查；匹配后做均衡性检验（标准化差异 < 0.1）。

**工具变量法（IV）**：
原理：工具变量 Z 满足：1）与处理 T 强相关；2）外生性（Z 与误差项不相关）；3）排他性（Z 只通过 T 影响 Y）。

2SLS：
- 第一阶段：T = a0 + a1*Z + d*X + e1，得 T_hat
- 第二阶段：Y = b0 + b1*T_hat + d*X + e2，b1 即 IV 估计

应用场景：随机发放优惠券（Z）-> 用户使用优惠券（T）-> 用户消费（Y）。

**常见错误**：
- 弱工具变量（第一阶段 F < 10），IV 估计有偏。
- 排他性条件被违反（Z 通过 T 以外路径影响 Y）。
- PSM 协变量太多导致共同支撑域太小。
- 混淆因果与相关——PSM 只能控制可观测混淆，不可观测混淆无能为力。

**扩展追问**：
- 敏感性分析在因果推断中的作用？E-value 如何评估不可观测混淆的影响？
- 什么是辛普森悖论？在实际数据中如何发现和处理？

**评分维度**：
- PSM 原理与应用（30%）
- IV 原理与应用（30%）
- 方法对比与选型（20%）
- 局限性与诊断方法（20%）
