## 一、基础题

### 1. 什么是 Serverless？它和传统服务器架构的主要区别是什么？

**评分维度**：
- 定义准确性（30%）
- 与传统架构对比（30%）
- 优缺点分析（40%）

**参考答案**：

Serverless 是一种云计算执行模型，开发者编写并部署函数代码，云平台自动管理服务器实例、运行时环境和弹性伸缩。开发者只需关注业务代码。

与传统服务器架构的核心区别：

| 维度 | Serverless | 传统服务器 |
|------|-----------|-----------|
| 运维 | 零运维，平台管理 | 需要运维团队管理服务器 |
| 弹性伸缩 | 自动从 0 到无限 | 需预先规划容量 |
| 计费 | 按调用次数和执行时长 | 按固定资源包月/包年 |
| 冷启动 | 存在冷启动延迟 | 无冷启动 |
| 执行时间 | 有限制（通常 1-15 分钟） | 无限制 |
| 状态 | 无状态（推荐） | 可有状态 |

**常见错误**：
- 误以为 Serverless 是"没有服务器"，实际上服务器依然存在，只是由平台管理。
- 忽略冷启动对生产环境的影响。
- 认为 Serverless 适合所有场景，实际上长时间任务和 WebSocket 不适合。

**扩展追问**：
- Serverless 如何实现自动伸缩？
- 你如何评估一个应用是否适合迁移到 Serverless？

### 2. Edge 计算是什么？它适合哪些前端场景？

**评分维度**：
- 基本概念（30%）
- 场景覆盖（40%）
- 场景原理解释（30%）

**参考答案**：

Edge 计算是将计算资源部署在 CDN 边缘节点上，在离用户最近的位置处理请求，从而显著降低延迟。

适合的前端场景：
1. **边缘 SSR**：在边缘节点渲染页面首屏，减少 TTFP。
2. **A/B 测试和灰度发布**：在边缘层根据请求特征分配不同版本。
3. **地理位置个性化**：根据用户 IP 位置返回本地化内容。
4. **安全网关**：在边缘层拦截恶意请求、DDoS 防护、IP 黑名单。
5. **请求改写和重定向**：URL 重写、移动端/桌面端重定向。
6. **缓存策略增强**：基于 Cookie/Header 的个性化缓存失效。
7. **JAMStack 站点动态化**：为静态站点添加动态 API 代理。

**常见错误**：
- 将 Edge 计算等同于 CDN 缓存——Edge 计算是动态执行代码，CDN 只是缓存静态资源。
- 忽略 Edge 函数的执行限制（CPU、内存、时长）。

**扩展追问**：
- Edge 函数和浏览器端渲染有什么区别？
- 什么场景不适合使用 Edge 计算？

### 3. 什么是冷启动？Serverless 冷启动的原因和缓解方案？

**评分维度**：
- 冷启动原因解释（40%）
- 缓解方法（40%）
- 不同平台差异（20%）

**参考答案**：

冷启动是指 Serverless 函数在长时间未被调用后，平台回收实例。当新请求到达时，平台需要重新加载代码、初始化运行时环境，这个过程带来的额外延迟。

**冷启动过程**：
```
请求到达 -> 平台发现无可用实例
  -> 下载代码包 -> 启动运行时
  -> 加载依赖 -> 执行 Handler 外部代码
  -> 执行 Handler -> 响应
```

**缓解方法**：
1. **预留并发（Provisioned Concurrency）**：AWS Lambda 等平台支持，保持指定实例始终 warm。
2. **减少代码体积**：Tree Shaking 移除无用代码，使用轻量依赖。
3. **使用 Edge 函数**：V8 Isolate 冷启动在微秒级，远快于容器。
4. **Warm-up 策略**：定时调用函数保持活跃（如每 5 分钟发一次请求）。
5. **优化初始化代码**：数据库连接、配置加载移到 Handler 外部复用。
6. **选择更快的运行时**：如使用 Node.js 而非 Python/Java。

**常见错误**：
- 以为冷启动只发生在第一次部署后——实际上平台会在一段时间不活跃后回收实例。
- 忽略依赖加载时间：一个 50MB 的 node_modules 会显著增加冷启动。

**扩展追问**：
- 冷启动在 AWS Lambda、Cloudflare Workers、Vercel Edge Functions 之间有什么差异？
- 如何监控和告警冷启动问题？

### 4. 什么是 JAMStack？它和 Serverless / Edge 有什么关系？

**评分维度**：
- JAMStack 定义和组成部分（30%）
- 与 Serverless 的关系（30%）
- 优势与局限（20%）
- 代码示例（20%）

**参考答案**：

JAMStack 代表 JavaScript、API 和 Markup，是一种现代 Web 架构模式。它在构建时预渲染静态 Markup，通过 JavaScript 和 API 实现动态功能。

**与 Serverless / Edge 的关系**：
- 静态页面部署在 CDN 边缘，由 Edge 快速分发。
- 动态功能通过 Serverless 函数（API）实现。
- Edge 函数可在边缘层处理表单提交、认证、个性化等逻辑。

**常见错误**：
- 以为 JAMStack 完全不需要服务器——API 层仍然运行在 Serverless 或传统服务器上。
- 忽略动态内容过多时 JAMStack 的优势会减弱。

**扩展追问**：

### 5. 对比 Vercel Edge Functions、Cloudflare Workers、AWS Lambda@Edge 的异同。

**评分维度**：
- 运行时差异（30%）
- 生态和集成（30%）
- 性能和限制（20%）
- 适用场景推荐（20%）

**参考答案**：

| 维度 | Vercel Edge Functions | Cloudflare Workers | AWS Lambda@Edge |
|------|----------------------|-------------------|----------------|
| 运行时 | V8 Isolate (JavaScript) | V8 Isolate (JS/Rust/Wasm) | Node.js / Python |
| 冷启动 | 微秒级 | 微秒级 | 毫秒级 |
| 节点数 | ~100 | 330+ | 400+ (CloudFront) |
| 持久存储 | Edge Config | KV + D1 + R2 + Queues | DynamoDB + S3 |
| 最大执行时间 | 30 秒 | 30 秒（付费版） | 5-30 秒 |
| 生态绑定 | Next.js / Vercel | Cloudflare 生态 | AWS 全家桶 |
| 适用场景 | Next.js SSR/ISR | 网关/KV/通用边缘 | 已有 AWS 基础设施 |

**选型建议**：
- **Vercel Edge**：团队使用 Next.js，需要快速 SSR 开发。
- **Cloudflare Workers**：需要全球覆盖、丰富的边缘存储、高性能。
- **Lambda@Edge**：已有 AWS 基础设施，需要边缘与现有服务集成。

**常见错误**：
- 忽略 V8 Isolate 环境中 API 受限，不能使用 Node.js 所有模块。
- 低估平台锁定风险——换平台的迁移成本可能很高。

**扩展追问**：
- 如果让你设计一个跨平台的 Edge 函数抽象层，你会怎么做？
- Wasm 在 Edge 计算中有什么应用前景？

## 二、进阶题

### 6. 如何在 Edge 上实现 SSR（Server-Side Rendering）？架构如何设计？

**评分维度**：
- SSR 流程理解（30%）
- Edge SSR 架构设计（30%）
- 缓存和性能优化（20%）
- 代码实现（20%）

**参考答案**：

**Edge SSR 架构流程**：
```
用户请求页面 -> CDN Edge 节点
  -> 缓存命中 -> 直接返回 HTML
  -> 缓存未命中 -> Edge Function 执行 SSR
    -> 1. 解析请求 URL 和参数
    -> 2. 获取页面数据（KV/API）
    -> 3. 渲染组件为 HTML
    -> 4. 注入数据 + 客户端脚本
    -> 5. 返回 HTML + 缓存到 CDN
```

**优化策略**：
1. Streaming SSR：使用 Suspense 分块发送 HTML，提升首屏体验。
2. 选择性缓存：对匿名用户缓存页面，登录用户单独处理。
3. 数据预取：在 Edge 层预先获取常用数据，减少源头压力。

**常见错误**：
- 在 Edge 函数中执行大量同步计算，阻塞事件循环。
- 忽略 Edge 环境的内存限制，渲染大型页面时 OOM。

**扩展追问**：
- Edge SSR 和传统 Node.js SSR 在性能上有多大差距？
- 如何在 Edge SSR 中处理国际化（i18n）？

### 7. 如何在边缘层实现用户认证和授权？

**评分维度**：
- 认证流程设计（30%）
- Token 验证机制（30%）
- 缓存和性能策略（20%）
- 安全性考量（20%）

**参考答案**：

**推荐方案：边缘无状态 JWT 认证**

用户登录时在中心服务器验证身份，签发 JWT。后续请求由 Edge 函数拦截并验证 Token。

Edge 函数验证流程：
1. 提取 Authorization Header 中的 Token。
2. 使用缓存的 JWKS 验证签名。
3. 检查过期时间和黑名单（KV 存储）。
4. 通过则转发请求（注入用户信息），失败则返回 401。

**安全考量**：
- 使用短生命周期 Access Token（15 分钟）配合 Refresh Token。
- 签名密钥定期轮换。
- 敏感操作必须回源验证。
- 使用 HTTPS 防止 Token 泄露。

**常见错误**：
- 在 Edge 函数中直接验证用户名密码——每次都要回源数据库，丧失延迟优势。
- Token 黑名单 TTL 过长，已撤销 Token 仍被接受。

**扩展追问**：
- Edge 环境下如何实现 OAuth2 / OIDC 流程？
- 如何设计跨多个边缘区域的 SSO 方案？

### 8. 全球部署的边缘应用如何设计数据存储和同步策略？

**评分维度**：
- 数据分层设计（25%）
- 同步策略（25%）
- 一致性和延迟权衡（25%）
- 故障恢复（25%）

**参考答案**：

**三层数据策略**：

Layer 1 - Edge 缓存（Cache API / KV）：
- 存储静态数据：页面缓存、配置、字典。
- TTL：分钟级，最终一致即可。

Layer 2 - 区域存储（Region DB）：
- 存储用户区域数据：本地化内容、区域配置。
- 跨区域异步复制，最终一致。

Layer 3 - 全局存储（Global DB）：
- 存储全局数据：用户账户、产品目录。
- 强一致性（主区域写入 + 异步复制）。

**同步策略**：
1. Cache-Aside：边缘读取缓存，未命中则回源。
2. Write-Through：先写主区域，异步同步到其他区域。
3. Stale-While-Revalidate：返回旧数据的同时后台刷新。

**常见错误**：
- 所有区域都设为可写导致写冲突，应使用主区域写入模式。
- 忽略边缘缓存和数据库之间的数据一致性窗口。

**扩展追问**：
- 如何设计一个全球唯一的 ID 生成器？
- Edge 架构下如何处理分布式事务？

### 10. 如何在边缘层实现 A/B 测试和灰度发布系统？

**评分维度**：
- 系统设计（30%）
- 流量分配算法（20%）
- 指标采集（20%）
- 实时切换能力（15%）
- 代码实现（15%）

**参考答案**：

**系统设计**：
1. 实验配置存储在分布式 KV 中（流量比例、版本定义、状态）。
2. Edge 函数读取请求特征，根据规则分配版本。
3. 一致性哈希确保同一用户一致性。
4. 在响应头中注入实验标识用于客户端采集。

**指标采集**：
- 曝光：Edge 函数写入支持批量异步写入。
- 转化：客户端上报数据到分析平台，关联实验标识。
- 实时切换：更新 KV 配置即可实时调整流量比例。

**常见错误**：
- 忽略用户一致性——同一用户在不同请求中应分配到同一版本。
- 实验配置变化时没有重新分配策略，导致流量倾斜。

**扩展追问**：
- 如何确保 A/B 测试的统计显著性？
- 多实验同时运行时的流量冲突如何解决？

### 11. ISR、SSR、SSG 在 Edge 架构中的区别和适用场景？

**评分维度**：
- 三种模式定义和原理（30%）
- 对比分析（30%）
- 适用场景判断（20%）
- Edge 优化（20%）

**参考答案**：

| 维度 | SSG | SSR | ISR |
|------|-----|-----|-----|
| 渲染时机 | 构建时 | 请求时 | 构建时 + 按需更新 |
| 内容新鲜度 | 构建时更新 | 实时 | 按需更新（最近缓存） |
| Edge 适用性 | CDN 直接分发 | Edge SSR 渲染 | Edge 触发重新生成 |
| 性能 | 最快 | 中等 | 快 |
| 适用场景 | 博客、文档 | 个性化页面 | 电商产品页、CMS |

**Edge 优化策略**：
- SSG + Edge：预构建页面 + Edge 注入个性化内容。
- SSR + Edge：边缘节点渲染 + Streaming SSR。
- ISR + Edge：Edge 触发 revalidate，按需更新缓存。

**常见错误**：
- 非此即彼的选择——最佳实践是混合使用。
- ISR revalidate 时间过短导致回源压力增大。

**扩展追问**：
- Next.js 中 ISR 的 revalidate 机制如何在 Edge 层实现？
- 如何设计自定义的 ISR 缓存失效策略？

### 12. 边缘函数的可观测性如何实现？

**评分维度**：
- 日志策略（25%）
- 监控指标（25%）
- 分布式追踪（25%）
- 告警诊断（25%）

**参考答案**：

**日志策略**：结构化 JSON 日志，包含 requestId、path、method、status、duration、region。异步写入避免阻塞请求路径。

**监控指标**：
- 请求指标：QPS、P50/P95/P99 延迟、错误率。
- 业务指标：冷启动率、缓存命中率、回源率。
- 平台指标：CPU 时间、内存使用、超时次数。

**分布式追踪**：使用 W3C Trace Context 标准传递 Trace ID，Edge 函数注入 traceparent 头。

**告警阈值建议**：
- P95 延迟 > 500ms
- 错误率 > 1%
- 冷启动率 > 5%
- 回源率 > 30%

**常见错误**：
- 在请求路径中同步等待日志写入，增加用户感知延迟。
- 忽略内存限制导致日志缓冲溢出。

**扩展追问**：
- Edge 函数中如何调试内存泄漏？
- 如何区分 Edge 层还是后端导致的延迟？

### 13. WebAssembly 在 Edge 计算中有什么应用？

**评分维度**：
- WebAssembly 基础（30%）
- 应用场景（30%）
- 集成方式（20%）
- 性能对比（20%）

**参考答案**：

**Wasm 在 Edge 中的优势**：
- 接近原生的执行性能，适合计算密集型任务。
- 多语言支持（Rust、C/C++、Go、Zig 编译为 Wasm）。
- 安全的沙箱环境，极快冷启动。

**应用场景**：
1. 图片处理：压缩、裁剪、格式转换。
2. 数据压缩：Brotli、zstd 在边缘压缩响应。
3. 加密/解密：TLS 卸载、内容加密。
4. 模板渲染：高性能模板引擎。
5. 轻量 ML 推理：ONNX 在边缘推理。

**常见错误**：
- 将所有逻辑都用 Wasm 实现——Wasm 适合计算密集，I/O 密集用 JS。
- 忽略 Wasm 模块加载时间——应预加载或缓存编译实例。

**扩展追问**：
- Wasm 和 JS 在 Edge 函数中的性能边界在哪里？

### 14. 设计一个多云/多平台的边缘架构，避免供应商锁定。

**评分维度**：
- 抽象层设计（25%）
- 平台适配（20%）
- 数据迁移策略（20%）
- 运维和部署（20%）
- 成本考量（15%）

**参考答案**：

**抽象层设计**：

```
应用代码
    │
┌──────────────────┐
│  Edge 抽象层      │
├──────────────────┤
│ Handler 适配器    │
│ KV 适配器         │
│ Cache 适配器      │
│ 请求/响应 适配器   │
└──────────────────┘
    │
┌────┴────┬────┐
│CF Workers│Vercel│Lambda@Edge
```

**核心策略**：
1. 使用标准 Web API（Fetch、Request、Response）。
2. 平台特定功能（KV、Cache）封装为抽象接口。
3. 核心逻辑与平台配置分离，通过适配器注入。
4. 每个平台独立 CI/CD 流水线。

**常见错误**：
- 过度抽象导致性能损失，放弃平台特有优化。
- 忽略各平台之间的功能差异（如 KV 一致性模型不同）。

### 15. 设计一个边缘原生（Edge-Native）应用的前端架构。

**评分维度**：
- 边缘原生设计原则（25%）
- 组件拆分策略（25%）
- 数据流设计（20%）
- 离线策略（15%）
- 性能目标（15%）

**参考答案**：

**边缘原生设计原则**：
1. **静态优先**：尽可能在构建时生成，运行时只处理动态部分。
2. **边缘计算**：动态逻辑推到边缘节点执行，减少回源。
3. **增量更新**：只更新变化的部分，避免全量部署。
4. **离线优先**：客户端具备离线能力，Edge 提供同步协调。

**架构示例**：
```
┌─────────────────────────────────────────┐
│             客户端 (PWA)                 │
│  Service Worker 缓存 + 离线支持          │
│  CRDT 本地状态                           │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│           Edge 层 (CDN + Edge Functions) │
│  ├── 静态资源分发 (SSG + ISR)            │
│  ├── API Gateway + 鉴权                  │
│  ├── 个性化渲染 (Edge SSR)              │
│  ├── 数据同步 (WebSocket / Durable Obj) │
│  └── A/B 测试 + 灰度                     │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│          后端服务 + 数据层               │
│  ├── Serverless APIs                    │
│  ├── 全局 KV + 区域 DB                  │
│  └── 事件队列 + 异步处理                 │
└─────────────────────────────────────────┘
```

**性能目标**：
- 首屏加载 < 500ms（全球 P95）
- 缓存命中率 > 90%
- 动态 API < 100ms（P95）
- 离线状态下核心功能可用

**常见错误**：
- 仍用传统服务器思维设计——应充分利用静态生成和边缘执行。
- 忽略离线场景，退化到完全不可用。

**扩展追问**：
- 边缘原生应用如何做 SEO？
- Service Worker 和 Edge Worker 如何协同工作？

### 16. 如何在边缘架构中实现实时功能（WebSocket、SSE、实时协作）？

**评分维度**：
- 实时通信方案选型（25%）
- 边缘 WebSocket 架构（25%）
- 状态同步（25%）
- 扩展性和可靠性（25%）

**参考答案**：

**方案对比**：

| 方案 | 适用场景 | Edge 支持 |
|------|---------|-----------|
| WebSocket | 双向实时通信、协作编辑 | Cloudflare Durable Objects |
| SSE（Server-Sent Events） | 服务端推送、通知 | Edge 函数支持 Streaming |
| 短轮询 | 简单实时更新 | 兼容所有 Edge 平台 |
| WebRTC | P2P 实时通信 | 信令可在 Edge 层处理 |

**边缘 WebSocket 架构（基于 Durable Objects）**：
```
用户 A (东京)                用户 B (纽约)
    │                           │
    ▼                           ▼
WebSocket ──► Durable Object ──► WebSocket
              (全局协调器)
                  │
                  ▼
            持久化 (D1 / KV)
```

**实现要点**：
1. 使用 Durable Objects 作为 WebSocket 协调器，确保全局状态一致性。
2. 连接断开时使用 Stored State 自动恢复。
3. 消息使用 CRDT 算法（如 Yjs）实现无冲突同步。
4. 房间/频道概念隔离不同实时会话。

**常见错误**：
- 直接用传统 WebSocket 服务器而非 Edge 原生方案，丧失全球覆盖优势。
- 忽略边缘节点间状态同步的延迟和一致性。

### 17. Serverless / Edge 架构下的成本优化策略。

**评分维度**：
- 计费模型理解（25%）
- 成本构成分析（25%）
- 优化策略（30%）
- 成本监控（20%）

**参考答案**：

**成本构成分析**：

| 成本项 | Serverless | Edge Functions |
|--------|-----------|----------------|
| 计算 | 按调用次数 x 执行时长 | 按请求次数 + CPU 时间 |
| 数据传输 | 出站流量 | 出站流量 + 边缘节点间传输 |
| 存储 | 按存储量 + 读写操作 | KV 读/写操作次数 |
| 额外服务 | API Gateway、日志等 | Workers Unbound、D1 等 |

**优化策略**：

1. **缓存优化**：
   - 提高缓存命中率（合理 TTL、Cache-Aside 模式）。
   - 边缘缓存减少回源请求和计算执行。

2. **代码优化**：
   - 减少函数执行时间（算法优化、并行处理）。
   - 合并相关逻辑到同一函数，减少跨函数调用。

3. **架构优化**：
   - 静态内容使用 SSG，避免动态生成。
   - 冷数据迁移到低成本存储（S3 Glacier / R2）。
   - 使用预留实例降低高频调用成本。

4. **监控和告警**：
   - 设置每日/每月预算上限。
   - 监控异常调用峰值（如爬虫导致费用飙升）。
   - 分析调用模式，优化资源分配。

**示例**：Vercel/Cloudflare 成本对比表
| 场景 | Vercel Pro ($20/月) | Cloudflare Workers ($5/月起) |
|------|--------------------|------------------------------|
| 小型博客 | 免费额度够用 | 免费额度够用 |
| 中型电商 | Hobby + 超出费用 | 按请求付费，通常更便宜 |
| 高流量 API | Pro 可能超限 | Workers Unbound 更划算 |

**常见错误**：
- 只关注计算费用，忽略数据传输和存储操作费用。
- 没有设置预算告警，被异常流量导致意外账单。

**扩展追问**：
- 如何预测 Serverless 架构的月度成本？
- 在流量波动大的场景中，Serverless 相比固定服务器何时更划算？

### 18. 设计一个从传统单体架构到 Edge 架构的迁移方案。

**评分维度**：
- 迁移阶段划分（25%）
- 技术选型和重构策略（25%）
- 风险控制（20%）
- 性能验证（15%）
- 团队组织（15%）

**参考答案**：

**六阶段迁移路线图**：

**阶段一：评估和规划（2-4 周）**
- 分析现有应用架构和依赖关系。
- 识别适合迁移的模块和需保留的模块。
- 确定目标 Edge 平台（Vercel / Cloudflare / AWS）。
- 制定性能基准（延迟 P95、QPS、错误率）。

**阶段二：基础设施搭建（2-3 周）**
- 配置 CDN 和边缘节点。
- 搭建 CI/CD 流水线（支持蓝绿部署）。
- 配置全局 KV 存储和分布式数据库。
- 建立可观测性（日志、指标、追踪）。

**阶段三：静态资源迁移（2-4 周）**
- 将静态资源（图片、CSS、JS）迁移到 CDN / 对象存储。
- 前端应用迁移到 SSG / ISR。
- 配置 CDN 缓存策略。

**阶段四：无状态 API 迁移（4-8 周）**
- 拆分无状态 API 为 Serverless 函数。
- 在 Edge 层配置 API Gateway（鉴权、限流、路由）。
- 使用灰度发布逐步切流（5% -> 25% -> 50% -> 100%）。
- 每阶段验证性能和正确性。

**阶段五：有状态服务改造（4-8 周）**
- Session 外部化到 Redis / KV 存储。
- 数据库从单体迁移到分布式 / 全局数据库。
- 文件上传迁移到对象存储。
- WebSocket 迁移到 Edge 原生方案（Durable Objects）。

**阶段六：边缘层深度优化（持续）**
- 将适合的处理逻辑下移到 Edge 函数。
- 优化冷启动和缓存策略。
- 全局多区域部署和延迟优化。
- 成本优化。

**风险控制矩阵**：

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 迁移期间性能下降 | 中 | 高 | 灰度发布 + 即时回滚 |
| 数据一致性 | 中 | 高 | 双写 + 校验 + 停机窗口 |
| 团队技能不足 | 高 | 中 | 培训 + 外部顾问 + POC |
| 成本超出预期 | 中 | 中 | 严格监控 + 预算告警 |

**常见错误**：
- 大爆炸式迁移——应该增量迁移，逐步验证。
- 忽略遗留系统的运维需求，迁移期间两套系统并行。
- 低估 Session 改造的复杂性。

**扩展追问**：
- 迁移期间如何保持新旧系统数据一致？
- 如何在不中断服务的情况下切换 DNS 指向 Edge 平台？
