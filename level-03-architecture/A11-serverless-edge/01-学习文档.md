# A11 Serverless / Edge 架构

> 目标：掌握 Serverless 和边缘计算在前端架构中的应用，构建低延迟、高可用的现代 Web 应用。

---

## 核心要点（TL;DR）

- Serverless 让开发者专注业务逻辑，无需管理服务器。
- Edge 计算将代码推到离用户最近的节点执行，降低延迟。
- 前端与边缘函数结合，可实现 SSR、A/B 测试、个性化、安全网关。
- 边缘架构需要考虑冷启动、状态管理、调试和供应商锁定。

---

## 1. Serverless 基础

### 1.1 核心概念

- **FaaS**：函数即服务，按需执行。
- **BaaS**：后端即服务，如认证、数据库、存储。
- **事件驱动**：HTTP 请求、队列消息、定时触发。

### 1.2 优势

- 按需付费，无闲置成本。
- 自动扩缩容。
- 快速上线。

### 1.3 劣势

- 冷启动延迟。
- 执行时长限制。
- 调试和本地开发复杂。
- 供应商锁定。

---

## 2. Edge 计算

### 2.1 什么是 Edge

- 在 CDN 边缘节点运行代码，靠近用户。
- 代表平台：Cloudflare Workers、Vercel Edge Functions、AWS Lambda@Edge。

### 2.2 Edge 适用场景

- 边缘 SSR / 渲染降级。
- A/B 测试和灰度分流。
- 地理位置相关的个性化内容。
- 安全：WAF、Bot 检测、请求改写。
- 边缘缓存和重定向。

### 2.3 Edge 限制

- 运行时长有限（通常毫秒级）。
- 内存和 CPU 受限。
- 不适合长时间计算或复杂状态。

---

## 3. 前端 + Edge 架构模式

### 3.1 边缘渲染

```
用户请求 → Edge 函数 → 获取数据/渲染 HTML → 返回页面
```

### 3.2 边缘网关

- 鉴权、限流、路由、改写请求。
- 减轻源站压力。

### 3.3 个性化与实验

- 边缘根据用户特征返回不同版本。
- 结合 CDN 缓存策略。

---

## 4. 技术选型

| 平台 | 特点 | 适用 |
|------|------|------|
| Vercel Edge | Next.js 集成好 | SSR、A/B 测试 |
| Cloudflare Workers | 全球节点多、性能强 | 网关、缓存、Workers |
| AWS Lambda@Edge | 与 AWS 生态集成 | 已有 AWS 架构 |
| Deno Deploy | TypeScript 原生 | 现代 Edge 应用 |

---

## 5. 最佳实践

- 边缘函数保持轻量，避免复杂逻辑。
- 使用边缘缓存减少回源。
- 注意冷启动和超时限制。
- 日志和监控接入可观测平台。
- 设计供应商无关的抽象层，降低锁定风险。

---

## 常见误区

| 误区 | 正确理解 |
|------|---------|
| Serverless 完全无服务器 | 只是不需要管理服务器 |
| Edge 适合所有场景 | 受限于执行时长和资源 |
| 边缘函数可以替代后端 | 复杂业务仍需传统后端 |
| 忽略冷启动 | 冷启动是 Serverless 的重要考量 |
