### 11. 如何设计一个完整的 CSP 策略？请给出从开发到上线的完整工作流。

**参考答案**：

开发到上线的 CSP 完整工作流：

1. **调研阶段**：梳理应用加载的所有资源（脚本、样式、字体、图片、连接等），列出来源域名。
2. **开发阶段**：使用 Content-Security-Policy-Report-Only 模式，不阻断违规，仅收集报告。
3. **灰度阶段**：分析报告，排查被阻止的合法资源，调整策略。
4. **上线阶段**：切换到 enforce 模式，保留 report-uri 持续监控。
5. **维护阶段**：监控违规报告，新功能上线前评估对 CSP 的影响。

使用 strict-dynamic 模式的推荐配置：

```http
Content-Security-Policy: default-src 'self';
                        script-src 'strict-dynamic' 'nonce-{random}' 'unsafe-inline' https:;
                        object-src 'none';
                        base-uri 'self';
                        report-uri /csp-report;
```

### 12. 你的项目采用 BFF 架构，请设计前端认证方案，包含登录、Token 刷新、登出。

async function login(username, password) {
  const response = await fetch('/api/auth/login', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ username, password }),
    credentials: 'include',
  });
  if (!response.ok) throw new Error('Login failed');
  return response.json();
}

let isRefreshing = false;
let refreshQueue = [];

async function fetchWithAuth(url, options = {}) {
  const response = await fetch(url, { ...options, credentials: 'include' });
  if (response.status === 401) {
    if (!isRefreshing) {
      isRefreshing = true;
      const refreshResponse = await fetch('/api/auth/refresh', {
        method: 'POST', credentials: 'include',
      });
      isRefreshing = false;
      if (refreshResponse.ok) {
        refreshQueue.forEach(cb => cb());
        refreshQueue = [];
        return fetchWithAuth(url, options);
      } else {
        window.location.href = '/login';
        throw new Error('Session expired');
      }
    } else {
      return new Promise((resolve) => {
        refreshQueue.push(() => resolve(fetchWithAuth(url, options)));
      });
    }
  }
  return response;
}

async function logout() {
  await fetch('/api/auth/logout', { method: 'POST', credentials: 'include' });
  window.location.href = '/login';
}
```

### 13. 解释 OAuth PKCE 流程，并说明为什么它比 Implicit Flow 更安全。

**参考答案**：

PKCE 的核心安全改进在于引入了 code_verifier 和 code_challenge：

1. **验证器绑定**：code_challenge 绑定到 authorization code，即使授权码被拦截也无法换取 Token。
2. **抗授权码拦截**：在公共客户端（如 SPA）中，授权码可能被重定向过程中的恶意应用拦截。没有 PKCE 时，拦截者可以用授权码直接换取 Token。有了 PKCE，拦截者还需要 code_verifier。
3. **不需要 client_secret**：SPA 无法安全存储 client_secret，PKCE 消除了对它的依赖。

Implicit Flow 的致命缺陷：
- Access Token 直接出现在 URL fragment 中，暴露在浏览器历史、服务器日志、Referer 头中
- 无法使用 Refresh Token
- 没有客户端身份验证

**评分维度**：
- PKCE 流程解释（40%）
- 安全分析（30%）
- Implicit Flow 的缺陷（20%）
- 表达能力（10%）

**常见错误**：
- 混淆 authorization code 和 access token
- 说不清 PKCE 如何防御具体攻击场景
- 认为 Implicit Flow 在使用了 fragment 模式后就安全了

**扩展追问**：

### 14. 描述 XSS 的防御纵深，从浏览器层面、框架层面、应用层面、监控层面分别说明。

- 避免使用 innerHTML、eval、new Function、document.write

  ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a'],

```

**监控层面**：
- CSP 违规报告收集和分析

### 15. 前端 CSRF 有哪些防御方式？现代浏览器提供了哪些新能力？

**参考答案**：

**传统防御**：

1. **CSRF Token**：
```html
<form action="/api/transfer" method="POST">
  <input type="hidden" name="_csrf" value="随机Token">
  <button type="submit">转账</button>
</form>
```

2. **SameSite Cookie**（现代浏览器推荐）：
```http
Set-Cookie: session=abc; SameSite=Lax; Secure; HttpOnly
```

3. **Origin / Referer 校验**：
```javascript
function isSameOrigin(request) {
  const origin = request.headers.origin;
  return origin === 'https://myapp.com';
}
```

4. **自定义请求头**：
```javascript
fetch('/api/transfer', {
  method: 'POST',
  headers: { 'X-Requested-With': 'XMLHttpRequest' },
  credentials: 'include',
});
```

**SameSite 详细**：
```http
# Lax（默认值）：GET 请求自动携带
Set-Cookie: session=abc; SameSite=Lax

# Strict：所有跨站请求都不携带
Set-Cookie: session=abc; SameSite=Strict

# None：需要配合 Secure
Set-Cookie: session=abc; SameSite=None; Secure
```

### 18. 如何在前端实现 WebAuthn / Passkeys 无密码登录？对比传统密码登录的安全性。

**参考答案**：

**WebAuthn 优势**：
- 防钓鱼：绑定到特定域名，不会被假冒站点欺骗
- 防撞库：不传输密码，服务端不存储密码
- 抗重放：每次认证包含挑战值（challenge）
- 便捷性：指纹/人脸/安全密钥

**安全性对比**：

| 特性 | 密码登录 | WebAuthn |
|------|---------|----------|
| 存储方式 | 服务端存哈希 | 服务端存公钥 |
| 传输风险 | 密码在传输中可能泄露 | 私钥不出设备 |
| 钓鱼攻击 | 容易 | 天然防御 |
| 重放攻击 | 需要额外防护 | 挑战值机制防御 |
| 撞库攻击 | 容易 | 不可行 |
| 社工攻击 | 密码可被猜测 | 生物特征或硬件 |

**评分维度**：
- 流程理解（30%）
- 安全性优势（30%）
- 实现考虑（20%）
- 与传统密码对比（20%）

**常见错误**：
- 混淆 WebAuthn 和 TOTP/2FA

**参考答案**：

**风险分析**：

### 20. 设计一个前端安全事件应急响应流程。

**参考答案**：

**应急响应流程**：

```
1. 发现（Detection）
   - 安全监控告警（CSP 违规、异常请求、用户举报）
   - 依赖漏洞通告（npm audit、Snyk）
   - 渗透测试报告

2. 评估（Triage）
   - 确认漏洞是否真实存在
   - 评估影响范围（用户数、数据类型、功能模块）
   - 评估风险等级（严重/高危/中危/低危）

3. 修复（Fix）
   - 严重/高危：立即修复并 hotfix 上线
   - 中危：纳入当前迭代
   - 低危：纳入下个迭代
   - 记录修复步骤和验证方法

4. 发布（Deploy）
   - 严重问题：跳过常规发布流程，紧急上线
   - 普通问题：走正常 CI/CD 流程
   - 发布后验证修复生效

5. 复盘（Postmortem）
   - 漏洞根因分析
   - 为什么在之前的测试中未发现
   - 改进检测手段
   - 更新安全规范和审查清单
```

**SLA 参考**：

| 风险等级 | 修复时间 | 上线时间 |
|---------|---------|---------|
| 严重 | 4 小时 | 1 小时 |
| 高危 | 8 小时 | 2 小时 |
| 中危 | 3 天 | 1 天 |
| 低危 | 下一迭代 | 下一迭代 |

**评分维度**：
- 流程完整性（30%）
- 分级响应（25%）
- 时间约束（20%）
- 复盘改进（25%）

**常见错误**：
- 没有分级，所有漏洞同等对待
- 修复后不做验证直接上线
- 缺少复盘环节，同类漏洞重复出现

**扩展追问**：
- 如何平衡紧急修复和代码质量？
- 安全事件的上报渠道有哪些？如何保证匿名举报？
