## 一、基础题

### 1. 什么是 i18n 和 l10n？有什么区别？

**参考答案**：

i18n（Internationalization）是国际化，使产品具备支持多语言和地区的能力。l10n（Localization）是本地化，针对特定地区进行适配，如翻译、RTL、货币、日期格式。

区别：i18n 是能力，l10n 是具体适配。

**评分维度**：
- 定义准确（50%）
- 区别说明（30%）
- 举例说明（20%）

- RTL 适配：布局镜像、图标、动画。
- 内容管理：翻译平台集成、版本控制。
- 地区路由：URL/域名切换语言。

## 二、进阶题

### 4. ICU MessageFormat 有什么作用？

**参考答案**：

ICU MessageFormat 用于处理复杂翻译场景，如插值、复数、选择、日期数字格式化。

示例：

```
{count, plural, =0 {无消息} one {1 条消息} other {# 条消息}}
```

**评分维度**：

### 9. 对比 react-i18next 和 react-intl 的核心差异，你如何选型？

**参考答案**：

| 维度 | react-i18next | react-intl |
|------|--------------|------------|
| ICU 标准 | 需插件（i18next-icu） | 原生支持 |
| 编译时提取 | 需 babel-plugin-i18next-extract | 内置 @formatjs/cli |
| 体积 | ~8KB | ~12KB |
| 社区生态 | 更大，插件丰富 | 相对小 |
| 非 React 支持 | 优秀（i18next 独立） | 仅 React |
| TypeScript | 优秀 | 良好 |

**选型建议**：
- 已有 react-i18next 生态或需要非 React 场景复用的选 react-i18next。
- 严格遵循 ICU 标准、重视编译时检查和类型安全的选 react-intl。

### 12. 如何设计 locale 自动检测方案？优先级顺序是什么？

**参考答案**：

优先级（高到低）：
1. 用户手动选择（持久化在 cookie/localStorage）
2. URL 路径/子域名（/zh/product, zh.example.com）
3. 服务端 Accept-Language 请求头
4. 客户端 navigator.language
5. 地理位置（IP Geo）
6. 配置的默认语言

```js
function resolveLocale(req, cookieLocale) {
  // 1. URL 参数 > 2. Cookie > 3. Accept-Language > 4. 默认
  const urlLocale = extractFromURL(req.url);
  if (supportedLocales.includes(urlLocale)) return urlLocale;
  
  if (cookieLocale && supportedLocales.includes(cookieLocale)) return cookieLocale;
  
  const acceptLocale = parseAcceptLanguage(req.headers["accept-language"]);
  if (acceptLocale && supportedLocales.includes(acceptLocale)) return acceptLocale;
  
  return "en";
}
```

**评分维度**：
- 优先级设计合理（30%）
- 各策略实现方案（25%）
- 服务端与客户端配合（25%）
- 边界情况处理（20%）

**常见错误**：
- 只有客户端检测，SSR 时先返回错误语言再闪烁。
- 未考虑语言回退（zh-HK -> zh -> en）。
- 忽略 SEO 影响，多语言 URL 结构不规范。

**扩展追问**：

### 13. 如何在微前端架构中实现多语言支持？有哪些方案和权衡？

**参考答案**：

**方案一：共享 i18n 实例**
- 主应用初始化 i18n 实例，通过共享上下文传递。
- 优点：一致性高，翻译复用。
- 缺点：耦合，版本冲突，更新需要协调。

**方案二：独立 i18n 实例**
- 每个微应用独立初始化 i18n。
- 优点：完全解耦，独立部署。
- 缺点：重复加载，翻译不一致。

**方案三：混合模式（推荐）**
- 使用命名空间隔离。
- 主应用提供 shell 命名空间。
- 子应用使用独立命名空间。
```js
t("shell:header.title");   // 主应用
t("appA:dashboard.title"); // 子应用 A
t("appB:checkout.title");  // 子应用 B
```
- 翻译文件独立加载，通过事件或 shared state 同步语言切换。

**评分维度**：
- 三种方案清晰对比（30%）
- 命名空间设计（25%）
- 解耦与协调权衡（25%）
- 语言切换一致性（20%）

**常见错误**：
- 未考虑独立部署场景下翻译文件版本不匹配。
- 使用全局变量共享 i18n 实例造成内存泄漏。
- 语言切换时只通知部分微应用。

**扩展追问**：
- Module Federation 如何共享 i18n 资源？
- 子应用如何感知主应用的语言切换事件？

### 14. 如何在不使用第三方库的情况下，实现一个轻量级 i18n 方案？

**参考答案**：

```js
class LightweightI18n {
  constructor(options = {}) {
    this.locale = options.locale || "en";
    this.fallbackLocale = options.fallback || "en";
    this.messages = {};
  }
  
  addMessages(locale, messages) {
    this.messages[locale] = messages;
  }
  
  t(key, params = {}) {
    let template = this.messages[this.locale]?.[key] 
      || this.messages[this.fallbackLocale]?.[key] 
      || key;
    
    // 简单插值
    return template.replace(/\{(\w+)\}/g, (_, k) => params[k] ?? `{${k}}`);
  }
  
  // 支持简单复数
  plural(count, options) {
    return options[count] || options["other"] || "";
  }
  
  changeLanguage(locale) {
    this.locale = locale;
    // 触发 re-render
  }
}

// 使用
const i18n = new LightweightI18n({ locale: "zh" });
i18n.addMessages("zh", {
  "welcome": "你好，{name}",
  "items": "你有 {count} 条消息"
});

console.log(i18n.t("welcome", { name: "Alice" }));
```

**评分维度**：
- 核心功能完整度（30%）
- 扩展性考虑（25%）
- 插值/复数实现（25%）
- 性能与体积（20%）

### 16. 设计一个多语言 SaaS 应用的错误处理策略，包括翻译缺失、加载失败、格式化异常。

**参考答案**：

```js
class I18nErrorBoundary {
  // 1. 翻译缺失 -> 显示 key 名称 + 日志告警
  t(key, options) {
    const translation = this.getTranslation(key);
    if (!translation) {
      console.warn(`[i18n] Missing translation: ${key} for locale ${this.locale}`);
      Sentry.captureMessage(`Missing i18n key: ${key}`, "warning");
      return key; // 显示 key 方便调试
    }
    return translation;
  }
  
  // 2. 翻译文件加载失败 -> 重试 + fallback
  async loadLocale(locale) {
    try {
      return await fetch(`/locales/${locale}.json`);
    } catch (err) {
      console.error(`[i18n] Failed to load locale ${locale}`, err);
      // 尝试加载 fallback 语言
      if (locale !== this.fallbackLocale) {
        return this.loadLocale(this.fallbackLocale);
      }
      // 最终 fallback
      return this.getInlineFallback();
    }
  }
  
  // 3. 格式化异常 -> try-catch 兜底
  formatDate(date, locale, options) {
    try {
      return new Intl.DateTimeFormat(locale, options).format(date);
    } catch (err) {
      console.warn("[i18n] Date formatting error", err);
      return String(date);
    }
  }
}
```

**评分维度**：
- 错误分类全面（30%）
- 兜底策略合理（30%）
- 可观测性集成（20%）
- 用户体验影响最小化（20%）

**常见错误**：
- 翻译缺失时直接崩溃或白屏。
- 忽略 Intl API 在不支持环境下的 polyfill。
- 加载失败没有重试机制。

- 如何在生产环境中监控翻译覆盖率？
