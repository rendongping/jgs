## 参考答案

### 第 1 题

::: details 查看答案与解析
**答案：A**

pnpm 通过硬链接和严格 node_modules 结构，实现更快安装和更小磁盘占用。
:::

### 第 2 题

::: details 查看答案与解析
**答案：B**

lockfile 锁定确切版本，保证不同环境构建结果一致。
:::

### 第 3 题

::: details 查看答案与解析
**答案：C**

依赖混淆是通过上传与内部包同名的公共包窃取敏感信息，属于供应链攻击。
:::

### 第 4 题

::: details 查看答案与解析
**答案：A**

`^1.2.3` 允许更新到 1.x 的最新版本，但不包括 2.0.0。
:::

### 第 5 题

::: details 查看答案与解析
**答案：B**

SBOM 记录软件所有依赖及其来源，用于审计、合规和漏洞响应。
:::

### 第 6 题

::: details 查看答案与解析
**参考思路**：
1. 确认漏洞影响范围和利用条件。
2. 查看是否有官方修复版本。
3. 若直接依赖不维护，考虑 fork 修复或替换依赖。
4. 临时使用 overrides/resolutions 强制指定安全版本。
5. 增加运行时监控和 WAF 规则。
:::

### 第 7 题

::: details 查看答案与解析
**参考思路**：
- 这是依赖混淆攻击。
- 防范：使用 scoped packages、私有 registry、安装前核对包名和来源、配置 registry 白名单。
:::

### 第 8 题

**参考流程**：
1. 开发者提交新包申请，说明用途和替代方案。
2. 评估：活跃度、体积、许可、安全记录。
3. 安全团队扫描漏洞。
4. 审批通过后加入白名单。
5. 定期复盘已引入包的使用情况。

### 第 9 题

::: details 查看答案与解析
**参考思路**：
- npm/yarn 扁平化 node_modules，可能导致间接依赖被直接引用（幽灵依赖）。
- pnpm 使用严格结构，只有直接依赖出现在顶层，间接依赖通过软链接访问。
- 因此 pnpm 不会出现幽灵依赖问题。
:::

### 第 10 题

**参考方案**：
- 引入：审批 + 来源审查 + 静态扫描。
- 审计：定期 npm audit + Snyk + SBOM。
- 更新：Renovate 自动提 PR，人工审核大版本。
- 应急：漏洞响应小组、快速评估、强制更新或替换。

### 第 11 题

::: details 查看答案与解析
**参考思路**：
- 使用 workspace 协议管理内部包。
- 公共依赖提升到 root。
- 统一版本策略，避免多版本共存。
- 使用 pnpm catalog 或 Rush 等工具统一管理。
- 构建依赖图，检测循环依赖。
:::
