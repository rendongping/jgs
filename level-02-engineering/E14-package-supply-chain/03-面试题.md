### 2. 什么是幽灵依赖？如何避免？

**参考答案**：

幽灵依赖是指项目直接使用了未在 package.json 中声明的间接依赖。npm/yarn 的扁平化结构容易导致这个问题。pnpm 的严格 node_modules 结构可以避免。

- 仅停留在概念层面，无法说明 pnpm 是如何通过 symlink 结构解决问题的。
- 忽略幽灵依赖在升级间接依赖时导致的断裂风险。

### 4. npm workspaces、yarn workspaces 和 pnpm workspaces 有什么区别？

**参考答案**：

三种工作区方案都用于支持 Monorepo 架构，但实现机制和特性有所不同：

- **npm workspaces**（v7+ 内置）：基于 node_modules 扁平化 hoisting，顶层安装公共依赖。配置简单，原生支持。缺点是仍存在幽灵依赖问题，多版本依赖时处理能力较弱。
- **yarn workspaces**（v1+）：与 npm 类似也采用扁平化 hoisting，但在依赖安装速度和缓存策略上更优。yarn v2+（Berry）引入了 PnP（Plug'n'Play）模式，不再依赖 node_modules 目录，彻底解决了幽灵依赖问题。PnP 通过 .pnp.cjs 文件映射依赖位置，安装速度极快。
- **pnpm workspaces**：基于内容寻址存储（content-addressable store）和严格 symlink 结构。所有包统一存储在全局 store 中，项目内通过硬链接和符号链接引用。node_modules 结构严格反映 package.json 的声明，杜绝幽灵依赖。安装速度最快，磁盘空间占用最小。

选择建议：
- 小团队或已有 npm 生态的项目选用 npm workspaces。
- 追求最佳性能和严格依赖隔离的项目选用 pnpm。
- 需要 PnP 等高级特性的团队选择 yarn Berry。

**评分维度**：
- 三种方案的基本原理（40%）
- 差异对比与优劣势（30%）
- 适用场景分析（30%）

**常见错误**：
- 认为三种 workspaces 实现原理相同，只是 CLI 命令不同。
- 忽略 pnpm 的内容寻址存储带来的磁盘效率优势。
- 不了解 yarn v2+ 的 PnP 模式与传统 node_modules 模式的本质区别。

**扩展追问**：
- 如果团队需要从 npm workspaces 迁移到 pnpm workspaces，需要注意哪些问题？
- pnpm 的 node_modules 结构为何能杜绝幽灵依赖？请画出目录结构图。
- yarn PnP 模式中，如何处理原生模块（native modules）的兼容性问题？

### 5. dependencies、devDependencies 和 peerDependencies 有什么区别？

**参考答案**：

三种依赖类型在安装时机、传递行为和用途上存在本质区别：

- **dependencies**：运行时依赖，应用或库正常运行所必需的包。执行 `npm install` 时默认安装。在发布包时，这些依赖会随包一起被安装到用户项目中。示例：react、lodash、express。
- **devDependencies**：开发时依赖，仅在开发、测试、构建阶段需要。执行 `npm install` 时默认安装，但在生产环境（`NODE_ENV=production`）下不会安装。包发布后，用户安装该包时不会安装其 devDependencies。示例：jest、webpack、eslint、typescript。
- **peerDependencies**：同伴依赖，声明当前包需要宿主环境提供某些依赖。不会自动安装，如果宿主环境未提供，npm v7+ 会自动安装。主要用于插件和库的开发，要求使用者自行安装特定版本的依赖。示例：react-dom 需要 react 作为 peerDependency、eslint 插件需要 eslint 作为 peerDependency。

最佳实践：
- 库的开发：将构建工具放在 devDependencies，运行时依赖放在 dependencies。
- 插件开发：将宿主框架放在 peerDependencies，避免多版本冲突。
- 应用开发：主要使用 dependencies 和 devDependencies，通常不需要 peerDependencies。

**评分维度**：
- 三种依赖的定义与用途（40%）
- 安装与传递行为（30%）
- 实际应用场景（30%）

**常见错误**：
- 把构建工具放在 dependencies 中，导致用户安装时下载无关依赖。
- 插件开发时未使用 peerDependencies，导致同一依赖存在多个版本冲突。
- 混淆 optionalDependencies 和 peerDependencies 的用途。

**扩展追问**：
- 如果某个依赖同时出现在 dependencies 和 devDependencies 中，哪个优先级更高？
- bundledDependencies 是什么，何时使用？
- 如何检测项目中是否有不必要的依赖被错误分类？

### 8. pnpm 的严格 node_modules 是如何工作的？解释其 symlink 结构以及为何能防止幽灵依赖。

**参考答案**：

pnpm 的 node_modules 结构采用"三明治"模型，与 npm/yarn 的扁平化结构完全不同：

**目录结构**：
```
node_modules/
  .pnpm/                        # 虚拟存储，所有包的真实位置
    react@18.2.0/
      node_modules/react        # 真实文件（硬链接自全局 store）
    lodash@4.17.21/
      node_modules/lodash
    express@4.18.2/
      node_modules/express
      node_modules/accepts      # express 的依赖
  react -> .pnpm/react@18.2.0/node_modules/react  # 顶层符号链接
  lodash -> .pnpm/lodash@4.17.21/node_modules/lodash
```

**工作原理**：
1. **全局 Store**：所有包解压到全局 content-addressable store（~/.pnpm-store），通过包名和版本哈希索引。
2. **硬链接**：从 store 硬链接到 .pnpm 目录，节省磁盘空间（同一版本的同文件只存一份）。
3. **符号链接**：从项目顶层 node_modules 创建 symlink 指向 .pnpm 中的具体包。
4. **严格嵌套**：每个包只能访问其 package.json 中声明的依赖。包 A 的依赖不会暴露给包 B。

**为何杜绝幽灵依赖**：
- npm/yarn 将依赖扁平化提升到顶层 node_modules，导致未声明的依赖也可被 require。
- pnpm 保持 node_modules 的原始嵌套语义，顶层 node_modules 只包含 package.json 中直接声明的依赖 symlink。
- 每个包内部 node_modules 仅包含该包直接依赖的 symlink，无法访问未声明的依赖。

**评分维度**：
- symlink 结构理解（35%）
- 内容寻址存储机制（25%）
- 与幽灵依赖的关系（25%）
- 性能与空间优势（15%）

**常见错误**：
- 将 pnpm 的硬链接和符号链接概念混淆。
- 认为 pnpm 完全消除 node_modules 目录（实际上只是优化了结构）。
- 不了解 pnpm 如何处理不同版本的同一依赖。

### 9. 如何为大型前端项目搭建依赖漏洞监控系统？

**参考答案**：

一个完整的依赖漏洞监控系统需要覆盖引入时检测、持续性监控、告警响应三个环节：

**1. 引入时检测（Shift-Left）**：
```yaml
# .github/workflows/dependency-audit.yml
name: Dependency Security Scan
on: [pull_request]
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci
      - run: npm audit --audit-level=high
      # 或使用更强大的工具
      - run: npx snyk test --severity-threshold=high
      # 检测许可合规
      - run: npx license-checker --failOn GPL
```

**2. 持续性监控**：
- **Dependabot**：GitHub 原生支持，自动检测并创建 PR 修复已知漏洞。
- **Renovate**：更灵活的自动更新工具，支持自定义更新策略和分组。
- **Snyk**：商业方案，提供实时监控、漏洞数据库和修复建议。
- **Socket.dev**：检测恶意包、typosquatting 和异常行为。

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "daily"
    open-pull-requests-limit: 10
    labels:
      - "security"
```

**3. 告警与响应**：
- 集成 Slack/钉钉/企业微信 webhook 发送告警。
- 基于 CVSS 评分定义响应 SLA：Critical（4h）、High（24h）、Medium（72h）。
- 建立依赖漏洞台账，跟踪修复进度。

**4. 阻断策略**：
- 在 CI 中设置 `npm audit` 失败门槛，高危及以上漏洞阻断合并。
- 使用 `--audit-level=high` 在安装时即告警。
- 私有 registry 中配置自动屏蔽含已知漏洞的包版本。

**评分维度**：
- 工具链完整性（30%）
- 流程覆盖面（30%）
- 自动化程度（20%）
- 应急响应机制（20%）

**常见错误**：
- 仅依赖 npm audit，忽视 Snyk、Socket.dev 等更专业的工具。
- 只监控直接依赖而忽略传递依赖的漏洞。
- 有检测无响应，告警过多导致团队产生告警疲劳。

**扩展追问**：
- 如何处理发现漏洞但上游包已不再维护的情况？
- 如何在保证安全的同时不影响开发效率？
- 对于 Monorepo 项目，如何避免重复扫描和告警膨胀？

### 10. 解释 npm 的安装算法。运行 `npm install` 时逐步发生了什么？

**参考答案**：

`npm install` 的执行过程可以分为以下阶段：

**第一阶段：解析与构建依赖树**
1. 读取 package.json，获取所有直接依赖及其版本范围。
2. 检查是否存在 lockfile（package-lock.json）：
   - 存在 lockfile：读取 lockfile 中的确切版本和依赖关系，跳过解析步骤。
   - 不存在 lockfile：进入依赖解析。
3. 依赖解析（无 lockfile 时）：
   - 从根依赖开始，递归查询每个依赖的 registry 元数据。
   - 确定每个包满足 semantic version 的最新版本。
   - 构建完整的依赖树，处理多版本冲突。
4. 生成依赖树：
   - npm v3+ 采用扁平化策略，尽可能将依赖提升到顶层。
   - 当出现版本冲突时，中间件会在对应包下嵌套安装冲突版本。
   - 使用回溯法（backtracking）解决版本冲突，找到满足所有约束的方案。

**第二阶段：下载与安装**
5. 检查缓存（~/.npm/_cacache）：
   - 缓存命中：直接从缓存复制到 node_modules。
   - 缓存未命中：从 registry 下载 tgz 包并存入缓存。
6. 解压与写入：
   - 将 tgz 包解压到 node_modules 对应位置。
   - 写入 package-lock.json（如果文件变更）。

**第三阶段：生命周期脚本**
7. 依赖树构建完成后，按拓扑顺序执行 lifecycle scripts：
   - preinstall → install → postinstall（每个安装的包）
   - 根包的 script 最后执行。
8. 重新写入 package-lock.json 锁定确切版本。

**性能优化**：npm v7+ 引入 `--prefer-offline` 优先使用缓存，`--legacy-peer-deps` 绕过严格 peerDeps 检查以加速安装。

**评分维度**：
- 依赖树构建过程（40%）
- 扁平化与冲突处理（30%）
- 缓存与生命周期（20%）
- lockfile 的作用（10%）

**常见错误**：
- 认为 npm 总是安装最新版本，忽略 lockfile 对版本的控制作用。
- 不理解扁平化算法中"嵌套"与"提升"的决策逻辑。
- 混淆 `npm install` 和 `npm ci` 的执行过程差异。

**扩展追问**：
- `npm ci` 与 `npm install` 的区别是什么？什么场景下应使用 `npm ci`？
- npm 的扁平化算法在哪些场景下性能显著下降？

### 11. 什么是 SBOM（软件物料清单）？如何为项目生成和维护 SBOM？

**参考答案**：

SBOM（Software Bill of Materials）是软件中包含的所有组件的清单列表，包括直接依赖、传递依赖、版本信息、许可证、供应商信息等。类似于食品包装上的成分表。

**SBOM 的核心价值**：
- 漏洞管理：当 CVE 发布时，通过 SBOM 快速判断是否受影响。
- 合规审计：验证许可合规性，满足法规要求（如美国 EO 14028、欧盟 Cyber Resilience Act）。
- 供应链透明度：了解项目中每个组件的来源和依赖关系。

**生成 SBOM 的工具**：

```bash
# 使用 CycloneDX 格式（OWASP 标准）
npx @cyclonedx/bom --spec-version 1.4 --output ./bom.xml

# 使用 SPDX 格式（Linux 基金会标准）
npx spdx-sbom-generator -p . -o ./sbom.spdx

# npm 原生支持（v10+）
npm sbom --format=cyclonedx

# 将 SBOM 上传到依赖监控平台
npx @cyclonedx/bom | curl -X POST -d @- https://api.example.com/sbom
```

**持续维护策略**：
1. **CI/CD 集成**：每次构建自动生成 SBOM 并存档。
2. **差异化对比**：比较两次构建的 SBOM，检测新增的依赖。
3. **漏洞关联**：将 SBOM 导入漏洞库（如 NVD、GitHub Advisory 数据库），自动匹配已知漏洞。
4. **版本管理**：将 SBOM 与版本发布 tag 关联，便于追溯。

```yaml
# .github/workflows/sbom.yml
name: Generate SBOM
on:
  push:
    tags:
      - "v*"
jobs:
  sbom:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm sbom --format=cyclonedx > bom.cdx.json
      - uses: actions/upload-artifact@v4
        with:
          name: sbom-${{ github.ref_name }}
          path: bom.cdx.json
```

**评分维度**：
- SBOM 概念理解（30%）
- 生成工具与方法（30%）
- 格式标准（CycloneDX vs SPDX）（20%）
- 持续集成方案（20%）

**常见错误**：
- 认为 SBOM 只是简单的 package.json 清单，忽略传递依赖。
- 仅生成一次 SBOM，缺乏持续更新的机制。
- 不了解 SBOM 在不同格式（CycloneDX、SPDX）之间的差异和适用场景。

### 13. npm link、yarn link 和 pnpm link 有什么区别？何时使用每种方式？

**参考答案**：

三种 link 命令都用于在本地开发中测试未发布的包，但实现机制和特性有所不同：

**npm link**：
- 机制：创建一个全局 symlink（`{prefix}/lib/node_modules/<pkg>` → 本地开发目录），然后在目标项目中执行 `npm link <pkg>` 创建引用。
- 优点：生态系统最广泛，几乎在所有环境中可用。
- 缺点：
  - 全局 link 可能污染全局 node_modules。
  - 共用同一份文件，修改后影响所有 link 该包的项目。
  - 依赖解析可能遇到问题（两个 `react` 实例）。
  ```bash
  # 在包目录
  cd ~/projects/my-lib && npm link
  # 在目标项目
  cd ~/projects/my-app && npm link my-lib
  # 取消 link
  cd ~/projects/my-app && npm unlink --no-save my-lib
  cd ~/projects/my-lib && npm unlink
  ```

**yarn link**：
- 机制：与 npm link 类似，但 yarn v2+ 支持 PnP 模式下的 link。
- 优势：PnP 模式下没有 node_modules，link 通过 .pnp.cjs 文件注册。
- 劣势：yarn v2+ 的 link 在某些跨项目管理器中兼容性较差。
  ```bash
  cd ~/projects/my-lib && yarn link
  cd ~/projects/my-app && yarn link my-lib
  ```

**pnpm link**：
- 机制：使用 pnpm 的全局 store 和硬链接/符号链接机制，比 npm link 更可控。
- 优势：
  - `pnpm link --global` 类似 npm link。
  - `pnpm link <pkg>` 仅在当前项目生效，不污染全局。
  - 使用 pnpm 的 store 机制，保持严格的依赖隔离。
- 特有功能：`--preserve-symlinks` 选项处理复杂的 Monorepo 场景。
  ```bash
  # 全局 link
  cd ~/projects/my-lib && pnpm link --global
  cd ~/projects/my-app && pnpm link --global my-lib
  # 直接 link（推荐，仅在本地生效）
  cd ~/projects/my-app && pnpm link ~/projects/my-lib
  ```

**选择建议**：
- 个人开发/简单测试：npm link。
- 使用 yarn v2+ PnP 的项目：yarn link。
- Monorepo 或需要严格隔离的项目：pnpm link。
- Monorepo 场景优先使用 workspace 协议（`workspace:*`），仅在需要跨仓库测试时使用 link。

**评分维度**：
- link 机制原理（40%）
- 三种包管理器的差异（30%）
- 实际应用场景（30%）

**常见错误**：
- 使用 link 后在生产构建中出现版本重复问题，未使用 `resolve.alias` 或 `dedupe` 处理。
- link 后忘记 unlink，导致 CI/CD 环境出现意外行为。
- 在 Monorepo 中滥用 link 应使用 workspace 协议的场景。

**扩展追问**：

### 16. 设计一个依赖防火墙架构，拦截所有包安装操作，应用安全策略，在恶意包到达开发者之前阻止它们。

**参考答案**：

依赖防火墙是一个位于开发环境和公共 Registry 之间的安全代理层，对每次包请求执行安全策略。

**架构设计**：

```
开发者/CI  -> 依赖解析器 -> 本地缓存 -> 防火墙引擎 -> 公共 Registry
                              ^              |
                              |              v
                          合法包存储    策略规则库
                                        |
                                        v
                                    安全管理面板
```

**核心组件**：

1. **拦截层（Proxy/Interception）**：
   - 使用 npm registry proxy（如 Verdaccio 或 CNPM）作为统一的包入口。
   - 配置 .npmrc 指向私有 registry，所有安装请求经过代理。

2. **策略引擎（Policy Engine）**：
   - 预安装检查：包下载前执行一系列安全规则。
   - 规则集示例：
     - 包名规则：禁止 typosquatting 命名（与内部包名 Levenshtein 距离小于 3）。
     - 版本规则：禁止安装标记为 deprecated 的版本。
     - 许可证规则：禁止 GPL/AGPL 许可包。
     - 维护规则：超过 2 年未更新的包需人工审批。
     - 依赖规则：传递依赖超过 50 个的包需审查。
   - 沙箱执行：对 postinstall 脚本在隔离沙箱中执行，检测恶意行为。

3. **行为分析（Behavioral Analysis）**：
   - 使用 AST 分析工具检查包代码中的可疑模式。
   - 静态分析：检测编码混淆、字符串加密、base64 解码等恶意行为特征。
   - 异常检测：首次发布、短时间内高频发布、作者邮箱身份存疑。

4. **审批工作流**：
   - 自动通过：绿色策略包，默认允许。
   - 需人工审查：黄色策略包，自动创建审批工单。
   - 自动阻断：红色策略包，返回错误信息并记录日志。

5. **审计与告警**：
   - 所有策略命中记录到日志系统（ELK）。
   - 自动阻断的尝试立即通知安全团队。
   - 月度依赖防火墙运行报告：阻断率、通过率、告警趋势。

**评分维度**：
- 架构完整性（30%）
- 策略规则设计（25%）
- 安全检测深度（25%）
- 可操作性与可维护性（20%）

**常见错误**：
- 防火墙策略过于严格，严重影响开发效率。
- 只做网络层拦截，未在包内容层面做安全检查。
- 忽视 postinstall 脚本的威胁，认为包内容安全即可。
- 缺少豁免机制，紧急情况无法绕过防火墙。

**扩展追问**：

- 防火墙如何处理 scoped packages 的权限控制？
- 如果私有 registry 宕机，如何设计降级策略而不阻塞开发？
- 如何集成第三方安全情报（如 Snyk、Socket.dev）到防火墙策略中？

### 17. 如何实现一个自定义包注册表代理（类似 Verdaccio）？考虑缓存、认证、包发布工作流和公共注册表集成。

**参考答案**：

一个功能完善的包注册表代理需要处理包的分发、缓存、认证、发布等核心环节。

**架构组件**：

```
用户/CI -> 反向代理 -> API 网关 -> 路由引擎 -> 上游（npm/public registry）
                          |
                          v
                    本地数据库（SQLite/PostgreSQL：存储包元数据、用户信息）
                          |
                          v
                    存储后端（本地文件系统/S3：存储 .tgz 文件）
                          |
                          v
                    缓存层（Redis：加速元数据响应）
```

**核心功能实现**：

1. **Upstream 代理与缓存**：
   - 请求到达时先查本地数据库，如不存在则从上游 registry 拉取。
   - 实现缓存策略：元数据缓存（TTL 5分钟）、包文件缓存（永久存储）。

2. **认证与权限**：
   - JWT/OAuth2 认证，根据团队和项目控制包访问权限。
   - 读取权限：开发人员可读所有公共包。
   - 发布权限：仅特定团队可发布到作用域。
   - Token 管理：支持 npm token 和 CI token 两种认证方式。

3. **包发布工作流**：
   - npm publish -> 代理验证用户身份。
   - 执行防火墙策略（扫描漏洞、检查许可证）。
   - 安全检查通过后，包存储到本地存储并更新数据库。
   - 可选择是否同步发布到上游公共 registry。
   - 支持多种发布策略：全部私有、混合（同名包优先私有）、代理穿透。

4. **Scoped Registry 配置**：
   ```
   registry=http://registry.company.com/
   @company:registry=http://registry.company.com/
   ```

5. **高可用与性能**：
   - 使用 CDN 分发高频下载的包文件。
   - 预缓存热门包的流行版本。
   - 数据库读写分离，元数据查询走只读副本。
   - 使用消息队列处理大规模包发布和同步。

6. **审计与日志**：
   - 记录所有包的下载、发布、更新操作。
   - 异常检测：短时间内大量下载、尝试发布同名包等。
   - 提供管理面板查看统计信息和操作日志。

**评分维度**：
- 代理架构理解（30%）
- 缓存策略设计（20%）
- 认证与权限（20%）
- 发布工作流（15%）
- 运维与可扩展性（15%）

**常见错误**：
- 未处理包版本冲突（私有包和公共包同名同版本）。
- 忽略 tgz 包的完整性校验（shasum 验证）。
- 缓存策略过于简单，热冷数据未差异化处理。
- 代理同步阻塞导致发布延迟。

**扩展追问**：

### 18. 设计一个供应链安全事件应急响应计划。包括：检测、分类、遏制、修复和事后复盘。在每个阶段应集成哪些工具？

**参考答案**：

一个完整的供应链安全事件应急响应计划需要覆盖事件全生命周期。

**第一阶段：检测（Detection）**

关键目标：在最短时间内发现安全事件。

工具集成：
```
监控层：
- Dependabot / Renovate         -> 自动检测已知漏洞并提 PR
- Snyk / Socket.dev              -> 实时漏洞情报和恶意包检测
- npm audit / pnpm audit         -> 基础漏洞扫描
- SBOM 差异化对比                -> 发现非预期的依赖变更
- Registry 访问日志分析          -> 检测异常下载模式
- CI/CD 流水线安全扫描           -> 构建时检测
- EDR/IDS（如 Wazuh）           -> 检测运行时异常行为
```

告警阈值：
- Critical（CVSS 9+）：自动呼叫 on-call 工程师。
- High（CVSS 7-8.9）：触发 PagerDuty 告警，15分钟内响应。
- Medium（CVSS 4-6.9）：工单系统，4小时内处理。
- Low（CVSS < 4）：下一迭代修复。

**第二阶段：分类与评估（Triage）**

关键目标：确定事件影响范围，划分严重等级。

执行流程：
1. **影响分析**：使用 SBOM 匹配受影响的包和版本。
2. **暴露评估**：确定受影响的代码路径和外部接口。
3. **严重性定级**：综合 CVSS、利用难度、业务影响确定优先级。

**第三阶段：遏制（Containment）**

关键目标：阻止事件的进一步扩散。

遏制措施：
1. **立即阻断**：
   - 在私有 registry 中标记受影响版本为 blocked，阻止新安装。
   - 在 CI 配置中强制执行 npm audit 失败阻断。
   - 在防火墙上添加包和版本的阻断规则。
2. **热修复**：
   - 如果商业 SaaS，立即回滚到上一个安全版本。
   - 如果自托管服务，应用紧急补丁或特征开关绕过受影响代码。
3. **版本锁定**：在 package.json 中使用 resolutions 强制降级或替换。

**第四阶段：修复（Remediation）**

关键目标：彻底消除安全风险。

修复策略：
1. **补丁升级**：升级到修复版本（如存在 CVE 修复补丁）。
2. **包替换**：如果维护者失能或不响应，使用 fork 或替代包。
3. **猴子补丁（Monkey Patch）**：临时在代码层修补问题函数。
4. **内部复刻**：对关键依赖进行内部 fork 并自行维护修复。

**第五阶段：事后复盘（Post-Mortem）**

关键目标：改进流程，避免事件再次发生。

复盘模板应包含：
```markdown
## 安全事件复盘

### 基本信息
- 事件编号：SEC-2025-001
- CVE 编号：CVE-2025-XXXXX
- 发现时间：2025-01-15 14:23 UTC
- 影响范围：3 个生产服务、2 个开发环境

### 时间线
- 14:23 检测到异常包下载
- 14:30 确认存在 CVE，触发告警
- 14:45 标记受影响版本，阻断新安装
- 15:30 完成所有受影响实例的补丁升级
- 16:00 恢复生产环境正常服务

### 根因分析
- 根本原因：开发人员在 PR 中间接引入了传递依赖
- 检测遗漏：本次引入绕过 CI 扫描（配置未覆盖传递依赖）

### 改进措施
- 更新 CI 扫描配置，覆盖所有传递依赖
- 增加传递依赖变更的 PR 标签
- 完善 SBOM 自动生成流程
- 缩减依赖引入审批豁免时间窗口
```

**评分维度**：
- 阶段划分完整性（25%）
- 检测与监控工具覆盖（20%）
- 遏制与修复策略（25%）
- 复盘与持续改进（15%）
- 与组织结构的结合（15%）

**常见错误**：
