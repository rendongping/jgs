# E13 前端部署与运维（SRE）

> 目标：掌握前端应用的部署、发布、监控、容灾和容量管理，保障线上服务稳定可用。

---

## 核心要点（TL;DR）

- 前端部署不只是把文件上传到 CDN，还包括缓存策略、回滚、灰度和可观测性。
- SRE 的核心是用工程化手段提升系统可靠性，而不是靠人力值守。
- 发布策略决定风险：全量、灰度、蓝绿、金丝雀、A/B 测试各有适用场景。
- 可观测性三大支柱：日志、指标、链路追踪。
- 容灾和回滚能力比发布速度更重要。

---

## 1. 前端部署基础

### 1.1 部署流程

```
构建产物 → 静态资源上传 → CDN 刷新 → DNS/路由切换 → 验证 → 监控
```

### 1.2 静态资源托管

- **对象存储**：OSS/S3/MinIO
- **CDN**：加速静态资源分发
- **边缘节点**：Cloudflare、Vercel Edge、AWS CloudFront

### 1.3 缓存策略

| 资源类型 | 缓存策略 | 说明 |
|----------|---------|------|
| HTML | no-cache / 短缓存 | 保证最新入口 |
| JS/CSS（带 hash） | 长期缓存 | 文件名变化自动失效 |
| 图片/字体 | 长期缓存 | 不常变化 |

## 2. 发布策略

### 2.1 全量发布

- 一次性对所有用户生效。
- 风险高，适合低风险变更或热修复。

### 2.2 灰度发布

- 先对少量用户发布，观察指标后再扩大范围。
- 可按用户 ID、地域、设备、随机比例灰度。

### 2.3 金丝雀发布

- 新版本与旧版本共存，逐步切流。
- 典型比例：1% → 5% → 20% → 50% → 100%。

### 2.4 蓝绿发布

- 准备两套完全独立的环境，切换流量。
- 回滚快，但资源成本高。

### 2.5 A/B 测试

- 同时运行多个版本，通过业务指标选择最优方案。
- 需要完善的分流和埋点体系。

---

## 3. 可观测性

### 3.1 日志（Logs）

- 前端错误日志：window.onerror、unhandledrejection。
- 行为日志：用户操作路径。
- 集中收集：Sentry、Logrocket、自研日志服务。

### 3.2 指标（Metrics）

- Web Vitals：LCP、INP、CLS、FCP、TTFB。
- 业务指标：页面转化率、错误率、白屏率。
- 工具：Google Analytics、自建 Prometheus。

### 3.3 链路追踪（Traces）

- 追踪用户请求从前端到后端的完整路径。
- 工具：OpenTelemetry、Jaeger、SkyWalking。

---

## 4. 容灾与高可用

### 4.1 回滚策略

- 代码回滚：快速切换到上一个稳定版本。
- 配置回滚：开关、灰度比例快速调整。
- 数据回滚：本地存储异常时清理或降级。

### 4.2 降级策略

- 功能降级：关闭非核心功能。
- 静态降级：服务端异常时展示静态兜底页。
- 网络降级：弱网时减少请求或切换离线包。

### 4.3 多活与异地容灾

- 多 CDN 服务商备份。
- 多区域部署，DNS 智能调度。
- RPO/RTO 目标设定。

---

## 5. 容量与成本

### 5.1 容量规划

- 评估峰值 QPS、带宽、存储需求。
- 压测确定系统瓶颈。

### 5.2 成本控制

- CDN 流量优化：压缩、去重、智能预热。
- 构建产物体积控制。
- 边缘计算减少回源。

---

## 常见误区

| 误区 | 正确理解 |
|------|---------|
| 部署成功就万事大吉 | 需要持续监控和快速回滚能力 |
| 缓存越久越好 | HTML 不能长期缓存，否则无法更新 |
| 可观测性只关注错误 | 还要关注性能、业务、用户体验 |
| 只追求发布速度 | 稳定性和可回滚性更重要 |
