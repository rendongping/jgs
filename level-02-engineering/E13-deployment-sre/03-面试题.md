## 一、基础题

### 1. 前端部署为什么要给 JS/CSS 加 hash？

**参考答案**：

加 hash 后文件名与内容绑定，内容变化 hash 变化，浏览器会重新请求；未变化的资源可长期缓存，提升缓存命中率。

**评分维度**：
- 缓存失效机制（50%）
- 长期缓存好处（30%）
- 构建实践（20%）

**常见错误**：
- 回答"加 hash 是为了防止缓存"——不准确，本质是实现精准缓存失效。
- 混淆 content hash 和 chunk hash，未说明 webpack/Rollup 的 `[contenthash]` 配置。
- 遗漏 HTML 入口文件的缓存策略（HTML 不应 hash 化，应使用协商缓存）。

**扩展追问**：
- Q: webpack 中 `[contenthash]` 和 `[chunkhash]` 的区别是什么？
- Q: runtime chunk 单独提取的目的是什么？不提取会有什么问题？
- Q: 如果只改了某行 CSS，但打出的 JS hash 也变了，可能是什么原因？

### 2. 灰度发布和 A/B 测试有什么区别？

**参考答案**：

灰度发布关注风险控制，逐步放量；A/B 测试关注效果对比，同时运行多个版本，用业务指标决策。

**评分维度**：
- 目标差异（50%）

- 决策依据（20%）

**常见错误**：
- 认为灰度发布和 A/B 测试可以完全互相替代——两者的目标不同，灰度发布是为了降低风险，A/B 测试是为了验证假设。
- 忽略同时做灰度 + A/B 的场景（如灰度中嵌套 A/B 实验）。
- 未提及灰度发布的熔断和回滚机制。

### 3. 什么是 CDN？前端如何利用 CDN？

**参考答案**：

CDN 是内容分发网络，通过边缘节点缓存静态资源，降低访问延迟。前端将 JS/CSS/图片/字体等资源部署到 CDN，并配置合理缓存策略。

**评分维度**：
- CDN 原理（40%）
- 前端应用（40%）
- 缓存策略（20%）

**常见错误**：
- 只回答了 CDN 的概念，没有结合前端具体实践。
- 忽略 CDN 缓存失效（purge/refresh）的代价——刷新 CDN 缓存不是实时的，有全网点位生效延迟。
- 未考虑 CDN 容灾——单 CDN 故障时的兜底方案。

**扩展追问**：
- Q: CDN 回源过程是怎样的？回源失败会怎样？

### 4. Docker 多阶段构建在前端部署中有哪些优势？请给出一个 Dockerfile 示例。

**参考答案**：

Docker 多阶段构建通过分离构建环境和运行环境，显著减小最终镜像体积。第一阶段使用 Node 镜像安装依赖并构建，第二阶段使用 nginx:alpine 镜像仅复制构建产物。

```dockerfile
# 第一阶段：构建
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# 第二阶段：运行
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

优势：最终镜像只有几十 MB（仅包含 nginx + 静态文件），而非 Node 镜像的 1GB+。同时减少了攻击面（不需要 node、npm、构建工具）。

**评分维度**：
- 多阶段构建原理（30%）
- Dockerfile 编写（30%）
- 镜像体积优化意识（25%）
- 安全考量（15%）

**常见错误**：
- 只使用单阶段 Node 镜像部署静态文件，镜像体积过大。
- 未使用 `npm ci`（比 `npm install` 更快且更确定）。
- 忘记 `.dockerignore` 排除 node_modules，导致构建效率低。

**扩展追问**：
- Q: 如何进一步减少 Docker 镜像层数和大小？
- Q: 如何使用 Docker BuildKit 加速构建？
- Q: 在前端部署中，Docker 对比直接上传 OSS/CDN 的优缺点？

### 5. CI/CD 流水线的核心环节有哪些？每个环节的质量门禁如何设定？

**参考答案**：

CI/CD 流水线通常包含以下环节：

1. **代码提交**：ESLint + Prettier + TypeScript 检查，Commit Message 规范校验。
2. **单元测试**：Jest/Vitest 运行，覆盖率门禁 >= 80%，失败阻断流水线。
3. **构建**：代码分割 + Tree Shaking + 压缩，生成带内容 hash 的静态资源。
4. **安全扫描**：npm audit / Snyk / Trivy 扫描依赖漏洞，高危漏洞阻断。
5. **部署测试环境**：自动部署后运行 E2E 测试（Playwright/Cypress）。
6. **性能预算检查**：Lighthouse CI 对比基线，LCP > 2.5s 或 Bundle 增量 > 10% 告警。
7. **部署预发环境**：回归验证，人工审批（可选）。
8. **灰度发布生产**：5% -> 20% -> 50% -> 100%，每阶段观察错误率和性能。
9. **部署后验证**：冒烟测试 + 监控对比，异常自动回滚。

**评分维度**：
- 流水线完整性（30%）
- 质量门禁设计（30%）
- 自动化程度（20%）
- 可观测性和回滚（20%）

**常见错误**：
- 缺少安全扫描环节，依赖漏洞流入生产。
- 未做性能预算检查，导致页面性能逐步劣化。
- 流水线缺少人工审批和自动回滚的双重保障。

**扩展追问**：
- Q: CI/CD 流水线中如何做依赖缓存加速？
- Q: 如果流水线某一环节失败，如何快速定位问题？
- Q: 如何实现流水线的"不可变构建"——一次构建，多次部署？

## 二、进阶题

### 6. 如何设计前端错误监控体系？

**参考答案**：

- **捕获**：`window.onerror` 捕获 JS 运行时错误，`unhandledrejection` 捕获 Promise 异常，`addEventListener('error')` 捕获资源加载失败，axios/fetch 拦截器捕获接口异常。
- **上报**：采样策略（高频错误全量，低频错误按 10% 采样），聚合去重（相同错误栈聚合），批量上报减少请求数。
- **分析**：按错误类型、页面路径、版本号、浏览器、地域多维度聚合，sourcemap 还原源码位置。
- **告警**：核心错误 P0 实时通知（白屏率 > 5%、JS Error 率 > 3%），次要错误 P1/P2 延迟通知。

**评分维度**：
- 捕获完整性（30%）
- 上报策略（25%）
- 分析方法（25%）
- 告警机制（20%）

**常见错误**：
- 只捕获了 `window.onerror`，遗漏 `unhandledrejection` 和资源加载错误。
- 前端上报量过大未做采样/聚合，导致服务端压力过大。
- 未上传 sourcemap 或上传后未做访问控制，导致源码泄露。

**扩展追问**：
- Q: sourcemap 上传后如何做好访问权限控制，避免公网泄露源码？
- Q: 如何区分"已知错误"和"新增错误"，减少告警疲劳？
- Q: 跨域脚本的错误信息被浏览器屏蔽（Script error.），如何解决？

### 7. 发布回滚有哪些策略？如何实现快速回滚？

**参考答案**：

- **版本回滚**：部署系统保留最近 N 个版本的构建产物，通过配置中心或路由切换指向旧版本。
- **配置回滚**：关闭灰度开关，将流量 100% 切回旧版本。
- **流量切换**：蓝绿部署中切换回绿环境；DNS 切换需考虑 TTL 延迟。
- **K8s 回滚**：`kubectl rollout undo deployment/<name>`，回滚到上一 Revision，Pod 逐步替换。
- **数据修复**：清理异常写入的 localStorage/IndexedDB 数据，或通过灰度推修复脚本。

快速回滚的关键：保留历史产物 + 自动化脚本 + 部署系统的一键回滚能力。

**评分维度**：
- 策略种类（40%）
- 回滚速度（30%）
- 自动化程度（30%）

**常见错误**：
- 只依赖 Git revert（代码回滚不等于部署回滚，需要重新构建部署）。
- 未保留历史版本的构建产物，回滚时需重新构建（增加回滚时间）。
- 回滚后未验证服务是否正常，导致回滚失败未被发现。

**扩展追问**：
- Q: 回滚后如何处理已经写入异常数据（如错误的 localStorage 结构）的用户？
- Q: 数据库迁移（Migration）的回滚如何与前端的回滚协同？
- Q: Git revert 和 构建产物回滚有什么区别？各自的适用场景？

### 8. 如何用 Kubernetes 管理前端应用？Deployment、Service、Ingress 如何配置？

**参考答案**：

前端应用在 K8s 中通常以无状态 Deployment 运行，通过 Service 暴露，Ingress 配置域名和 TLS。

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  selector:
    matchLabels:
      app: frontend
  template:
    metadata:
      labels:
        app: frontend
    spec:
      containers:
      - name: nginx
        image: registry.example.com/frontend:1.2.3
        ports:
        - containerPort: 80
        livenessProbe:
          httpGet: { path: /healthz, port: 80 }
          initialDelaySeconds: 5
          periodSeconds: 10
        readinessProbe:
          httpGet: { path: /index.html, port: 80 }
          initialDelaySeconds: 3
          periodSeconds: 5
        resources:
          requests: { cpu: 100m, memory: 128Mi }
          limits:   { cpu: 500m, memory: 256Mi }
---
apiVersion: v1
kind: Service
metadata:
  name: frontend-svc
spec:
  selector:
    app: frontend
  ports:
  - port: 80
    targetPort: 80
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: frontend-ingress
spec:
  rules:
  - host: app.example.com
    http:
      paths:
      - pathType: Prefix
        path: "/"
        backend:
          service:
            name: frontend-svc
            port:
              number: 80
  tls:
  - hosts: [app.example.com]
    secretName: app-tls
```

关键点：滚动更新策略 `maxUnavailable: 0` 保证零停机，Readiness Probe 确保新 Pod 就绪才接入流量。

**评分维度**：
- K8s 资源理解（30%）
- 滚动更新配置（30%）
- 健康检查（20%）
- 资源限制（20%）

**常见错误**：
- 混淆 liveness 和 readiness probe 的作用。
- 未设置 resources requests/limits，导致资源争抢。
- 使用默认的 RollingUpdate 配置（可能 maxUnavailable 为 25% 导致短暂容量不足）。

**扩展追问**：
- Q: PodDisruptionBudget（PDB）的作用是什么？如何配置？
- Q: 如何利用 HPA（HorizontalPodAutoscaler）实现前端应用的自动扩缩容？

### 9. 如何设计前端性能监控与告警策略？

**参考答案**：

- **采集指标**：
  - Web Vitals：LCP（首屏加载）、INP（交互延迟）、CLS（布局偏移）。
  - 传统指标：FP、FCP、DOMContentLoaded、首屏耗时、白屏时间。
  - 资源指标：各类型资源加载耗时、CDN 命中率、DNS 解析时间。
  - API 指标：接口耗时 P50/P90/P99、成功率、慢接口 TOP N。
- **采集方式**：Performance API + PerformanceObserver + 自定义埋点，`navigator.sendBeacon` 上报。
- **聚合分析**：按版本、地域、设备、浏览器聚合，计算分位值，建立基线。
- **告警策略**：
  - P0：LCP > 4.5s（所有用户）、核心 API 成功率 < 90%。
  - P1：LCP 相比基线劣化 > 20%、CLS > 0.25、FCP > 3s。
  - P2：Bundle 大小环比增量 > 15%、第三方资源加载异常。
  - 告警抑制：排除已知的夜间维护窗口、大促峰值。

**评分维度**：
- 指标覆盖（30%）
- 采集方式（20%）
- 分析方法（20%）
- 告警分级与抑制（30%）

**常见错误**：
- 只关注 LCP 忽视 INP 和 CLS，用户体验评估不全面。
- 未排除异常值（只上报平均值，未用 P50/P75/P90/P99）。
- 告警阈值没有基线，凭经验设定导致告警风暴或漏报。

**扩展追问**：
- Q: 如何建立性能基线？版本间的性能 regression 如何自动化检测？
- Q: LAB 数据（Lighthouse）和 FIELD 数据（RUM）的差异以及各自的应用场景？
- Q: 长任务（Long Task）和 INP 的关系是什么？

### 10. 前端多环境配置管理与环境隔离的最佳实践？

**参考答案**：

- **配置文件方案**：使用 `.env.development`、`.env.staging`、`.env.production`，构建时注入 `process.env` 变量。
- **运行时变量**：部分配置（如接口域名）通过运行时 JS 变量注入而非构建时硬编码，同一构建产物可部署到不同环境。
- **环境隔离**：
  - 开发环境：Mock 或 dev API，热更新。
  - 测试环境：测试 API，启用调试日志。
  - 预发环境：与生产环境配置一致（数据库、缓存独立），仅流量差异。
  - 生产环境：正式 API，关闭调试，CDN 加速。
- **密钥管理**：敏感配置（API Key、证书）使用 Secret 管理（Vault/K8s Secret），不提交到代码仓库。

**评分维度**：
- 配置管理策略（30%）
- 构建时 vs 运行时注入（25%）
- 环境隔离粒度（25%）
- 安全性（20%）

**常见错误**：
- 将 API 地址等环境变量硬编码在代码中。
- 将敏感信息提交到 Git 仓库，造成安全泄露。
- 多环境配置差异过大导致预发验证失效（"在我机器上没问题"）。

**扩展追问**：
- Q: 编译时注入（webpack DefinePlugin）和运行时注入（window.__ENV__）各自的优缺点？
- Q: 如何使用 K8s ConfigMap 管理前端 Nginx 配置？
- Q: 不可变构建如何实现——如何用同一构建产物部署到不同环境？

### 11. 多环境晋升（Promotion）策略如何设计？如何保证生产环境零停机？

**参考答案**：

- **晋升原则**：同一构建产物逐环境晋升，代码只构建一次，环境差异仅通过配置文件隔离。
- **晋升流程**：
  1. Develop 分支构建 -> Dev 环境部署，开发者自测。
  2. 合入 Release 分支 -> Staging 部署，QA 全量测试。
  3. 晋升镜像 Tag（`frontend:staging-123` -> `frontend:preprod-123`）-> Pre-prod 部署，回归验证。
  4. 晋升镜像 Tag（`frontend:preprod-123` -> `frontend:prod-123`）-> 灰度发布生产。
- **零停机保障**：
  - Deployment rollingUpdate `maxUnavailable: 0`，确保始终有 Pod 服务。
  - Readiness Probe 确保新 Pod 就绪后才接流量。
  - PreStop Hook 优雅退出，等待现有请求处理完。
  - PDB 保证最少可用 Pod 数（`minAvailable: 2`）。

**评分维度**：
- 晋升流程设计（30%）
- 不可变构建（25%）
- 零停机技术（25%）
- 环境一致性（20%）

**常见错误**：
- 每个环境重新构建，无法保证"在测试环境通过的在生产也相同"。
- 晋升过程中只关注了前端，忽略了后端 API 版本兼容性。
- 未配置 PreStop Hook，滚动更新时正在处理的请求被断连。

**扩展追问**：
- Q: 构建产物的版本化策略——镜像 tag 用语义版本还是 Git commit SHA？
- Q: 前端晋升（静态资源）和后端晋升（API 服务）如何协调？
- Q: 如何自动化晋升流程中的人工审批环节？

### 12. 灰度发布中如何做好指标观测和自动化熔断？

**参考答案**：

- **观测指标**：
  - 错误指标：JS Error 率、白屏率、API 错误率、资源加载失败率。
  - 性能指标：LCP、FCP、CLS、INP 的分位值。
  - 业务指标：PV/UV、转化率、核心流程完成率。
  - 基础设施：CDN 命中率、带宽、响应状态码分布。
- **对比方式**：灰度组 vs 对照组（上一版本或同版本的存量用户），计算指标的统计显著性。
- **自动化熔断机制**：
  - 熔断触发条件：任一核心指标超过阈值（如 JS Error 率 > 3%，且相比对照组超过 2x）。
  - 熔断动作：停止灰度放量，将已灰度用户自动切回旧版本。
  - 熔断通知：即时通知到 SRE 值班群。
  - 熔断恢复：人工确认后重新灰度。

**评分维度**：
- 指标全面性（30%）
- 对比方法（20%）
- 熔断策略（30%）
- 自动化程度（20%）

**常见错误**：
- 只监控了错误率，忽略了性能劣化（用户虽然能打开但体验变差）。
- 灰度组和对照组样本量差异过大导致对比不公平。
- 熔断阈值设置太严格（频繁误熔断）或太宽松（形同虚设）。

**扩展追问**：
- Q: 灰度发布的最小样本量如何计算？如何确定有统计意义的对比结果？
- Q: 熔断阈值如何基于历史数据动态调整？
- Q: 灰度过程中如何保证数据库或后端兼容性？

### 13. 如何将安全扫描集成到前端 CI/CD 流水线中？

**参考答案**：

- **依赖扫描**：
  - `npm audit` 扫描已知漏洞，CI 中配置 `--audit-level=high` 阻断高危漏洞。
  - Snyk / Dependabot 持续监控，自动创建修复 PR。
  - Trivy 扫描 Docker 镜像中的 OS 级漏洞。
- **代码扫描**：
  - ESLint 安全规则插件（`eslint-plugin-security`）检测不安全模式。
  - SonarQube 静态代码分析，检测注入、XSS 风险。
- **密钥泄露检测**：
  - git-secrets / truffleHog 扫描提交历史中的敏感信息。
  - pre-commit hook 检测密钥泄露。
- **供应链安全**：
  - `package-lock.json` 锁定依赖版本，防止恶意篡改。
  - 验证 npm 包的完整性（`npm audit signatures`）。
  - 使用私有 npm registry 或代理缓存，控制依赖来源。

**评分维度**：
- 扫描类型覆盖（30%）
- CI 集成方式（25%）
- 安全门禁配置（25%）
- 供应链安全（20%）

**常见错误**：
- 只在 CI 中运行 `npm audit` 但未阻断构建，高危漏洞仍然上线。
- 未对 Docker 镜像做漏洞扫描，基础镜像中的 CVE 被带入生产。
- 依赖锁定文件（lockfile）未提交到 Git，导致每次安装版本不一致。

- Q: 如何保证从 npm 公共源下载的包未被篡改？

### 14. 如何保障大促期间前端稳定性？

**参考答案要点**：

1. **提前压测和容量评估**：使用 Lighthouse 批量跑分，k6 模拟高并发访问静态资源，确定 CDN 和源站的容量瓶颈。
2. **CDN 预热和带宽扩容**：提前将大促资源预热到 CDN 全节点，联系 CDN 厂商临时扩容带宽。
3. **多 CDN 容灾**：配置主备 CDN，任一 CDN 故障时自动切换（通过 DNS 权重或 HTTP DNS）。
4. **限流降级**：后端 API 限流降级时前端兜底——展示降级页面、缓存数据、离线容灾。
5. **静态化**：核心页面（如活动页）提前静态化部署到 CDN，不依赖动态 API。
6. **监控与预案**：大促专项监控大盘，预演故障场景，SRE 24 小时值班。

**评分维度**：
- 预防措施（30%）
- 容量保障（20%）
- 容灾降级（25%）
- 监控响应（15%）
- 预案完整性（10%）

**常见错误**：
- 只关注扩容（加机器）而忽略了架构层面的限流降级。
- 大促前的压测只测了后端，未对前端静态资源做压力测试。
- 没有降级预案，后端一旦异常前端直接白屏。

**扩展追问**：
- Q: 大促期间如果 CDN 带宽打满，如何紧急降低带宽占用？
- Q: 静态页面降级方案如何设计？Service Worker 可以做离线降级吗？
- Q: 如何提前"模拟"大促流量做全链路压测？

### 15. 设计一个前端零停机发布方案。

**参考答案要点**：

1. **构建产物版本化**：所有静态资源以版本目录组织（`/v1.2.3/`），保留历史版本。
2. **入口控制**：网关或 Edge Function 根据版本策略分发 index.html，支持灰度。
3. **缓存的精妙控制**：带 hash 的 JS/CSS 使用 `immutable` 缓存，HTML 使用 `no-cache`。
4. **滚动发布流程**：K8s Deployment RollingUpdate（`maxUnavailable: 0`），新增 Pod 就绪后再摘除旧 Pod。
5. **蓝绿部署**：切流前保持绿环境运行，蓝环境验证通过后切换流量，异常时立即切回。
6. **PreStop Hook**：Nginx 优雅关闭，确保正在处理的请求完成。
7. **实时验证**：切换后自动化冒烟测试 + 监控指标对比。

**评分维度**：
- 版本管理（25%）
- 流量控制（25%）
- 灰度策略（20%）
- 回滚能力（20%）
- 用户体验（10%）

**常见错误**：
- 只关注前端服务零停机，忽略了后端 API 新老版本兼容问题。
- 切换流量时大规模刷新 CDN 缓存，导致回源风暴压垮源站。
- 没有自动化验证步骤，全量切完后才发现问题。

**扩展追问**：

### 16. 如何设计前端应用的多集群/多活部署架构？

**参考答案**：

- **架构目标**：任一区域故障不影响整体服务可用性，RTO < 1 分钟，RPO ≈ 0。
- **部署拓扑**：
  - 区域 A（华北）：CDN 主节点 + K8s 集群 + OSS 存储。
  - 区域 B（华东）：CDN 备用节点 + K8s 集群 + OSS 存储。
  - 全局流量调度层（HTTP DNS 或 GSLB），按用户地理位置分配最近节点。
- **数据同步**：
  - 静态资源：各区域 OSS 独立存储，跨区域同步（OSS 跨区域复制）。
  - 构建产物发布时自动推送到所有区域 OSS。
  - 用户会话数据：无状态设计（JWT Token），不需跨区域同步。
- **故障切换**：
  - 单区域故障：DNS 切换将流量全部导向健康区域。
  - CDN 故障：自动切换到备用 CDN 或直接回源到 OSS。
- **一致性保障**：
  - 使用不可变部署（同一版本在所有区域部署相同资源）。
  - 发布时先灰度一个区域，验证通过后再推其他区域。

**评分维度**：
- 架构完整性（30%）
- 故障切换机制（25%）
- 数据同步策略（20%）
- 部署一致性（25%）

**常见错误**：
- 认为前端不需要多活——静态资源 404 同样导致业务中断。
- 只考虑跨区域容灾，未考虑跨 CDN 厂商容灾。
- 各区域部署时间不一致，导致用户在区域间切换时拿到不同版本。

**扩展追问**：
- Q: GSLB 和 HTTP DNS 的区别？TTL 如何设置才能在故障时快速切换？
- Q: 多区域部署时发布节奏如何控制？如何防止"脑裂"？
- Q: 边缘渲染（Edge Rendering）在多活架构中的作用？

### 17. 如何为前端应用定义和度量 SLO？如何设计告警预算？

**参考答案**：

- **SLO 指标选取**：
  - 可用性 SLO：首页 HTTP 状态码成功率 >= 99.95%（排除客户端网络问题）。
  - 性能 SLO：LCP P75 < 2.5s，INP P75 < 200ms，CLS P75 < 0.1。
  - 错误 SLO：JS Error 率（用户影响面）< 1%。
  - 发布 SLO：灰度到全量平均耗时 < 2h，回滚完成 < 5min。
- **SLI 采集**：
  - 前端 RUM 数据（Performance API + 自定义埋点）作为主要 SLI 数据源。
  - 合成监控（Synthetic Monitoring）定期拨测作为辅助验证。
- **错误预算（Error Budget）**：
  - 每月错误预算 = (1 - SLO) * 总请求量 / 时间。
  - 例如 99.95% 可用性 SLO，每月允许不可用时间 = 43200 \* 0.05% ≈ 21.6 分钟。
  - 错误预算消耗超过 50% 时冻结新功能发布，优先修复稳定性。
- **告警策略**：
  - 燃尽率告警（Burn Rate）：短期错误预算消耗过快时即时告警。
  - 多窗口告警：1h/6h/3d 多时间窗口检测，平衡响应速度和误报率。

**评分维度**：
- SLI/SLO 定义（30%）
- 度量方法（25%）
- 错误预算机制（25%）
- 告警策略（20%）

**常见错误**：
- SLI 定义不清晰（如"页面打开快"没有量化标准）。
- SLO 设置过于严格（99.999%）导致错误预算为 0，任何发布都被阻塞；或过于宽松形同虚设。
- 只定义了 SLO 没有配套的错误预算和行动机制。

**扩展追问**：

- Q: 如何区分"客户端网络问题"和"服务端故障"对 SLO 的影响？
- Q: 错误预算耗尽后，团队如何平衡新功能开发和稳定性投入？

### 18. 什么是前端混沌工程？如何在前端实施混沌实验？

**参考答案**：

- **概念**：混沌工程是通过主动注入故障，验证系统在异常条件下的韧性（Resilience）。前端混沌工程关注 CDN 故障、API 延迟/错误、第三方 SDK 异常、资源加载失败等场景。
- **实验方法**：
  - **CDN 降级实验**：通过浏览器扩展或代理模拟 CDN 资源 404，验证 fallback 策略是否生效。
  - **API 故障注入**：Mock Service Worker（MSW）或代理工具（Charles）模拟 API 超时/5xx，验证前端降级和重试逻辑。
  - **第三方 SDK 异常**：拦截第三方脚本加载失败，验证核心流程不被阻断。
  - **网络条件模拟**：Chrome DevTools 模拟弱网（3G/Offline），验证离线体验和加载表现。
  - **内存压力测试**：模拟低内存设备，验证页面是否 OOM 崩溃。
- **工具链**：Chaos Mesh（K8s 层注入）、Gremlin、自建 Chrome 扩展 + Puppeteer 自动化。
- **原则**：逐步扩大爆炸半径、实验前确定假设和度量指标、实验后自动化回滚。

**评分维度**：
- 对混沌工程的理解（25%）
- 实验场景设计（30%）
- 工具与方法（25%）
- 安全性和自动化（20%）

**常见错误**：
- 把简单的"浏览器降级测试"等同于混沌工程——缺乏系统性和可重复性。
- 在生产环境直接做实验没有回滚方案（前端的 chaos engineering 推荐从预发开始）。
- 只关注了网络故障，忽略了内存、CPU、第三方依赖等维度。
