# 前端部署与运维（SRE）练习册

> 通过练习掌握部署策略、可观测性和容灾能力。

---

## 难度分级

- 🟢 基础：理解概念。
- 🟡 进阶：能选择合适策略。
- 🔴 深入：能设计完整 SRE 体系。

---

## 一、选择题

### 第 1 题（🟢）

前端静态资源最适合长期缓存的是？

A. HTML  
B. 带 hash 的 JS/CSS  
C. API 响应  
D. 用户上传图片

### 第 2 题（🟢）

金丝雀发布的特点是？

A. 一次性全量上线  
B. 小范围验证后逐步扩大  
C. 同时运行两套独立环境  
D. 只给内部员工使用

### 第 3 题（🟡）

可观测性三大支柱不包括？

A. 日志  
B. 指标  
C. 链路追踪  
D. 单元测试

### 第 4 题（🟡）

RTO 表示？

A. 恢复点目标  
B. 恢复时间目标  
C. 恢复资源目标  
D. 恢复流量目标

### 第 5 题（🔴）

以下哪种方式最能降低前端发布风险？

A. 全量发布  
B. 灰度发布 + 实时监控 + 一键回滚  
C. 凌晨手动发布  
D. 不做测试直接发布

## 参考答案

### 第 1 题

**答案：B**

带 hash 的 JS/CSS 文件名变化即失效，可长期缓存。

### 第 2 题

**答案：B**

金丝雀发布先小范围验证，再逐步扩大流量。

### 第 3 题

**答案：D**

可观测性三大支柱是日志、指标、链路追踪。

### 第 4 题

**答案：B**

RTO（Recovery Time Objective）是恢复时间目标。

### 第 5 题

**答案：B**

灰度发布 + 监控 + 回滚是降低发布风险的最佳实践。

### 第 6 题

**参考思路**：
1. 查看错误监控系统，定位白屏用户设备和浏览器。
2. 检查 CDN 是否刷新、资源 404。
3. 查看 JS 执行错误，可能是兼容性问题。
4. 快速关闭灰度或回滚版本。
5. 修复后重新灰度验证。

### 第 7 题

**参考思路**：
- 原因：CDN 带宽不足、源站压力过大、资源体积过大、DNS 解析慢。
- 优化：预热 CDN、启用 HTTP/2 或 HTTP/3、压缩资源、开启 Brotli、增加边缘缓存。

### 第 8 题

**参考思路**：
- 保留最近 N 个版本的构建产物。
- 通过配置中心切换版本路由。
- CDN 刷新回旧版本资源。
- 自动化脚本：切换路由 + 验证 + 告警。

### 第 9 题

**参考策略**：
- HTML：no-cache，确保入口最新。
- JS/CSS（带 hash）：1 年长期缓存。
- 图片/字体：1 年长期缓存。
- API 数据：按业务场景设置短缓存或协商缓存。

### 第 10 题

**参考设计**：
- 配置中心维护灰度规则。
- 网关或 CDN 边缘函数根据规则分流。
- 规则类型：用户 ID 取模、地域、设备、随机比例。
- 实时监控灰度组指标，异常自动熔断。

### 第 11 题

**参考体系**：
- 错误：JS 错误、接口错误、资源加载错误。
- 性能：Web Vitals、FP/FCP/LCP/INP/CLS。
- 业务：转化率、白屏率、核心流程成功率。
- 告警：按阈值分级，P0 立即通知，P1/P2 延时通知。
