### 1. 什么是 SOLID 原则？请结合前端场景分别说明。

**问题**：请解释 SOLID 五个原则的含义，并结合前端开发的实际场景举例说明每个原则如何应用。

**参考答案**：

SOLID 是五个面向对象设计原则的首字母缩写：

**S — 单一职责原则（SRP）**
一个类/模块只负责一个职责。在前端中，一个组件只做一件事。
```typescript
// 违反 SRP：一个组件既获取数据又渲染表格又处理导出
function BadUserTable() {
  const [users, setUsers] = useState([]);
  useEffect(() => { fetch("/api/users").then(setUsers); }, []);
  const exportCSV = () => { /* ... */ };
  return <table>{users.map(u => <tr><td>{u.name}</td></tr>)}</table>;
}

// 符合 SRP：拆分自定义 Hook、纯展示组件、工具函数
function useUsers() { /* 只负责数据获取 */ }
function UserTable({ users }: { users: User[] }) { /* 只负责渲染 */ }
function exportCSV(users: User[]) { /* 只负责导出 */ }
```

**O — 开闭原则（OCP）**
对扩展开放，对修改关闭。React 组合模式是最佳实践。
```typescript
// 通过 render props / children 扩展，不修改内部实现
function List<T>({ items, renderItem }: { items: T[]; renderItem: (item: T) => ReactNode }) {
  return <ul>{items.map(renderItem)}</ul>;
}
// 外部扩展新渲染方式，List 本身无需修改
<List items={users} renderItem={u => <li>{u.name}</li>} />
```

**L — 里氏替换原则（LSP）**
子类型必须能替换基类型。React 中组件 props 类型应遵循该原则。
```typescript
interface BaseProps { title: string; children?: ReactNode; }
function BaseCard(props: BaseProps) { return <div>{props.title}{props.children}</div>; }
// ExtendedCard 可以替换 BaseCard 的位置，且不破坏原有功能
function ExtendedCard(props: BaseProps) { return <div><h2>{props.title}</h2>{props.children}</div>; }
```

**I — 接口隔离原则（ISP）**
不应强迫客户端依赖它不用的接口。在前端体现为 props 应该精简。
```typescript
// 违反 ISP：传入大量不需要的 props
interface ObeseProps { name: string; age: number; email: string; phone: string; address: string; }

// 符合 ISP：最小化接口，按需拆分
interface UserNameProps { name: string; }
interface UserContactProps { email: string; phone: string; }
```

**D — 依赖倒置原则（DIP）**
依赖抽象而非具体实现。在前端中体现为依赖注入。
```typescript
// 依赖抽象：依赖于"存储接口"而非具体 localStorage
interface Storage {
  get(key: string): Promise<string | null>;
  set(key: string, value: string): Promise<void>;
}

class UserService {
  constructor(private storage: Storage) {} // 注入抽象
}

// 可以注入 localStorage 实现、内存实现或 IndexedDB 实现
class LocalStorageImpl implements Storage { /* ... */ }
class MemoryStorageImpl implements Storage { /* ... */ }
```

**评分维度**：
- SRP 理解与举例（20%）
- OCP 理解与举例（20%）
- LSP 理解与举例（20%）
- ISP 理解与举例（20%）
- DIP 理解与举例（20%）

**常见错误**：
- 机械背诵原则定义，不会结合前端场景举例
- 将 LSP 理解为"子类必须完全一模一样"，混淆 LSP 和接口继承
- 认为 DIP 就是"依赖注入"，其实 DI 只是 DIP 的一种实现方式
- 在 React 中用高阶组件过度包装，反而违反 OCP

**扩展追问**：
- 在实际项目中，你如何平衡 SRP 和组件数量？过度拆分也会带来问题吗？
- 前端中哪些框架或库的 API 设计体现了 OCP？请举例。
- TypeScript 的 interface 和 type 与 ISP 有何关系？

### 2. 单例模式有什么优缺点？在前端中如何正确实现？

**问题**：请分析单例模式的优缺点，并给出在前端场景下的最佳实践实现方式。

**参考答案**：

单例模式确保一个类只有一个实例，并提供全局访问点。

**优点**：
- 全局唯一实例，节省内存资源
- 延迟初始化，按需创建
- 提供全局状态管理入口（如 Vuex / Redux store）

**缺点**：
- 隐藏依赖关系，代码可测试性差
- 违反 SRP（既管理实例又执行业务）
- 难以扩展和替换实现
- 全局状态可能导致隐式耦合

**前端最佳实践实现**：

```typescript
// 方式一：ES Module 天然单例（推荐）
// store.ts
export const store = {
  state: new Map<string, any>(),
  get(key: string) { return this.state.get(key); },
  set(key: string, val: any) { this.state.set(key, val); },
};
// ES Module 只会执行一次，import 共享同一实例

// 方式二：类 + 静态属性
class Logger {
  private static instance: Logger;
  private constructor() {} // 阻止外部 new

  static getInstance(): Logger {
    if (!Logger.instance) {
      Logger.instance = new Logger();
    }
    return Logger.instance;
  }

  log(msg: string) { console.log(`[App] ${msg}`); }
}

// 方式三：在 React 中用 useRef 保证组件内单例
function useSingleton<T>(factory: () => T): T {
  const ref = useRef<T | null>(null);
  if (!ref.current) {
    ref.current = factory();
  }
  return ref.current;
}
```

**评分维度**：
- 优点分析全面性（25%）
- 缺点分析全面性（25%）
- 实现代码质量（30%）
- 前端场景举例（20%）

**常见错误**：
- 只谈优点不谈缺点，或只说"不利于测试"但不解释原因
- 用全局变量代替单例，不考虑模块作用域
- 在 Node.js 中忽略模块缓存机制
- 多线程/SSR 场景下忽略竞态条件

### 3. 观察者模式和发布订阅模式有什么区别？分别在什么场景下使用？

**问题**：请从结构、耦合度、实现方式三个维度对比观察者模式和发布订阅模式的异同。

**参考答案**：

**观察者模式（Observer Pattern）**
目标和观察者直接通信，目标维护观察者列表。
```typescript
// 观察者模式：Subject 直接通知 Observer
interface Observer {
  update(data: any): void;
}

class Subject {
  private observers: Observer[] = [];

  attach(obs: Observer) { this.observers.push(obs); }
  detach(obs: Observer) {
    this.observers = this.observers.filter(o => o !== obs);
  }
  notify(data: any) {
    for (const obs of this.observers) {
      obs.update(data); // 直接调用
    }
  }
}
```

**发布订阅模式（Pub-Sub Pattern）**
发布者和订阅者通过事件中心桥接，彼此不知道对方存在。
```typescript
// 发布订阅模式：通过 EventEmitter 解耦
type Handler = (...args: any[]) => void;

class EventEmitter {
  private handlers = new Map<string, Handler[]>();

  on(event: string, handler: Handler) {
    if (!this.handlers.has(event)) {
      this.handlers.set(event, []);
    }
    this.handlers.get(event)!.push(handler);
  }

  off(event: string, handler: Handler) {
    const handlers = this.handlers.get(event);
    if (handlers) {
      this.handlers.set(event, handlers.filter(h => h !== handler));
    }
  }

  emit(event: string, ...args: any[]) {
    const handlers = this.handlers.get(event);
    if (handlers) {
      handlers.forEach(handler => handler(...args));
    }
  }

  removeAllListeners(event?: string) {
    if (event) {
      this.handlers.delete(event);
    } else {
      this.handlers.clear();
    }
  }
}
```

**核心区别**：

| 维度 | 观察者模式 | 发布订阅模式 |
|------|-----------|-------------|
| 耦合度 | 目标和观察者直接耦合 | 完全解耦，通过事件中心 |
| 通信方式 | 直接调用 | 间接通过调度中心 |
| 角色 | Subject + Observer | Publisher + Broker + Subscriber |
| 复杂度 | 简单 | 中等，但更灵活 |
| 适用场景 | 关注对象变化 | 跨层级/跨模块通信 |

**评分维度**：
- 结构差异描述（30%）
- 耦合度分析（25%）
- 代码实现质量（25%）
- 场景举例能力（20%）

**常见错误**：
- 认为两者完全等同，没有区分耦合度差异
- 实现时忽略取消订阅（导致内存泄漏）
- 混淆了"事件驱动"和"发布订阅"的概念范围
- 没有理解观察者模式在 RxJS 中的应用

**扩展追问**：
- Vue 的响应式系统是观察者模式还是发布订阅？为什么？
- Redux 是哪种模式？dispatch / subscribe 属于什么？
- EventEmitter 中如何实现 once？（只触发一次）

### 4. 简单工厂、工厂方法和抽象工厂有什么区别？

**问题**：请对比三种工厂模式的异同，并给出前端开发中的实际应用场景。

**参考答案**：

**简单工厂（Simple Factory）**
通过一个工厂函数创建产品，封装创建逻辑。
```typescript
// 简单工厂
type ButtonTheme = "primary" | "secondary" | "danger";

function createButton(theme: ButtonTheme) {
  switch (theme) {
    case "primary":
      return { type: "button", className: "btn-primary", onClick: () => {} };
    case "secondary":
      return { type: "button", className: "btn-secondary", onClick: () => {} };
    case "danger":
      return { type: "button", className: "btn-danger", onClick: () => {} };
  }
}
```

**工厂方法（Factory Method）**
将实例化延迟到子类，通过继承改变创建的产品。
```typescript
// 工厂方法模式
interface UIFactory {
  createButton(): Button;
  createDialog(): Dialog;
}

// 具体工厂创建具体产品族
class MaterialUIFactory implements UIFactory {
  createButton() { return new MaterialButton(); }
  createDialog() { return new MaterialDialog(); }
}

class AntdFactory implements UIFactory {
  createButton() { return new AntdButton(); }
  createDialog() { return new AntdDialog(); }
}
```

**抽象工厂（Abstract Factory）**
创建相关或依赖对象的家族，而不指定具体类。
```typescript
// 抽象工厂：创建主题相关的整套组件
interface ThemeFactory {
  createButton(): Button;
  createInput(): Input;
  createModal(): Modal;
  createDropdown(): Dropdown;
}

class DarkThemeFactory implements ThemeFactory {
  createButton() { return new DarkButton(); }
  createInput() { return new DarkInput(); }
  createModal() { return new DarkModal(); }
  createDropdown() { return new DarkDropdown(); }
}

class LightThemeFactory implements ThemeFactory {
  createButton() { return new LightButton(); }
  createInput() { return new LightInput(); }
  createModal() { return new LightModal(); }
  createDropdown() { return new LightDropdown(); }
}
```

**核心区别**：

| 模式 | 粒度 | 扩展方式 | 前端场景 |
|------|------|---------|---------|
| 简单工厂 | 单个产品 | 修改工厂函数 | Toast/Notification 创建 |
| 工厂方法 | 单个产品 | 继承子类 | 多平台 UI 组件创建 |
| 抽象工厂 | 产品家族 | 实现新工厂 | 主题系统、组件库换肤 |

**评分维度**：
- 三种模式定义准确（30%）
- 代码实现差异（30%）
- 前端场景举例（25%）
- 优缺点对比（15%）

**常见错误**：
- 混淆工厂方法和抽象工厂，不理解"产品族"概念
- 认为工厂就是消除 new 关键字，不理解其解耦价值
- 在前端中过度使用工厂模式，增加不必要的抽象层

**扩展追问**：
- React.createElement 是哪种工厂模式？
- 前端主题系统适合用抽象工厂吗？Vue / React 如何实现主题化？

### 5. 装饰器模式（Decorator）和代理模式（Proxy）有什么区别？

**问题**：请从意图、结构、实现三个维度对比装饰器模式和代理模式，并给出前端实际应用。

**参考答案**：

**装饰器模式**：动态地为对象添加新职责。
**代理模式**：为对象提供替代品或占位符以控制对它的访问。

**核心区别**：

| 维度 | 装饰器模式 | 代理模式 |
|------|-----------|---------|
| 意图 | 增强功能 | 控制访问 |
| 关系 | 递归组合，多层包装 | 代理持有真实对象引用 |
| 创建时机 | 运行时动态添加 | 编译时/运行时 |
| 客户端感知 | 透明，客户端对装饰无感 | 客户端可能感知到代理 |
| 关注点 | "添加"行为 | "控制"行为 |

**装饰器模式实现**：
```typescript
// 装饰器模式：给组件层层添加能力
interface Component {
  render(): string;
}

class TextInput implements Component {
  render() { return "<input type='text' />"; }
}

// 装饰器基类
class InputDecorator implements Component {
  constructor(protected component: Component) {}
  render() { return this.component.render(); }
}

// 具体装饰器
class ValidationDecorator extends InputDecorator {
  render() {
    return `<div class="validate">${super.render()}</div>`;
  }
}

class TooltipDecorator extends InputDecorator {
  render() {
    return `<div class="tooltip-wrapper">${super.render()}<span>?</span></div>`;
  }
}

// 使用：可以任意组合
const input = new TooltipDecorator(new ValidationDecorator(new TextInput()));
```

**代理模式实现**：
```typescript
// 代理模式：控制访问
interface Image {
  display(): void;
}

class RealImage implements Image {
  constructor(private url: string) {
    this.loadFromServer();
  }
  private loadFromServer() { console.log(`Loading: ${this.url}`); }
  display() { console.log(`Displaying: ${this.url}`); }
}

class ImageProxy implements Image {
  private realImage: RealImage | null = null;

  constructor(private url: string) {}

  display() {
    // 延迟加载：首次访问才创建真实对象
    if (!this.realImage) {
      this.realImage = new RealImage(this.url);
    }
    this.realImage.display();
  }
}
```

**前端应用场景**：
- **装饰器**：HOC（React Higher-Order Component）、Angular/Python 装饰器语法、中间件链
- **代理**：虚拟代理（图片懒加载）、保护代理（权限控制）、缓存代理（API 响应缓存）、ES6 Proxy 对象

**评分维度**：
- 意图区别理解（30%）
- 代码实现质量（30%）
- 前端场景举例（25%）
- 优缺点分析（15%）

**常见错误**：
- 认为"装饰器和代理结构一样所以是一样的"
- 不理解装饰器可以多层嵌套而代理通常只有一层
- 把 AOP（面向切面编程）完全等同于装饰器
- 混淆 ES6 Proxy 和装饰器模式的概念

**扩展追问**：

### 6. 如何在前端设计一个可扩展的插件系统？

**问题**：请设计一个插件系统架构，支持插件注册、生命周期管理和依赖管理，给出核心接口定义和实现。

**参考答案**：

一个良好的前端插件系统应包含：插件注册、生命周期钩子、插件上下文、依赖管理。

```typescript
// 插件核心接口
interface PluginContext {
  app: App;
  config: Record<string, any>;
  logger: Logger;
}

interface Plugin {
  name: string;
  version: string;
  dependencies?: string[];  // 声明依赖的其他插件

  // 生命周期钩子
  onInstall?(ctx: PluginContext): void | Promise<void>;
  onUninstall?(ctx: PluginContext): void | Promise<void>;
  onActivate?(ctx: PluginContext): void | Promise<void>;
  onDeactivate?(ctx: PluginContext): void | Promise<void>;

  // 扩展点：允许插件向应用注册功能
  registerHooks?(): Record<string, Function>;
}

// 插件管理器
class PluginManager {
  private plugins = new Map<string, Plugin>();
  private hooks = new Map<string, Set<Function>>();
  private ctx: PluginContext;

  constructor(app: App) {
    this.ctx = { app, config: {}, logger: new Logger() };
  }

  async install(plugin: Plugin) {
    // 依赖检查
    if (plugin.dependencies) {
      for (const dep of plugin.dependencies) {
        if (!this.plugins.has(dep)) {
          throw new Error(`Missing dependency: ${dep} for ${plugin.name}`);
        }
      }
    }

    this.plugins.set(plugin.name, plugin);
    await plugin.onInstall?.(this.ctx);

    // 注册扩展钩子
    if (plugin.registerHooks) {
      const pluginHooks = plugin.registerHooks();
      for (const [hookName, handler] of Object.entries(pluginHooks)) {
        if (!this.hooks.has(hookName)) {
          this.hooks.set(hookName, new Set());
        }
        this.hooks.get(hookName)!.add(handler);
      }
    }
  }

  async uninstall(name: string) {
    const plugin = this.plugins.get(name);
    if (!plugin) return;
    await plugin.onUninstall?.(this.ctx);
    this.plugins.delete(name);
  }

  // 触发扩展点
  async invokeHook(hookName: string, ...args: any[]) {
    const handlers = this.hooks.get(hookName);
    if (!handlers) return;
    for (const handler of handlers) {
      await handler(...args);
    }
  }

  async activate(name: string) {
    await this.plugins.get(name)?.onActivate?.(this.ctx);
  }

  async deactivate(name: string) {
    await this.plugins.get(name)?.onDeactivate?.(this.ctx);
  }
}

// 使用示例
const analyticsPlugin: Plugin = {
  name: "analytics",
  version: "1.0.0",
  onInstall(ctx) {
    console.log(`Analytics plugin installed in ${ctx.app.name}`);
  },
  registerHooks() {
    return {
      "page:view": (page: string) => {
        // 发送页面浏览事件
        console.log(`Track page view: ${page}`);
      },
    };
  },
};
```

**评分维度**：
- 插件接口设计（25%）
- 生命周期管理（20%）
- 依赖管理（15%）
- 扩展点/Hook 机制（25%）
- 错误处理与隔离（15%）

**常见错误**：
- 插件间紧耦合，没有通过事件/钩子解耦

### 7. 实现一个类型安全的 EventEmitter。

**问题**：请实现一个 EventEmitter，支持 on、off、emit、once 方法，并提供 TypeScript 类型安全支持。

**参考答案**：

```typescript
// 类型定义：事件名到参数类型的映射
type EventMap = Record<string, any[]>;

// 类型安全的 EventEmitter
class TypedEventEmitter<Events extends EventMap> {
  private listeners = new Map<keyof Events, Set<(...args: any[]) => void>>();
  private onceListeners = new Map<keyof Events, Set<(...args: any[]) => void>>();

  on<E extends keyof Events>(
    event: E,
    listener: (...args: Events[E]) => void
  ): () => void {
    if (!this.listeners.has(event)) {
      this.listeners.set(event, new Set());
    }
    this.listeners.get(event)!.add(listener as (...args: any[]) => void);

    // 返回 unsubscribe 函数
    return () => this.off(event, listener);
  }

  off<E extends keyof Events>(
    event: E,
    listener: (...args: Events[E]) => void
  ): void {
    this.listeners.get(event)?.delete(listener as (...args: any[]) => void);
    this.onceListeners.get(event)?.delete(listener as (...args: any[]) => void);
  }

  emit<E extends keyof Events>(event: E, ...args: Events[E]): void {
    // 执行普通监听器
    this.listeners.get(event)?.forEach(listener => {
      listener(...args);
    });

    // 执行 once 监听器，执行后立即移除
    this.onceListeners.get(event)?.forEach(listener => {
      listener(...args);
    });
    this.onceListeners.delete(event);
  }

  once<E extends keyof Events>(
    event: E,
    listener: (...args: Events[E]) => void
  ): () => void {
    if (!this.onceListeners.has(event)) {
      this.onceListeners.set(event, new Set());
    }
    this.onceListeners.get(event)!.add(listener as (...args: any[]) => void);
    return () => this.off(event, listener);
  }

  removeAllListeners(event?: keyof Events): void {
    if (event) {
      this.listeners.delete(event);
      this.onceListeners.delete(event);
    } else {
      this.listeners.clear();
      this.onceListeners.clear();
    }
  }

  listenerCount(event: keyof Events): number {
    return (this.listeners.get(event)?.size ?? 0)
      + (this.onceListeners.get(event)?.size ?? 0);
  }
}
```

```typescript
interface MiddlewareContext {
  request: Request;
  response: Response;
  [key: string]: any; // 允许中间件添加自定义属性

// 中间件类型：可同步或异步
type Middleware = (
  ctx: MiddlewareContext,
  next: () => Promise<void>
) => Promise<void> | void;

class MiddlewareChain {
  private middlewares: Middleware[] = [];

  use(mw: Middleware): this {
    this.middlewares.push(mw);
    return this;
  }

  // 在指定位置插入中间件
  insertBefore(target: Middleware, mw: Middleware): boolean {
    const idx = this.middlewares.indexOf(target);
    if (idx === -1) return false;
    this.middlewares.splice(idx, 0, mw);
    return true;
  }

  // 组合中间件为单一处理函数
  compose(): (ctx: MiddlewareContext) => Promise<void> {
    return async (ctx: MiddlewareContext) => {
      let index = -1;

      const dispatch = async (i: number): Promise<void> => {
        if (i <= index) {
          throw new Error("next() called multiple times");
        }
        index = i;

        const middleware = this.middlewares[i];
        if (!middleware) return; // 全部执行完毕

        try {
          await middleware(ctx, () => dispatch(i + 1));
        } catch (err) {
          // 错误传播：传递给错误处理中间件
          const errorMw = this.middlewares.find(m => (m as any).isErrorHandler);
          if (errorMw) {
            ctx.error = err;
            await errorMw(ctx, () => Promise.resolve());
          } else {
            throw err;
          }
        }
      };

      await dispatch(0);
    };
  }

  // 特殊：error 中间件工厂
  static errorHandler(
    handler: (ctx: MiddlewareContext & { error: any }) => void
  ): Middleware {
    const mw = (ctx: MiddlewareContext & { error: any }, next: () => Promise<void>) => {
      if (ctx.error) {
        handler(ctx, next);
      } else {
        return next();
      }
    };
    (mw as any).isErrorHandler = true;
    return mw;
  }
}
```

```typescript
        errors.push({ field, rule: "", message: error });
      }
    }

    return { valid: errors.length === 0, errors };
  }
```

```typescript
// Undo/Redo 管理器
class UndoRedoManager {
  private undoStack: Command[] = [];
  private redoStack: Command[] = [];
  private maxSize: number;
  // 是否正在执行 undo/redo，防止重复记录
  private isUndoingOrRedoing = false;

  constructor(maxSize = 100) {
    this.maxSize = maxSize;
  }

  execute(command: Command) {
    command.execute();
    this.undoStack.push(command);
    // 执行新命令时清空 redo 栈
    this.redoStack = [];
    // 限制栈大小
    if (this.undoStack.length > this.maxSize) {
      this.undoStack.shift();
    }
  }

  undo() {
    const command = this.undoStack.pop();
    if (!command) return;
    this.isUndoingOrRedoing = true;
    command.undo();
    this.redoStack.push(command);
    this.isUndoingOrRedoing = false;
  }

  redo() {
    const command = this.redoStack.pop();
    if (!command) return;
    this.isUndoingOrRedoing = true;
    command.execute();
    this.undoStack.push(command);
    this.isUndoingOrRedoing = false;
  }

  // 批量操作：一组命令作为一个原子操作
  executeBatch(commands: Command[], batchName = "Batch") {
    const batchCommand: Command = {
      name: batchName,
      execute() { commands.forEach(c => c.execute()); },
      undo() {
        // 逆序撤销
        [...commands].reverse().forEach(c => c.undo());
      },
    };
    this.execute(batchCommand);
  }

  // 清空历史
  clear() {
    this.undoStack = [];
    this.redoStack = [];
  }

  get canUndo() { return this.undoStack.length > 0; }
  get canRedo() { return this.redoStack.length > 0; }
  get undoCount() { return this.undoStack.length; }
  get redoCount() { return this.redoStack.length; }
}

class SnapshotUndoRedo<T> {
  private past: T[] = [];
  private future: T[] = [];
  private maxSize: number;

  constructor(private initialState: T, maxSize = 50) {
    this.maxSize = maxSize;
  }

  pushState(state: T) {
    this.past.push(state);
    if (this.past.length > this.maxSize) {
      this.past.shift();
    }
    this.future = []; // 新操作清空 future
  }

  undo(currentState: T): T | null {
    const previous = this.past.pop();
    if (!previous) return null;
    this.future.push(currentState);
    return previous;
  }

  redo(currentState: T): T | null {
    const next = this.future.pop();
    if (!next) return null;
    this.past.push(currentState);
    return next;
  }
}
```

```typescript
type Reducer<S, A> = (state: S, action: A) => S;
type Listener = () => void;
type Middleware<S, A> = (store: Store<S, A>) => (next: (action: A) => void) => (action: A) => void;

interface Store<S, A> {
  getState(): S;
  dispatch(action: A): void;
  subscribe(listener: Listener): () => void;
}

// 核心 Store 实现
function createStore<S, A>(
  reducer: Reducer<S, A>,
  initialState: S,
  middlewares: Middleware<S, A>[] = []
): Store<S, A> {
  let state = initialState;
  const listeners = new Set<Listener>();

  // 发布订阅模式：subscribe / notify
  const getState = () => state;

  const dispatch = (action: A) => {
    state = reducer(state, action); // 策略模式：reducer 处理不同 action
    listeners.forEach(listener => listener()); // 通知所有订阅者
  };

  const subscribe = (listener: Listener): () => void => {
    listeners.add(listener);
    return () => listeners.delete(listener); // 返回 unsubscribe
  };

  // 中间件装饰器模式
  let enhancedDispatch = dispatch;
  if (middlewares.length > 0) {
    // 使用 compose 组合中间件（函数组合）
    const store = { getState, dispatch: (action: A) => enhancedDispatch(action), subscribe };
    
    const chain = middlewares.map(mw => mw(store));
    // 洋葱模型组合中间件
    enhancedDispatch = chain.reduceRight(
      (next, mw) => mw(next),
      dispatch
    );
  }

  const store: Store<S, A> = {
    getState,
    dispatch: (action: A) => enhancedDispatch(action),
    subscribe,
  };

  return store;
}
```

**问题**：请设计一个前端依赖注入容器，支持服务注册、解析、生命周期管理和作用域控制。分析 React 中如何实现 DI。

```typescript
// 依赖标识符
type Token<T> = string;

// 服务提供者
interface Provider<T> {
  token: Token<T>;
  useClass?: new (...args: any[]) => T;
  useFactory?: (...args: any[]) => T;
  useValue?: T;
  deps?: Token<any>[];
  scope?: "singleton" | "transient" | "request";
}

// DI 容器
class DIContainer {
  private providers = new Map<Token<any>, Provider<any>>();
  private instances = new Map<Token<any>, any>();
  // 解决循环依赖
  private resolving = new Set<Token<any>>();

  register<T>(provider: Provider<T>): void {
    if (this.providers.has(provider.token)) {
      throw new Error(`Provider already registered: ${provider.token}`);
    }
    this.providers.set(provider.token, provider);
  }

  registerMultiple<T>(providers: Provider<T>[]): void {
    providers.forEach(p => this.register(p));
  }

  resolve<T>(token: Token<T>): T {
    // 循环依赖检测
    if (this.resolving.has(token)) {
      throw new Error(`Circular dependency detected: ${token}`);
    }

    const provider = this.providers.get(token);
    if (!provider) {
      throw new Error(`No provider for token: ${token}`);
    }

    // Singleton 检查
    if (provider.scope === "singleton" || provider.scope == null) {
      if (this.instances.has(token)) {
        return this.instances.get(token);
      }
    }

    this.resolving.add(token);

    let instance: T;

    if (provider.useValue !== undefined) {
      instance = provider.useValue;
    } else if (provider.useFactory) {
      const deps = this.resolveDeps(provider.deps ?? []);
      instance = provider.useFactory(...deps);
    } else if (provider.useClass) {
      const deps = this.resolveDeps(provider.deps ?? []);
      instance = new provider.useClass(...deps);
    } else {
      throw new Error(`Invalid provider for: ${token}`);
    }

    this.resolving.delete(token);

    if (provider.scope === "singleton" || provider.scope == null) {
      this.instances.set(token, instance);
    }

    return instance;
  }

  private resolveDeps(deps: Token<any>[]): any[] {
    return deps.map(dep => this.resolve(dep));
  }

  // 清理（主要用于测试）
  clear() {
    this.providers.clear();
    this.instances.clear();
    this.resolving.clear();
  }
}
```

### 13. 设计一个支持拦截器的 API Client。

**问题**：请设计一个前端 HTTP 客户端，支持请求/响应拦截器、请求重试、超时控制、请求取消和错误统一处理。

**参考答案**：

```typescript
// 请求配置
interface RequestConfig {
  url: string;
  method: "GET" | "POST" | "PUT" | "DELETE" | "PATCH";
  headers?: Record<string, string>;
  params?: Record<string, string>;
  data?: any;
  timeout?: number;       // 超时时间（ms）
  retry?: number;        // 重试次数
  retryDelay?: number;   // 重试间隔
  signal?: AbortSignal;  // 取消信号
}

// 拦截器
interface Interceptor<T> {
  onFulfilled?: (value: T) => T | Promise<T>;
  onRejected?: (error: any) => any;
}

class ApiClient {
  private baseURL: string;
  private requestInterceptors: Interceptor<RequestConfig>[] = [];
  private responseInterceptors: Interceptor<Response>[] = [];

  constructor(baseURL: string) {
    this.baseURL = baseURL;
  }

  // 请求拦截器（用于注入 token、添加 header 等）
  addRequestInterceptor(interceptor: Interceptor<RequestConfig>) {
    this.requestInterceptors.push(interceptor);
    return () => {
      const idx = this.requestInterceptors.indexOf(interceptor);
      if (idx > -1) this.requestInterceptors.splice(idx, 1);
    };
  }

  // 响应拦截器（用于统一处理错误、转换数据等）
  addResponseInterceptor(interceptor: Interceptor<Response>) {
    this.responseInterceptors.push(interceptor);
    return () => {
      const idx = this.responseInterceptors.indexOf(interceptor);
      if (idx > -1) this.responseInterceptors.splice(idx, 1);
    };
  }

  // 通过拦截器链处理请求配置
  private async processRequestConfig(config: RequestConfig): Promise<RequestConfig> {
    let transformedConfig = { ...config, url: `${this.baseURL}${config.url}` };

    for (const interceptor of this.requestInterceptors) {
      if (interceptor.onFulfilled) {
        try {
          transformedConfig = await interceptor.onFulfilled(transformedConfig);
        } catch (err) {
          if (interceptor.onRejected) {
            return interceptor.onRejected(err);
          }
          throw err;
        }
      }
    }
    return transformedConfig;
  }

  // 通过拦截器链处理响应
  private async processResponse(response: Response): Promise<Response> {
    let transformedResponse = response;

    for (const interceptor of this.responseInterceptors) {
      if (interceptor.onFulfilled) {
        try {
          transformedResponse = await interceptor.onFulfilled(transformedResponse);
        } catch (err) {
          if (interceptor.onRejected) {
            return interceptor.onRejected(err);
          }
          throw err;
        }
      }
    }
    return transformedResponse;
  }

  // 核心请求方法（带重试和超时）
  private async request<T = any>(config: RequestConfig): Promise<T> {
    const processedConfig = await this.processRequestConfig(config);
    const { retry = 0, retryDelay = 1000, timeout, signal, ...fetchConfig } = processedConfig;

    const controller = new AbortController();
    const timeoutId = timeout
      ? setTimeout(() => controller.abort(), timeout)
      : null;

    // 合并外部信号和超时信号
    const combinedSignal = signal
      ? this.combineSignals(signal, controller.signal)
      : controller.signal;

    let lastError: any;

    for (let attempt = 0; attempt <= retry; attempt++) {
      try {
        const response = await fetch(fetchConfig.url, {
          method: fetchConfig.method,
          headers: fetchConfig.headers,
          body: fetchConfig.data ? JSON.stringify(fetchConfig.data) : undefined,
          signal: combinedSignal,
        });

        const processedResponse = await this.processResponse(response);

        if (!processedResponse.ok) {
          throw new ApiError(
            `HTTP ${processedResponse.status}`,
            processedResponse.status,
            await processedResponse.json().catch(() => null)
          );
        }

        return processedResponse.json() as Promise<T>;
      } catch (err) {
        lastError = err;
        if (attempt < retry && !(err instanceof DOMException && err.name === "AbortError")) {
          await new Promise(resolve => setTimeout(resolve, retryDelay));
        }
      } finally {
        if (timeoutId) clearTimeout(timeoutId);
      }
    }

    throw lastError;
  }

  private combineSignals(...signals: AbortSignal[]): AbortSignal {
    const controller = new AbortController();
    signals.forEach(s => {
      if (s.aborted) return controller.abort(s.reason);
      s.addEventListener("abort", () => controller.abort(s.reason), { once: true });
    });
    return controller.signal;
  }

  get<T>(url: string, config?: Partial<RequestConfig>) {
    return this.request<T>({ url, method: "GET", ...config });
  }

  post<T>(url: string, data?: any, config?: Partial<RequestConfig>) {
    return this.request<T>({ url, method: "POST", data, ...config });
  }

  put<T>(url: string, data?: any, config?: Partial<RequestConfig>) {
    return this.request<T>({ url, method: "PUT", data, ...config });
  }

  delete<T>(url: string, config?: Partial<RequestConfig>) {
    return this.request<T>({ url, method: "DELETE", ...config });
  }
}

class ApiError extends Error {
  constructor(message: string, public status: number, public body: any) {
    super(message);
    this.name = "ApiError";
  }
}

// 使用示例
const api = new ApiClient("https://api.example.com");

// 注入认证 token
api.addRequestInterceptor({
  onFulfilled: (config) => ({
    ...config,
    headers: {
      ...config.headers,
      Authorization: `Bearer ${localStorage.getItem("token")}`,
    },
  }),
});

// 统一错误处理
api.addResponseInterceptor({
  onRejected: (error) => {
    if (error instanceof ApiError && error.status === 401) {
      // 重定向到登录页
      window.location.href = "/login";
    }
    return Promise.reject(error);
  },
});

// 调用
const users = await api.get<User[]>("/users", { retry: 2, timeout: 5000 });
```

### 15. 如何理解"组合优于继承"（Composition over Inheritance）？

**问题**：请解释为什么组合优于继承，对比两者的优缺点，并给出前端开发中的实际案例。

**参考答案**：

继承是白盒复用（子类了解父类实现细节），耦合度高；组合是黑盒复用（通过接口交互），更灵活。

```typescript
// === 继承方式：问题示例 ===
class Animal {
  constructor(protected name: string) {}
  eat() { console.log(`${this.name} is eating`); }
  sleep() { console.log(`${this.name} is sleeping`); }
}

// 深层继承导致脆弱性
class Bird extends Animal {
  fly() { console.log(`${this.name} is flying`); }
}

class Penguin extends Bird {
  // 问题：Penguin 不会飞！违反了 LSP
  fly() { throw new Error("Penguins can't fly!"); }
}

// === 组合方式：更好的方案 ===
interface Behavior {
  execute(): void;
}

class FlyBehavior implements Behavior {
  constructor(private name: string) {}
  execute() { console.log(`${this.name} is flying`); }
}

class SwimBehavior implements Behavior {
  constructor(private name: string) {}
  execute() { console.log(`${this.name} is swimming`); }
}

class Creature {
  constructor(
    private name: string,
    private behaviors: Map<string, Behavior>
  ) {}

  perform(action: string) {
    const behavior = this.behaviors.get(action);
    if (behavior) behavior.execute();
  }

  addBehavior(name: string, behavior: Behavior) {
    this.behaviors.set(name, behavior);
  }
}

// 不同生物可以自由组合能力
const duck = new Creature("Duck", new Map([
  ["fly", new FlyBehavior("Duck")],
  ["swim", new SwimBehavior("Duck")],
]));

const penguin = new Creature("Penguin", new Map([
  ["swim", new SwimBehavior("Penguin")], // 没有 fly 能力
]));
```

**前端中的体现**：
- **React**：组合是核心哲学。"组件组合"优于"组件继承"。
- **Vue**：Composition API（组合式 API）优于 Options API 的 mixin 继承。
- **HOC vs Hooks**：Hooks 通过组合逻辑替代了 HOC 继承式复用。

```typescript

    return <div>{super.render()} <AdminPanel /></div>;

// 组合方式（推荐）
```

**对比总结**：

| 维度 | 继承 | 组合 |
|------|------|------|
| 耦合度 | 高（子类依赖父类实现） | 低（通过接口交互） |
| 复用粒度 | 类级别 | 方法/行为级别 |
| 扩展性 | 脆弱（基类变化影响所有子类） | 灵活（可动态组合） |
| 测试 | 需要完整的继承链 | 可以独立测试每个组件 |
| 前端现状 | class extends 很少使用 | props / slots / hooks 大量使用 |

- 概念差异解释（30%）
- 优缺点对比（25%）

- 对框架设计理念的理解（20%）

**常见错误**：
- 认为"组合就是复制粘贴代码"
- 把 "extends" 当作唯一的代码复用手段
- 在 React 中试图用继承创建"基础组件"，再从中派生
- 不理解组合可能导致"回调地狱"式嵌套（wrapper hell）

```typescript
// === 第二层：Token 转换器（Transform Pipeline）===
type TokenType = "color" | "dimension" | "fontSize" | "fontFamily" | "shadow" | "number";
type OutputFormat = "css" | "scss" | "less" | "tailwind" | "js" | "android" | "ios";

interface Token {
  value: any;
  type: TokenType;
  description?: string;
}

interface TransformOptions {
  prefix?: string;
  format: OutputFormat;
  theme?: "light" | "dark";
  platform?: "web" | "mobile" | "miniapp";
}

class TokenPipeline {
  constructor(private tokens: Record<string, any>) {}

  transform(opts: TransformOptions): string {
    switch (opts.format) {
      case "css":
        return this.toCSS(opts);
      case "scss":
        return this.toSCSS(opts);
      case "tailwind":
        return this.toTailwindConfig(opts);
      case "js":
        return this.toJavaScript(opts);
      default:
        return this.toCSS(opts);
    }
  }

  private toCSS({ prefix = "ds" }: TransformOptions): string {
    const result: string[] = [":root {"];
    this.flattenTokens(this.tokens, prefix).forEach(([name, value]) => {
      result.push(`  --${name}: ${value};`);
    });
    result.push("}");
    return result.join("\n");
  }

  // 扁平化嵌套 Token 为 CSS 变量名
  private flattenTokens(
    obj: any,
    prefix: string,
    path: string[] = []
  ): Array<[string, string]> {
    const result: Array<[string, string]> = [];

    for (const [key, value] of Object.entries(obj)) {
      const currentPath = [...path, key];
      if (value.value !== undefined && value.type) {
        // 叶子节点：Token 值
        result.push([currentPath.join("-"), this.formatValue(value)]);
      } else if (typeof value === "object") {
        // 内部节点：继续递归
        result.push(...this.flattenTokens(value, prefix, currentPath));
      }
    }

    return result;
  }

  private formatValue(token: Token): string {
    if (token.type === "shadow" && typeof token.value === "object") {
      const { x, y, blur, color } = token.value;
      return `${x} ${y} ${blur} ${color}`;
    }
    return String(token.value);
  }
}
```

### 19. 设计一个前端 A/B Testing 基础设施。

**问题**：请设计一个前端 A/B 测试系统，支持实验分组、流量分配、埋点上报、数据分析和动态加载实验方案。

```typescript
interface ExperimentConfig {

interface Variant {
  id: string;
  name: string;              // "control" | "treatment_a" | "treatment_b"
  weight: number;            // 权重 0-1（所有 variant 权重和为 1）
  payload: Record<string, any>; // 实验参数（如按钮颜色、文案等）
}

interface TargetCriteria {
  geos?: string[];           // 地区
  userPercentile?: number[]; // 用户百分位范围
  attributes?: Record<string, string[]>; // 用户属性筛选
}

// === 核心实验引擎 ===
class ABTestEngine {
  private experiments: Map<string, ExperimentConfig> = new Map();
  private activatedExperiments: Map<string, string> = new Map(); // expId -> variantId
  private userResolver: UserResolver;

  constructor(userResolver: UserResolver) {
    this.userResolver = userResolver;
  }

  // 加载远程实验配置
  async loadExperiments(configUrl: string) {
    const response = await fetch(configUrl);
    const configs: ExperimentConfig[] = await response.json();
    configs.forEach(c => this.experiments.set(c.id, c));
  }

  // 用户分桶：确定用户被分配到哪个实验变体
  private bucketUser(userId: string, experiment: ExperimentConfig): Variant | null {
    // 1. 检查流量比例
    const userHash = this.hashUser(userId, experiment.id);
    if (userHash > experiment.traffic) {
      return null; // 用户不在实验流量中
    }

    // 2. 检查目标条件
    if (!this.meetsCriteria(userId, experiment.targetCriteria)) {
      return null;
    }

    // 3. 根据用户哈希和变体权重分配
    const hashInTraffic = userHash / experiment.traffic; // 归一化到 0-1
    let cumulativeWeight = 0;

    for (const variant of experiment.variants) {
      cumulativeWeight += variant.weight;
      if (hashInTraffic <= cumulativeWeight) {
        return variant;
      }
    }

    return experiment.variants[experiment.variants.length - 1];
  }

  // 获取用户在所有实验中的分配结果
  activateExperiments(userId: string): Record<string, string> {
    const assignments: Record<string, string> = {};

    for (const [expId, experiment] of this.experiments) {
      if (experiment.status !== "running") continue;

      const variant = this.bucketUser(userId, experiment);
      if (variant) {
        assignments[expId] = variant.id;
        this.activatedExperiments.set(expId, variant.id);
      }
    }

    return assignments;
  }

  // 获取指定实验的参数
  getExperimentValue<T>(experimentId: string, key: string, defaultValue: T): T {
    const variantId = this.activatedExperiments.get(experimentId);
    if (!variantId) return defaultValue;

    const experiment = this.experiments.get(experimentId);
    const variant = experiment?.variants.find(v => v.id === variantId);
    return variant?.payload?.[key] ?? defaultValue;
  }

  // 确定性的用户哈希
  private hashUser(userId: string, salt: string): number {
    const hash = this.simpleHash(`${userId}:${salt}`);
    return (hash % 10000) / 10000; // 返回 0-1 之间的值
  }

  private simpleHash(str: string): number {
    let hash = 0;
    for (let i = 0; i < str.length; i++) {
      const char = str.charCodeAt(i);
      hash = ((hash << 5) - hash) + char;
      hash = hash & hash;
    }
    return Math.abs(hash);
  }

  private meetsCriteria(userId: string, criteria: TargetCriteria): boolean {
    if (!criteria) return true;
    // 实际项目中检查用户地理位置、设备类型等
    return true;
  }
}
```

```typescript
// 操作定义
interface Operation {
  type: "insert" | "delete" | "retain";
  position: number;
  chars?: string;
  length?: number;
  userId: string;
  seq: number;        // 操作序号
  revision: number;   // 文档版本号
}

// OT 客户端
class OTClient {
  private doc: string = "";
  private revision: number = 0;
  private pendingOps: Operation[] = [];       // 已发送但未确认
  private pendingBuffer: Operation[] = [];    // 正在发送中

  // 本地操作
  applyLocal(op: Operation) {
    // 在本地文档上应用操作
    op.revision = this.revision;
    op.seq = this.pendingOps.length;
    this.doc = this.transformDoc(this.doc, op);
    this.pendingOps.push(op);
    return op;
  }

  // 收到远程操作
  receiveRemote(remoteOp: Operation) {
    // 对 pendingOps 进行 OT 转换
    const transformedRemoteOp = this.transformAgainstPending(remoteOp);

    // 应用转换后的操作
    this.doc = this.transformDoc(this.doc, transformedRemoteOp);
    this.revision++;
  }

  // 收到服务器确认
  acknowledge(seq: number) {
    this.pendingOps = this.pendingOps.filter(op => op.seq !== seq);
  }

  // 针对 pending 操作转换远程操作
  private transformAgainstPending(remoteOp: Operation): Operation {
    let transformedOp = { ...remoteOp };
    for (const pending of this.pendingOps) {
      transformedOp = OT.transform(transformedOp, pending);
    }
    return transformedOp;
  }

  private transformDoc(doc: string, op: Operation): string {
    if (op.type === "insert") {
      return doc.slice(0, op.position) + op.chars + doc.slice(op.position);
    } else if (op.type === "delete") {
      return doc.slice(0, op.position) + doc.slice(op.position + op.length!);
    }
    return doc;
  }
}

// OT 核心转换算法（简化版）
class OT {
  static transform(op1: Operation, op2: Operation): Operation {
    if (op1.type === "insert" && op2.type === "insert") {
      if (op1.position < op2.position) return op1;
      if (op1.position > op2.position) {
        return { ...op1, position: op1.position + (op2.chars?.length ?? 0) };
      }
      // 相同位置：按 userId 排序
      return op1.userId < op2.userId
        ? op1
        : { ...op1, position: op1.position + (op2.chars?.length ?? 0) };
    }
    // ... 更多转换规则
    return op1;
  }
}
```

```typescript
import React, { Component, ErrorInfo, ReactNode } from "react";

interface ErrorBoundaryProps {
  children: ReactNode;
  fallback?: ReactNode | ((error: Error, retry: () => void) => ReactNode);
  onError?: (error: Error, errorInfo: ErrorInfo) => void;
  retryCount?: number;        // 自动重试次数
  retryDelay?: number;        // 重试间隔（ms）
  resetOnPropsChange?: boolean;
}

interface ErrorBoundaryState {
  hasError: boolean;
  error: Error | null;
  retryAttempts: number;
}

class SmartErrorBoundary extends Component<ErrorBoundaryProps, ErrorBoundaryState> {
  state: ErrorBoundaryState = {
    hasError: false,
    error: null,
    retryAttempts: 0,
  };

  static getDerivedStateFromError(error: Error): Partial<ErrorBoundaryState> {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error, errorInfo: ErrorInfo) {
    console.error("ErrorBoundary caught:", error, errorInfo);
    this.props.onError?.(error, errorInfo);
  }

  // 自动重试
  componentDidUpdate(_: ErrorBoundaryProps, prevState: ErrorBoundaryState) {
    if (
      prevState.retryAttempts < prevState.retryAttempts + 1 &&
      this.state.hasError &&
      this.state.retryAttempts < (this.props.retryCount ?? 0)
    ) {
      const delay = this.props.retryDelay ?? 2000;
      setTimeout(() => this.retry(), delay);
    }
  }

  componentDidMount() {
    if (this.props.resetOnPropsChange) {
      // 监听 props 变化自动重置
    }
  }

  retry = () => {
    this.setState(prev => ({
      hasError: false,
      error: null,
      retryAttempts: prev.retryAttempts + 1,
    }));
  };

  render() {
    if (this.state.hasError) {
      if (this.props.fallback) {
        if (typeof this.props.fallback === "function") {
          return (this.props.fallback as Function)(this.state.error!, this.retry);
        }
        return this.props.fallback;

      return (

          <h3>出错了</h3>

  }

// === 异步操作的 Circuit Breaker（断路器）===
class CircuitBreaker {
  private failures = 0;
  private lastFailureTime = 0;
  private state: "closed" | "open" | "half-open" = "closed";

  constructor(
    private threshold: number = 5,       // 失败阈值
    private cooldownPeriod: number = 30000, // 冷却时间 30s
    private halfOpenMaxRequests: number = 1
  ) {}

  async call<T>(fn: () => Promise<T>): Promise<T> {
    if (this.state === "open") {
      if (Date.now() - this.lastFailureTime > this.cooldownPeriod) {
        this.state = "half-open";
      } else {
        throw new Error("Circuit breaker is open");
      }
    }

    try {
      const result = await fn();
      this.onSuccess();
      return result;
    } catch (err) {
      this.onFailure();
      throw err;
    }
  }

  private onSuccess() {
    this.failures = 0;
    this.state = "closed";
  }

  private onFailure() {
    this.failures++;
    this.lastFailureTime = Date.now();
    if (this.failures >= this.threshold) {
      this.state = "open";
    }
  }
}

function useFetchWithRetry<T>(
  fetcher: () => Promise<T>,
  options: { retries?: number; circuitBreaker?: CircuitBreaker } = {}
) {
  const [data, setData] = useState<T | null>(null);
  const [error, setError] = useState<Error | null>(null);
  const [loading, setLoading] = useState(true);
  const retryCount = useRef(0);
  const maxRetries = options.retries ?? 3;

  const execute = useCallback(async () => {
    setLoading(true);
    setError(null);
    try {
      const result = options.circuitBreaker
        ? await options.circuitBreaker.call(fetcher)
        : await fetcher();
      setData(result);
      retryCount.current = 0;
    } catch (err) {
      setError(err as Error);
      if (retryCount.current < maxRetries) {
        retryCount.current++;
        // 指数退避
        const delay = Math.min(1000 * Math.pow(2, retryCount.current), 10000);
        setTimeout(execute, delay);
      }
    } finally {
      setLoading(false);
    }
  }, [fetcher]);

  return { data, error, loading, retry: execute };
}
```

```typescript
class AuthEngine {
  private roles: Map<string, Role> = new Map();
  private currentUser: User | null = null;

  // 加载权限配置（可来自服务端）
  async init(configUrl: string) {
    const data = await fetch(configUrl).then(r => r.json());
    data.roles.forEach((role: Role) => this.roles.set(role.id, role));
  }

  setUser(user: User) {
    this.currentUser = user;
  }

  // 检查用户是否拥有指定权限
  can(permission: Permission): boolean {
    if (!this.currentUser) return false;

    // 1. 先检查直接赋予的权限
    if (this.currentUser.permissions?.includes(permission)) {
      return true;
    }

    // 2. 检查角色继承的权限
    return this.currentUser.roles.some(roleId => {
      const role = this.roles.get(roleId);
      return role?.permissions.includes(permission);
    });
  }

  // 检查是否拥有所有指定权限
  canAll(permissions: Permission[]): boolean {
    return permissions.every(p => this.can(p));
  }

  // 检查是否拥有任一指定权限
  canAny(permissions: Permission[]): boolean {
    return permissions.some(p => this.can(p));
  }
}
```

- 可持续性（15%）

**常见错误**：
- 没有测试覆盖就贸然重构（"没有安全网"）
- 一次重构范围过大，难以 Review 和回滚
- 重构中混合了功能变更（应"变更代码结构"和"变更功能"分开）
- 认为重构只是在"重写"，而不理解"小步改进"

**扩展追问**：
- 如何向产品经理/业务方解释"需要重构"？（量化技术债）
- 在大项目中如何做"渐进式架构迁移"？（如 jQuery 迁移到 React）
- 如何衡量重构的效果？（代码复杂度指标 / 开发效率 / Bug 率）
- 什么是"Boy Scout Rule"？它在实际项目中如何落地？

## 四、面试技巧与策略

### 1. 回答设计模式问题的通用框架

建议使用 **STAR + Pattern 框架** 组织回答：

1. **S（Situation）**：描述场景和上下文
2. **T（Task）**：明确要解决的问题
3. **A（Action）**：选择的具体模式和执行方案
4. **R（Result）**：最终效果和改进度量

**示例**：
> "在之前的电商项目中（S），我们需要支持多种支付方式且未来可能新增（T）。我采用了策略模式（A），定义了 PaymentStrategy 接口，每种支付方式实现该接口。结果新支付方式接入只需新增类无需修改核心逻辑（R）。"

### 2. 回答的五个层次

| 层级 | 特征 | 面试评分参考 |
|------|------|-------------|
| L1 | 能说出模式定义 | 及格线（60%） |
| L2 | 理解模式优缺点和适用场景 | 良好（75%） |
| L3 | 有实际项目代码经验 | 优秀（85%） |
| L4 | 能对比多种方案做出选型依据 | 突出（90%） |
| L5 | 能分析框架/源码中的模式应用 | 顶尖（95%+） |

### 3. 常见面试陷阱提醒

- **不要只背定义**：面试官期望的是"在你项目中怎么用的"而不只是"书上怎么写"
- **不要过度设计**：谈到设计模式时自然地提及"这个场景是否需要"——展示你的判断力
- **代码要可运行**：不要写伪代码，面试官会追问细节
- **模式不是目的**：强调"解决问题"而非"使用模式"，模式是手段不是目标
- **准备一个完整案例**：选一个你真实项目中复杂度适中的重构案例，能用 3-5 分钟说清楚

### 4. 高频追问准备

- **如果不使用这个模式，你会怎么做？**（考察对替代方案的理解）
- **这个模式的缺点是什么？**（考察全面思考能力）
- **这个场景还有其他模式可选吗？**（考察知识广度）
- **在实际落地中遇到过什么坑？**（考察实战经验）

### 5. 各层级面试准备建议

**初级前端（1-3 年）**：
- 熟练掌握基础设计模式（单例、观察者、策略、工厂）
- 能说出每种模式的优缺点
- 重点准备 SOLID 原则

**中级前端（3-5 年）**：
- 掌握进阶模式（装饰器、代理、命令、中介者）
- 能结合框架源码分析模式应用
- 重点准备状态管理和组件通信方案

**高级前端（5 年以上）**：
- 能设计复杂架构（插件系统、微前端、协同编辑）
- 理解模式背后的设计哲学（组合优于继承、控制反转）
- 重点准备架构权衡和技术选型决策
