## 一、基础题

### 1. 什么是 Web 无障碍（a11y）？为什么要关注它？

**参考答案**：

Web 无障碍是指让网站和应用能够被尽可能多的人使用，包括视觉、听觉、运动、认知障碍者，以及临时性障碍用户。

关注原因：
- **法律合规**：国内外法规要求数字产品达到一定可访问性标准。
- **商业收益**：扩大用户群体，提升 SEO 和用户体验。
- **社会责任**：体现包容性设计价值。
- **技术质量**：无障碍代码通常语义化更好、可维护性更高。

**评分维度**：
- 定义准确（40%）
- 列举至少 3 个关注原因（40%）
- 举例说明（20%）

### 2. 解释 WCAG 的 POUR 四项原则。

**参考答案**：

| 原则 | 含义 |
|------|------|
| Perceivable（可感知） | 用户能感知信息和界面 |
| Operable（可操作） | 用户能操作界面 |
| Understandable（可理解） | 内容和操作可理解 |
| Robust（健壮） | 内容可被各种辅助技术解析 |

每项原则下都有具体成功标准，按 A/AA/AAA 三级划分。

**评分维度**：
- 完整解释四项原则（60%）

- 举例说明（15%）

### 5. 为什么不应该只用颜色传达信息？

**参考答案**：

色盲、低视力或在强光环境下的用户可能无法区分颜色。如果仅用颜色表示状态，这些用户会丢失关键信息。

正确做法：
- 配合图标、文字、形状。
- 使用纹理或图案区分。
- 确保信息有多个感知通道。

示例：
- 错误：`<span class="red">失败</span>`
- 正确：`<span class="error">✕ 失败</span>`

## 二、进阶题

### 6. 什么是 ARIA？应该在什么情况下使用？

**参考答案**：

ARIA（Accessible Rich Internet Applications）是一组属性，用于补充 HTML 无法表达的语义信息，包括角色（role）、状态（state）和属性（property）。

使用原则：
1. **优先使用原生 HTML**：能用 `<button>` 就不用 `role="button"`。
2. **需要时使用 ARIA**：自定义组件（标签页、对话框、下拉菜单）需要补充语义。
3. **确保状态同步**：ARIA 属性必须随组件状态实时更新。
4. **提供可访问名称**：每个交互元素都应有名称。

**评分维度**：
- 解释 ARIA 组成（40%）
- 说明使用原则（40%）

### 7. 如何设计一个可访问的模态对话框？

**参考答案**：

1. **语义结构**：使用 `<dialog>` 或 `role="dialog"`、`aria-modal="true"`。
2. **标题关联**：使用 `aria-labelledby` 关联标题。
3. **焦点管理**：
   - 打开时焦点移到对话框内第一个可聚焦元素或标题。
   - Tab 在对话框内循环（焦点陷阱）。
   - 关闭时焦点返回触发按钮。
4. **关闭方式**：支持 Esc 键和明确的关闭按钮。
5. **背景处理**：建议禁用背景交互（非必须，但提升体验）。

### 12. 解释一下 aria-live 的使用场景和注意事项。

**参考答案要点**：

`aria-live` 用于告知屏幕阅读器某个区域的内容会动态更新，需要播报。

取值：
- `off`：不自动播报（默认）。
- `polite`：当前任务完成后再播报。
- `assertive`：立即打断当前播报。

使用场景：
- 表单提交后的成功/错误提示。
- 实时通知、Toast 消息。
- 加载状态变化。

注意事项：
- 避免过度使用 `assertive`，会打断用户。
- 动态区域应在页面加载时就存在，而不是动态创建。
- 搭配 `aria-atomic` 和 `aria-relevant` 控制播报范围。

- 解释 aria-live 取值（30%）
- 使用场景（30%）

### 13. 解释什么是焦点陷阱（Focus Trap）？在哪些场景下需要使用？实现时需要注意什么？

**参考答案**：

焦点陷阱是指将键盘焦点限制在某个 DOM 子树的范围内，使 Tab 键只能在限定区域内循环，无法跳出。

**适用场景**：
- **模态对话框（Modal Dialog）**：用户必须处理对话框后才能返回主内容，焦点不应逃逸到背景。
- **侧边抽屉面板（Slide-out Panel）**：面板展开时，焦点应限制在面板内，防止与背景内容混淆。
- **全屏覆盖层（Overlay/Sidebar）**：类似 Cookie 同意横幅、移动端导航菜单等临时覆盖层。

**实现方式对比**：
- **动态 tabindex 管理**：在焦点到达第一个和最后一个可聚焦元素时，手动将焦点循环回另一端点。需要维护可聚焦元素列表，监听键盘事件。
- **使用 tabindex="-1" + focus()**：在容器外拦截 Tab 键，通过 JS 控制焦点在容器内循环。更简洁但需注意事件冒泡和顺序。

**关闭时焦点恢复**：
- 打开前记录触发元素的引用，关闭后调用 `element.focus()` 将焦点归还。
- 这是 WCAG 2.1 2.4.3 Focus Order 的明确要求。

**常见实现错误**：
- 忘记恢复焦点，用户停留在未知位置。
- 可聚焦元素列表未随动态内容更新（如异步加载的表单项）。
- 在容器溢出隐藏时仍允许焦点移到不可见元素。
- 只处理 Tab 不处理 Shift+Tab 反向循环。
- 使用 `focus` 事件监听导致递归触发。

**评分维度**：
- 准确定义焦点陷阱（20%）
- 列举适用场景（25%）
- 说明实现方式及对比（30%）
- 指出常见错误（25%）

**常见错误**：
- 仅考虑正向 Tab 忽略 Shift+Tab
- 未在动态内容变化后更新可聚焦元素列表
- 焦点恢复处理遗漏

### 14. 什么是 Roving Tabindex 模式？适用于哪些组件？请举例说明。

**参考答案**：

Roving Tabindex 是一种键盘导航模式，其核心思想是：在组件（如工具栏、菜单、标签页）中，任何时候只有一个子元素在 Tab 顺序中（tabindex="0"），其余子元素的 tabindex="-1"。用户通过箭头键在子元素间移动焦点，编程式地将 tabindex="0" 转移到当前聚焦项，同时将前一项设为 tabindex="-1"。

**适用组件**：
- **Menu / Menubar**：方向键在菜单项间移动，Enter 激活项。
- **Toolbar**：方向键在工具按钮间移动，Tab 离开工具栏。
- **Tablist**：方向键切换标签页，Tab 进入/离开标签面板。
- **Radiogroup**：方向键切换选中项，Tab 进入/离开整个组。
- **Tree / Treegrid**：方向键结合展开/折叠进行层级导航。
- **Listbox**：方向键在选项间移动。

**与 aria-activedescendant 的区别**：
- Roving Tabindex 将实际的 DOM 焦点移动到每个项上，屏幕阅读器自然跟随焦点。
- aria-activedescendant 将焦点保持在容器（如 input）上，通过 `aria-activedescendant="id"` 指示当前活动项。适用于 combobox、listbox 等输入型组件。
- 选择依据：当使用原生交互元素（如 input）时倾向 aria-activedescendant；当容器本身无焦点能力时使用 Roving Tabindex。

**键盘事件处理**：
- 方向键（→/↓/←/↑）：在可选项间移动焦点，支持循环或不循环两种模式。
- Home/End：跳转到第一个或最后一个可选项。
- 字符键（可选）：快速跳转到以该字母开头的项。
- 注意：Tab 键应退出组件而非在内部循环，这与焦点陷阱不同。

**无障碍树影响**：
- 屏幕阅读器感知到的是"一次只聚焦一个元素"，避免在 Tab 遍历时朗读整个列表。
- 辅助技术用户通过方向键在组件内探索，体验更流畅。
- 搭配适当的 ARIA 角色（role="menuitem"、role="tab" 等）确保语义正确。

**评分维度**：
- 解释 Roving Tabindex 工作机制（30%）
- 列举适用组件（20%）
- 对比 aria-activedescendant（25%）
- 说明键盘处理（25%）

**常见错误**：
- 未处理 Home/End 键，用户需多次按键才能到达两端。
- 在 Tab 键上也使用循环而非退出，与用户预期不符。
- 忘记同步 `aria-selected`、`aria-checked` 等状态属性。

**扩展追问**：

- 如何用 React Hook 封装可复用的 Roving Tabindex 逻辑？

### 15. 如何为单页应用（SPA）实现合理的焦点管理和页面标题更新？

**参考答案**：

**路由变化时的焦点管理**：
- 导航完成后，将焦点移到新页面的 `<h1>` 主标题或 `<main>` 内容区域。
- 为标题或容器设置 `tabindex="-1"`，使其可被编程聚焦但不进入 Tab 顺序。
- 聚焦后屏幕阅读器会朗读标题，用户立即知道当前页面内容。
- 实现方式：在路由守卫或组件 mount 时调用 `ref.current.focus()`。

**动态页面标题更新**：
- 每次路由变化时更新 `<title>` 标签内容，确保屏幕阅读器朗读新标题。
- 使用框架的路由元数据（如 React Router 的 route handle 或 Vue Router 的 meta）定义各页面标题。
- 标题格式建议："页面名称 - 站点名称"，便于浏览器标签和屏幕阅读器识别。

**使用 aria-live 进行路由通知**：
- 在页面顶层放置一个 `aria-live="polite"` 区域，预留给路由状态播报。
- 路由变化时向该区域注入描述性文字，如"已跳转到帮助中心页面"。
- 配合视觉焦点移动，提供双重确认，增强用户信心。

**动态加载内容的焦点顺序**：
- 异步加载的数据插入 DOM 后，应评估是否影响焦点顺序。
- 如果新内容出现在当前聚焦元素之前，可能导致焦点意外跳回。
- 最佳实践：在异步内容上方使用 `aria-busy="true"`，加载完成后移除。
- 使用 `intersectionObserver` 结合 `focus()` 管理新增可聚焦元素的焦点位置。

**评分维度**：
- 路由变化的焦点管理方法（30%）
- 页面标题更新策略（25%）
- aria-live 路由通知（25%）
- 动态加载内容处理（20%）

**常见错误**：
- 路由切换后焦点丢失（回退到页面顶部），用户需要重新 Tab 导航。
- 忘记设置 tabindex="-1"，浏览器默认不允许非交互元素聚焦。
- 标题更新仅用于浏览器标签，未考虑屏幕阅读器朗读。
- 动态内容插入后未考虑焦点顺序被打乱。

### 16. 解释 WCAG 2.2 新增的 Focus Not Obscured（2.4.11）和 Target Size（2.5.8）标准。它们对前端开发有什么实际影响？

**参考答案**：

**Focus Not Obscured（2.4.11，AA 级）**：
- 要求元素的焦点环（focus indicator）不能被其他元素遮挡。
- 典型问题：粘性头部（sticky header）或固定底部栏（fixed footer）覆盖了页面中部元素的焦点轮廓。
- 实现方案：
  - 使用 `scroll-margin` 确保聚焦元素滚动到可见位置不被遮挡。
  - 粘性元素使用 `z-index` 管理时，考虑聚焦元素的可视性。
  - 动画或变换效果不应使焦点环部分隐藏。
- Level AAA（2.4.12）进一步要求焦点环完全可见且不被任何元素遮挡。

**Target Size（2.5.8，AA 级）**：
- 要求交互目标的尺寸至少为 24x24 CSS 像素。
- 例外情况：行内元素（如文本链接）、浏览器默认控件、法律文件中必需的展现方式。
- 实际影响：
  - 移动端菜单按钮、关闭图标、表单控件等需满足最小尺寸。
  - 紧密排列的图标按钮（如编辑/删除操作列）需加大间距或尺寸。
  - 可通过 padding 扩展点击区域而不必改变视觉大小。
- Level AAA（2.5.3 旧标准）要求 44x44，2.5.8 的 24x24 是 AA 级的较低门槛。

**对前端开发的实践影响**：
- CSS 布局调整：确保粘性头部不遮挡元素。可以设置 `scroll-padding-top` 匹配粘性头部高度。
- 组件库更新：按钮、表单控件、链接等交互元素需满足 24x24 最小尺寸。
- Touch 目标优化：在触屏设备上尤为重要，使用 `min-width/height` 结合 padding 替代纯视觉尺寸。
- 测试工具升级：axe-core 4.5+ 和 Lighthouse 已支持 2.2 新标准检查。

**评分维度**：
- 解释 Focus Not Obscured 要求（30%）
- 解释 Target Size 要求（30%）
- 分析 CSS/布局影响（25%）
- 说明工具支持情况（15%）

**常见错误**：
- 认为只影响移动端，忽略桌面端 sticky header 遮挡。
- 仅关注视觉尺寸，未通过 padding 扩展可点击区域。
- 忽略动态内容的尺寸变化（如加载后内容撑开导致焦点被遮挡）。

### 17. 如何设计一个可访问的拖拽排序组件（Drag and Drop）？

**参考答案**：

**ARIA 角色与属性**：
- 容器设置 `role="application"` 或 `role="list"` 或相应语义角色，告知屏幕阅读器这是一个可交互区域。
- 可拖拽项设置 `role="listitem"` 等适当角色。
- 使用 `aria-grabbed="true/false"` 标识当前正被拖拽的项（此属性已标记为 deprecated，但仍有参考价值）。
- 使用 `aria-dropeffect="move/copy/execute"` 标识放置区效果（同样 deprecated，推荐用实时文本播报替代）。
- 现代实践中，更推荐用 `aria-roledescription` 和实时区域播报提供拖拽状态。

**键盘替代方案**：
- 提供"上移"和"下移"按钮作为主要的键盘操作方式，对所有用户可用，不限于屏幕阅读器用户。
- 拖拽句柄（drag handle）可聚焦（tabindex="0"），Enter/Space 进入拖拽模式，方向键移动，Enter 确认放置。
- 支持 Home/End 跳转到列表首尾。

**屏幕阅读器播报**：
- 使用 `aria-live="polite"` 区域播报拖拽状态变化。
- 播报内容示例："项目 3 已移动到位置 2"、"当前拖拽 项目 5"、"已放置"。
- 使用 `aria-posinset` 和 `aria-setsize` 声明项在列表中的位置和总数。

**拖拽句柄设计**：
- 句柄元素具备 `tabindex="0"` 和 `role="button"`。
- 句柄应有清晰的可访问名称（`aria-label="拖拽排序"`）。
- 句柄的焦点环（focus indicator）必须可见。

**视觉反馈**：
- 被拖拽项有明确的视觉偏移或投影。
- 放置位置有清晰的目标线或高亮。
- 所有状态变化都通过 ARIA 属性同步。

**评分维度**：
- ARIA 角色与属性使用（30%）
- 键盘替代操作设计（30%）
- 屏幕阅读器播报（25%）
- 句柄焦点与视觉反馈（15%）

**常见错误**：
- 只实现鼠标拖拽，完全无法键盘操作。
- 拖拽状态无屏幕阅读器通知，盲人用户无法感知排序变化。
- 没有提供撤销/恢复功能，用户犯错后难以回退。
- 拖拽句柄过小，无法聚焦或难以触达。

**扩展追问**：
- 如何结合 HTML5 Drag and Drop API 与 ARIA 属性实现完整方案？
- 触摸设备的拖拽排序无障碍处理有哪些特殊考虑？
- 如何在表格行拖拽排序中保持正确的可访问名称和状态？

### 18. 对比手动可访问性测试和自动化可访问性测试的优劣势。应该如何制定团队的测试策略？

**参考答案**：

**自动化测试优势**：
- 速度快，可在每次 CI 构建中运行。
- 覆盖率广，能扫描全站数千个页面。
- 结果可重复，便于回归和趋势分析。
- 能发现约 30-50% 的可访问性问题（低悬果实）。

**自动化测试局限**：
- 无法检测语义正确性（如 `aria-label` 有无但内容是否合理）。
- 无法评估键盘流程的合理性。
- 无法感知屏幕阅读器的实际朗读体验。
- 无法发现用户体验层面的问题（如焦点顺序虽合规但不自然）。
- 误报和漏报率较高，需人工审核。

**工具能力对比**：
- **axe-core**：规则引擎最成熟，支持自定义规则，集成能力最强（CI 插件、浏览器扩展、测试框架）。适合核心规则检查。
- **Lighthouse**：面向性能审计，可访问性检查是附加模块。适合开发自检和 CI 概览。
- **Pa11y**：CI 友好，支持批量 URL 扫描和结果快照对比。适合全站扫描和趋势监控。
- 策略建议：axe-core 做深度规则检查，Lighthouse 做快速开发反馈，Pa11y 做全站回归扫描。

**可访问性测试金字塔**：
1. **单元/组件级**：jest-axe 或 testing-library 的 `toBeAccessible` 对单个组件进行规则检查。每条 PR 必跑。
2. **集成/页面级**：axe-core 在 E2E（Playwright/Cypress）中对完整页面扫描。每日构建运行。
3. **手动测试**：键盘全站操作 + 屏幕阅读器验证。每个迭代（sprint）至少覆盖一次核心用户流程。
4. **用户测试**：邀请真实障碍用户参与。每季度或重大改版前进行。

**CI/CD 集成**：
- 设置两条门禁：自动化检查失败则阻塞 PR；手动测试结果需在发布前确认。
- 使用 axe 的 `violations` 阈值逐步收紧，避免一次性严格准入导致团队抵触。
- 记录无障碍债务，与产品 backlog 关联，逐渐修复历史遗留问题。

**手动测试频率与清单**：
- 频率：核心流程每次迭代，非核心流程每两个迭代。
- 清单示例：键盘完整遍历、屏幕阅读器朗读验证、缩放 200% 无断行、高对比度模式、关闭样式后内容可读。

**评分维度**：
- 自动化优劣势分析（20%）
- 工具对比与选择（25%）
- 测试金字塔设计（30%）
- CI/CD 集成策略（25%）

**常见错误**：
- 完全依赖自动化测试，认为通过 axe 就代表无障碍达标。
- 测试金字塔扁平化，只有端到端扫描没有组件级检查。
- 手动测试频率过低或没有固定清单，导致流于形式。

**扩展追问**：
- 如何量化可访问性债务并推动管理层关注修复？
- 在 Storybook 中集成 a11y 测试的最佳实践是什么？
- 如何培训非专业 QA 人员完成有效的手动可访问性测试？

### 19. 解释 APCA（Accessible Perceptual Contrast Algorithm）与 WCAG 2.x 对比度算法的区别。为什么社区在推动 APCA？

**参考答案**：

**WCAG 2.x 对比度算法的问题**：
- 基于简单的相对亮度比公式：`(L1 + 0.05) / (L2 + 0.05)`，结果在 1:1 到 21:1 之间。
- 对 4.5:1（正常文本）和 3:1（大文本）的阈值是"一刀切"，不考虑字体粗细、字号、环境光等感知因素。
- 在深色模式（dark mode）下表现不佳：高饱和度亮色在深色背景上 WCAG 算出来未达标，但实际可读性良好。
- 对细体字和粗体字的可读性差异不作区分。
- 不感知空间频率（笔画宽度、字号对可读性的影响）。

**APCA 的改进**：
- 基于**感知均匀性**（perceptual uniformity）：算法模拟人眼对亮度和对比度的非线性感知。
- 输出值范围约为 0-105，而非简单的比例值（如 4.5:1）。
- 区分**字体粗细**（font weight）和**字号**：粗体大字所需对比度低于细体小字。
- 对**极性敏感**（polarity-aware）：深色文字在浅色背景上的对比感知与浅色文字在深色背景上不同，APCA 能正确定义这两种方向的感知区别。
- 在**深色模式**中表现更合理，高饱和度色不再被错误判定为不可读。

**对暗色模式和大文本的影响**：
- 浅色文字在深色背景上通常需要更高 APCA 值才能达到与传统 WCAG 等效的可读性。
- 大文本（>24px 粗体 或 >36px 常规体）的 APCA 阈值相应降低，不再使用固定的 3:1。
- 设计师可以根据实际的字体权重和大小选择对比度，而不再被单一阈值束缚。

**当前采纳状态**：
- WCAG 3.0（原 Silver）计划采用 APCA 替代 WCAG 2.x 的对比度公式。
- WCAG 3.0 仍在草案阶段，预计还需数年才能成为正式推荐标准。
- 浏览器和工具（如 axe-core、Lighthouse）尚未全面支持 APCA。
- 目前建议团队在 WCAG 2.x 对比度标准基础上，用 APCA 作为补充参考，特别是在深色模式和高饱和度品牌色场景。

**评分维度**：
- WCAG 2.x 对比度算法的局限性（30%）
- APCA 的核心改进（40%）
- 对暗色模式和实践的影响（20%）
- 当前采纳状态（10%）

**常见错误**：
- 误认为 APCA 已是官方标准，立即废弃 WCAG 2.x 检测。
- 不理解极性感知，认为浅色背景和深色背景的对比度公式可以互换。
- 忽略字体粗细和字号对可读性的综合影响。

**扩展追问**：
- 如何用 Sass/Less 函数或 CSS clamp 实现 WCAG 2.x 与 APCA 的双重对比度检测？
- APCA 对品牌色保持又不违反无障碍要求的设计策略有哪些？
- WCAG 3.0 除了 APCA 还有哪些重大变化？

### 20. 在 React 中设计可访问组件时，有哪些关键原则和常用库？

**参考答案**：

**关键原则**：
1. **语义化优先**：使用原生 HTML 元素（`<button>`、`<dialog>`）而非纯 `<div>` 加 ARIA。React 的 JSX 应反映语义结构。
2. **焦点管理**：使用 `useRef` 和 `useEffect` 管理焦点。自定义组件（Modal、Drawer）需实现焦点陷阱和焦点恢复。
3. **ARIA 状态同步**：组件的 ARIA 属性必须与 state 保持同步。使用 `useMemo` 或 `useCallback` 传递动态 ARIA 属性。
4. **键盘交互**：所有交互操作必须可通过键盘完成。使用 `onKeyDown` 处理方向键、Esc、Enter 等。
5. **可访问名称**：每个交互元素都要通过 `aria-label`、`aria-labelledby` 或可见文本获得可访问名称。

**常用库与模式**：
- **@react-aria（React Aria）**：提供无样式 Hooks 和组件逻辑，覆盖按钮、下拉菜单、对话框、网格等 50+ 组件。内置焦点管理、键盘交互、ARIA 属性，是编写可访问组件的事实标准参考实现。
- **Radix UI**：提供无样式（headless）原语（Primitive），每个组件已内置无障碍实现。支持自定义样式和动画，适合产品级组件库。常用：Dialog、DropdownMenu、Popover、Tabs。
- **Headless UI（Tailwind Labs）**：同样是无样式组件，与 Tailwind CSS 配合良好。组件较少但实现稳定，适合快速搭建。
- **Reach UI**：历史上被 React Aria 吸收，部分组件已迁移。新项目建议直接使用 React Aria 或 Radix。

**Headless UI vs 全量组件库**：
- Headless（React Aria / Radix）：提供行为逻辑和 ARIA，样式完全自定义。适合需要品牌独特视觉的团队。
- 全量组件库（Material UI / Ant Design）：绑定了样式和主题，开箱即用。无障碍水平取决于具体版本，需额外验证。

**React 常见陷阱**：
- **Fragment <> </>**：不产生 DOM 元素，可能导致 ARIA 关系断裂。必要时用 `<div>` 包裹。
- **Portal**：`createPortal` 将组件渲染到 DOM 树的其他位置，焦点管理和 ARIA 父级关系需手动处理。模态框场景确保 focus trap 和 `aria-modal` 正确传递。
- **重新渲染**：不必要的重渲染可能导致 ARIA 属性闪烁，屏幕阅读器重复播报。使用 `React.memo` 或状态隔离优化。
- **Suspense 和懒加载**：懒加载组件插入 DOM 时可能破坏焦点顺序。加载完成后的焦点回位需要额外处理。

**测试工具**：
- **jest-axe**：在 Jest 测试中对渲染后的 DOM 运行 axe-core 规则。适合组件级断言：`expect(container).toHaveNoViolations()`。
- **@testing-library/user-event**：模拟真实键盘和鼠标事件，比 `fireEvent` 更接近用户行为。
- **@testing-library/jest-dom**：提供 `toBeVisible`、`toHaveFocus`、`toHaveAttribute` 等断言，协助验证可访问性状态。
- **storybook addon-a11y**：在 Storybook 中实时显示可访问性违规，帮助开发者在开发早期发现和修复问题。

**评分维度**：
- React a11y 关键原则（25%）
- 常用库对比与适用场景（30%）
- React 特有陷阱分析（25%）
- 测试工具与方法（20%）

**常见错误**：
- 完全依赖组件库的无障碍实现，未做手工验证。
- 使用 `<div onClick={...} role="button">` 替代 `<button>`，丢失默认键盘行为。
- Portal 组件中忘记将 `aria-labelledby` 和 `aria-describedby` 的 ID 引用放在 Portal 内部。
- 测试中只过 axe 规则，未模拟屏幕阅读器实际朗读流程。
