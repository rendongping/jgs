# A12 前端数据工程

> 目标：掌握前端数据埋点、AB 实验、指标体系和数据看板架构，用数据驱动产品和技术决策。

---

## 核心要点（TL;DR）

- 前端数据工程是连接用户行为与业务决策的桥梁。
- 埋点体系需要标准化、可治理、可验证。
- AB 实验是验证产品假设的科学方法，需要正确的分流和指标设计。
- 指标体系应分层：业务指标、产品指标、技术指标。
- 数据看板需要实时性、准确性和可解释性。

---

## 1. 数据埋点体系

### 1.1 埋点类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 点击埋点 | 用户点击元素 | 按钮点击、链接点击 |
| 曝光埋点 | 元素进入可视区 | 广告曝光、商品曝光 |
| 页面埋点 | 页面访问 | PV、UV、停留时长 |
| 自定义事件 | 业务特定行为 | 加入购物车、提交订单 |

### 1.2 埋点设计

- 事件模型：Who、When、Where、What、How。
- 属性标准化：用户 ID、设备、页面、模块、元素、时间戳。
- 埋点规范文档化，避免口径不一致。

### 1.3 埋点 SDK

- 自动埋点 vs 手动埋点。
- 全埋点：自动采集所有交互。
- 代码埋点：精确控制，适合核心业务。
- 可视化埋点：运营通过工具配置。

## 2. AB 实验

### 2.1 实验流程

```
提出假设 → 设计实验 → 分流 → 运行 → 分析 → 决策
```

### 2.2 分流策略

- 用户级分流：保证同一用户始终看到同一版本。
- 设备级分流：基于设备 ID。
- 会话级分流：每次会话可能不同。

### 2.3 指标设计

- 核心指标（OEC）：直接衡量实验目标。
- 护栏指标：防止副作用，如错误率、加载时长。
- 样本量和实验时长计算：基于统计功效。

### 2.4 常见陷阱

- 样本污染：对照组受实验组影响。
- 多重比较：同时跑太多实验导致假阳性。
- 实验时间不足：未收集足够样本。

---

## 3. 指标体系

### 3.1 分层指标

- **业务指标**：GMV、转化率、留存率、LTV。
- **产品指标**：功能使用率、任务完成率、NPS。
- **技术指标**：加载时间、错误率、可用性、CLS。

### 3.2 北极星指标

- 最能反映产品核心价值的单一指标。
- 所有优化都应围绕北极星指标展开。

---

## 4. 数据看板

### 4.1 看板架构

```
数据采集 → 数据清洗 → 数据仓库 → 指标计算 → 可视化展示
```

### 4.2 实时 vs 离线

- 实时：Kafka + Flink + 实时数据库。
- 离线：Hive/Spark + 数据仓库。

### 4.3 可视化设计

- 选择合适的图表类型。
- 突出关键指标。
- 支持下钻和筛选。

---

## 5. 最佳实践

- 埋点先验收，后上线。
- 指标口径统一，避免不同团队各说各话。
- AB 实验要有对照组和随机分流。
- 数据看板要服务于决策，不是越多越好。

---

## 常见误区

| 误区 | 正确理解 |
|------|---------|
| 埋点越多越好 | 埋点要围绕核心指标，过多会增加治理成本 |
| AB 实验只看核心指标 | 还要关注护栏指标和长期影响 |
| 数据看板等于监控 | 看板面向决策，监控面向告警 |
| 忽略数据质量 | 脏数据会导致错误决策 |
