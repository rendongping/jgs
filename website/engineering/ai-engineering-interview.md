## 一、基础题

<InterviewCard id="ai-engineering-interview-q1" title="1. 什么是 Prompt Engineering？有哪些常用的 Prompt 模式？" difficulty="🟢" questionHtml="" answerHtml="&lt;p&gt;&lt;strong&gt;参考答案：&lt;/strong&gt;&lt;/p&gt;&#10;&lt;p&gt;Prompt Engineering 是设计和优化输入文本，让 LLM 产生更准确、可控、结构化输出的技术。核心目标是降低模型幻觉、提升回答一致性、适配业务场景。&lt;/p&gt;&#10;&lt;p&gt;常用模式：&lt;/p&gt;&#10;&lt;ol&gt;&#10;&lt;li&gt;&lt;strong&gt;Zero-shot&lt;/strong&gt;：直接提问，不加示例。适合模型已有充分知识的简单任务。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;Few-shot&lt;/strong&gt;：给 1-3 个输入/输出示例，引导模型遵循特定格式或风格。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;Chain-of-Thought（CoT）&lt;/strong&gt;：要求模型“一步步思考”，提升推理和数学能力，如“请解释你的推理过程”。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;ReAct（Reasoning + Acting）&lt;/strong&gt;：让模型交替进行推理和行动，适合需要调用工具或外部知识的 Agent 场景。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;Structured Output&lt;/strong&gt;：强制模型返回 JSON、Markdown 表格等结构化格式，便于前端解析。&lt;/li&gt;&#10;&lt;/ol&gt;&#10;&lt;p&gt;写好 Prompt 的原则：角色设定清晰、任务具体、上下文充分、输出格式明确、给出示例和约束条件。&lt;/p&gt;&#10;&lt;p&gt;&lt;strong&gt;评分维度：&lt;/strong&gt;&lt;/p&gt;&#10;&lt;ul&gt;&#10;&lt;li&gt;概念准确（30%）&lt;/li&gt;&#10;&lt;li&gt;模式举例完整且能说明差异（35%）&lt;/li&gt;&#10;&lt;li&gt;能写出角色/格式/示例等 Prompt 要素（20%）&lt;/li&gt;&#10;&lt;li&gt;有实际调优经验（15%）&lt;/li&gt;&#10;&lt;/ul&gt;&#10;&lt;hr&gt;&#10;" />

<InterviewCard id="ai-engineering-interview-q4" title="4. Function Calling 是什么？前端在其中扮演什么角色？" difficulty="🟢" questionHtml="" answerHtml="&lt;p&gt;&lt;strong&gt;参考答案：&lt;/strong&gt;&lt;/p&gt;&#10;&lt;p&gt;Function Calling（也称 Tool Calling）是 LLM 根据用户输入判断需要调用外部函数，并返回函数名和参数的机-制。它是 Agent 和复杂 AI 应用的核心能力，架起了自然语言与结构化动作之间的桥梁。&lt;/p&gt;&#10;&lt;p&gt;完整流程：用户提问 → LLM 解析意图 → 返回 &lt;code&gt;function_call&lt;/code&gt; → 后端/前端执行函数 → 结果回传 LLM → LLM 生成自然语言回答 → 前端展示。&lt;/p&gt;&#10;&lt;p&gt;前端角色：&lt;/p&gt;&#10;&lt;ol&gt;&#10;&lt;li&gt;&lt;strong&gt;工具定义与注册&lt;/strong&gt;：与后端对齐工具 schema，前端渲染可执行工具卡片（如天气、地图、图表）。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;状态展示&lt;/strong&gt;：展示“正在查询天气…”“正在生成图表…”等中间状态，降低用户焦虑。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;结果渲染&lt;/strong&gt;：根据工具返回数据渲染富媒体组件（ECharts、表格、卡片）。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;用户确认&lt;/strong&gt;：对写操作（如下单、发送邮件）设计确认流程，防止误操作。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;错误处理&lt;/strong&gt;：工具调用失败时展示降级信息。&lt;/li&gt;&#10;&lt;/ol&gt;&#10;&lt;p&gt;&lt;strong&gt;评分维度：&lt;/strong&gt;&lt;/p&gt;&#10;&lt;ul&gt;&#10;&lt;li&gt;能解释 Function Calling 原理（30%）&lt;/li&gt;&#10;&lt;li&gt;能说清前后端职责（25%）&lt;/li&gt;&#10;&lt;li&gt;有交互设计意识（25%）&lt;/li&gt;&#10;&lt;li&gt;有实战经验（20%）&lt;/li&gt;&#10;&lt;/ul&gt;&#10;&lt;hr&gt;&#10;" />

<InterviewCard id="ai-engineering-interview-q5" title="5. 生产环境中，前端为什么不建议直接调用大模型 API？" difficulty="🟢" questionHtml="" answerHtml="&lt;p&gt;&lt;strong&gt;参考答案：&lt;/strong&gt;&lt;/p&gt;&#10;&lt;p&gt;生产环境中，前端应通过后端或 Edge Function 代理调用大模型，而不是直接调用。原因包括：&lt;/p&gt;&#10;&lt;ol&gt;&#10;&lt;li&gt;&lt;strong&gt;API Key 暴露风险&lt;/strong&gt;：前端代码无法安全保存 API Key，容易被逆向获取。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;成本控制&lt;/strong&gt;：用户可能滥用接口导致费用激增，后端可以统一计费、限流、配额管理。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;RAG 与私有知识&lt;/strong&gt;：私有数据需要后端检索和拼接，不能直接暴露给前端。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;Prompt 注入风险&lt;/strong&gt;：系统 Prompt 在前端暴露后，攻击者可构造恶意输入绕过安全限制。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;无法统一审计&lt;/strong&gt;：后端便于记录调用日志、合规审计和异常检测。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;结果后处理&lt;/strong&gt;：后端可以对生成结果做安全过滤、格式校验和敏感词检测。&lt;/li&gt;&#10;&lt;/ol&gt;&#10;&lt;p&gt;推荐架构：前端 → BFF/Edge Function → LLM Provider，后端负责鉴权、RAG、工具调用、成本控制和安全过滤。&lt;/p&gt;&#10;&lt;p&gt;&lt;strong&gt;评分维度：&lt;/strong&gt;&lt;/p&gt;&#10;&lt;ul&gt;&#10;&lt;li&gt;安全风险分析到位（30%）&lt;/li&gt;&#10;&lt;li&gt;架构理解清晰（30%）&lt;/li&gt;&#10;&lt;li&gt;能提出替代方案（20%）&lt;/li&gt;&#10;&lt;li&gt;有生产环境经验（20%）&lt;/li&gt;&#10;&lt;/ul&gt;&#10;&lt;hr&gt;&#10;" />

## 三、高级题

<InterviewCard id="ai-engineering-interview-q9" title="9. 如何设计一个支持多模型切换的 AI 应用层？" difficulty="🔴" questionHtml="" answerHtml="&lt;p&gt;&lt;strong&gt;参考答案：&lt;/strong&gt;&lt;/p&gt;&#10;&lt;p&gt;多模型切换的核心是“统一抽象 + 适配器模式”。&lt;/p&gt;&#10;&lt;p&gt;设计要点：&lt;/p&gt;&#10;&lt;ol&gt;&#10;&lt;li&gt;&lt;strong&gt;统一接口抽象&lt;/strong&gt;：定义统一的 &lt;code&gt;chat(messages, options)&lt;/code&gt;、&lt;code&gt;embed(text)&lt;/code&gt;、&lt;code&gt;streamChat(messages)&lt;/code&gt; 接口，屏蔽不同模型差异。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;Provider 适配器&lt;/strong&gt;：为 OpenAI、Anthropic、Azure、文心、通义等模型分别实现适配器，处理请求格式、认证、响应解析。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;模型路由&lt;/strong&gt;：根据任务类型（聊天、摘要、代码、嵌入）自动选择模型，或允许用户手动切换。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;Fallback 机制&lt;/strong&gt;：主模型失败时自动降级到备用模型，保证可用性。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;成本与性能监控&lt;/strong&gt;：记录不同模型的延迟、Token 消耗、成功率，支持数据驱动选型。&lt;/li&gt;&#10;&lt;li&gt;&lt;strong&gt;前端配置化&lt;/strong&gt;：模型列表、参数（temperature、maxTokens）通过配置下发，无需发版即可接入新模型。&lt;/li&gt;&#10;&lt;/ol&gt;&#10;&lt;pre&gt;&lt;code class=&quot;language-ts&quot;&gt;interface LLMProvider {&#10;  chat(messages: Message[], options: ChatOptions): Promise&amp;lt;string&amp;gt;;&#10;  streamChat(messages: Message[], options: ChatOptions): AsyncIterable&amp;lt;string&amp;gt;;&#10;}&#10;&lt;/code&gt;&lt;/pre&gt;&#10;&lt;p&gt;&lt;strong&gt;评分维度：&lt;/strong&gt;&lt;/p&gt;&#10;&lt;ul&gt;&#10;&lt;li&gt;抽象设计合理（30%）&lt;/li&gt;&#10;&lt;li&gt;能说明适配器和路由机制（30%）&lt;/li&gt;&#10;&lt;li&gt;考虑降级和监控（25%）&lt;/li&gt;&#10;&lt;li&gt;有架构落地经验（15%）&lt;/li&gt;&#10;&lt;/ul&gt;&#10;&lt;hr&gt;&#10;" />
