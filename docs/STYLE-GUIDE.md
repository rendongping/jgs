# 前端架构师知识库 — 文档写作规范

> 本文档规定本知识库所有 Markdown 文件的写作标准，确保风格一致、内容易读、便于维护。

---

## 一、文档目标读者

本知识库面向**初级到高级前端工程师**，最终目标是培养前端架构师。因此文档写作应兼顾：

- **初级学员**：能用通俗语言理解概念，有代码示例和类比。
- **中高级学员**：能获取原理深度、选型思路和实战细节。
- **自学者**：能按文档独立完成练习和面试准备。

---

## 二、每篇学习文档的标准结构

每篇 `01-学习文档.md` 建议按以下结构组织：

```markdown
# 领域标题：一句话定义

> 学习目标：读完本文后，你能……

## 核心要点（TL;DR）
- 要点 1
- 要点 2
- 要点 3

## 学习时长与前置知识
- **建议学习时长**：X-X 周
- **前置知识**：列出依赖的领域编号与基础知识

## 一、为什么需要这个领域？
## 二、核心概念与原理
## 三、常见方案与对比
## 四、实战案例
## 五、常见误区与反模式
## 六、最佳实践
## 七、相关领域
- [F01 JavaScript](../foundation/javascript)
- [A03 Performance](../architecture/performance)

## 八、总结
## 九、延伸阅读与资源

> **最后更新**：YYYY-MM-DD
```

### 结构说明

| 章节 | 必备 | 说明 |
|------|------|------|
| 一句话定义 | 是 | 帮助读者快速判断这篇文档讲什么。 |
| 学习目标 | 是 | 明确学习产出。 |
| 核心要点（TL;DR） | 是 | 3-7 条，方便复习和面试前快速回顾。 |
| 学习时长与前置知识 | 是 | 便于学习者规划时间，明确依赖。 |
| 为什么需要 | 是 | 回答“为什么学这个”，建立学习动机。 |
| 核心概念与原理 | 是 | 系统讲解知识点。 |
| 常见方案与对比 | 推荐 | 技术选型必须对比。 |
| 实战案例 | 推荐 | 用真实或贴近真实的场景说明。 |
| 常见误区与反模式 | 推荐 | 告诉读者“不要怎么做”。 |
| 最佳实践 | 推荐 | 可直接落地的经验。 |
| 相关领域 | 推荐 | 至少链接 3 个相关领域，构建知识网络。 |
| 总结 | 是 | 归纳全文。 |
| 延伸阅读 | 是 | 链接到书籍、文章、官方文档、源码。 |
| 最后更新时间 | 是 | 标注 `> **最后更新**：YYYY-MM-DD`。 |

## 四、语言表达规范

### 4.1 通俗易懂

- 优先使用口语化表达，避免过度学术化。
- 鼓励使用**生活化比喻**，但要确保比喻准确、不误导。

**好的比喻**：
> 事件循环像一家餐厅的服务流程：厨师（调用栈）做菜，顾客点单（任务队列），服务员按顺序上菜。

**不好的比喻**：
> 闭包像黑洞。（过于抽象，没有帮助理解）

### 4.2 避免绝对化表述

避免使用“最好”“一定”“必须”等绝对化词汇，除非在明确的最佳实践中。

| ❌ 不推荐 | ✅ 推荐 |
|----------|---------|
| React 是最好的框架。 | 在需要高度生态和社区支持时，React 通常是稳妥选择。 |
| 你一定要用 TypeScript。 | 对于中大型团队和长期维护项目，TypeScript 能显著降低类型相关 Bug。 |

### 4.3 中英文混排

- 中文与英文、数字之间保留一个空格。
- 英文专有名词首字母大写，如 React、Vue、Webpack。
- 首次出现缩写时给出全称，如 前端架构师能力自评表（Frontend Architect Capability Assessment，FACA）。

## 五、代码规范

### 5.1 代码块语言标记

统一使用以下语言标记：

| 语言 | 标记 |
|------|------|
| JavaScript | `js` |
| TypeScript | `ts` |
| HTML | `html` |
| CSS | `css` |
| Bash/Shell | `bash` |
| JSON | `json` |
| Markdown | `md` |
| Mermaid | `mermaid` |

### 5.2 代码示例要求

- 示例应可运行或接近可运行，避免明显错误。
- 复杂示例需附带注释说明关键逻辑。
- 避免使用已被废弃的 API。

### 5.3 代码长度控制

- 单个代码块不宜超过 50 行。
- 超过 50 行的代码建议拆分或只保留关键片段。

## 六、标题与层级规范

### 6.1 标题层级

- 一级标题 `#`：文档主标题，每篇文档只有一个。
- 二级标题 `##`：主要章节。
- 三级标题 `###`：小节。
- 四级标题 `####`：细节点。
- **不建议使用五级及以下标题**，如果内容过多，建议拆分子文档。

### 6.2 标题命名

- 标题应简洁明确，避免过长。
- 同一领域的多个文档标题风格应一致。

## 七、图表与视觉化

### 7.1 鼓励使用图表

- 流程图、架构图、对比表能显著提升理解效率。
- 推荐使用 Mermaid 语法绘制图表，便于版本控制。

### 7.2 Mermaid 示例

```mermaid
graph LR
    A[用户输入] --> B[前端应用]
    B --> C[BFF 层]
    C --> D[后端服务]
```

### 7.3 表格使用场景

- 技术方案对比
- 概念对比
- 最佳实践清单
- 错误与正确做法对比

## 十二、标签与知识图谱规范

### 12.1 标签使用目的

为每篇学习文档、练习册、面试题和题库题目标注统一标签，用于：
- AI 学习助手精准推荐
- 学习数据中心分析薄弱点
- 构建机器可读知识图谱

### 12.2 标签来源

标签应从 `docs/tags-taxonomy.md` 中选取，每个领域有预设标签词表。新增标签需同步更新词表。

### 12.3 标签标注位置

**学习文档/练习册/面试题**：在文档末尾增加标签区：

```markdown
---

**标签**：`#html` `#css` `#响应式设计` `#性能优化`
```

**题库 JSON**：使用 `tags` 字段：

```json
{
  "id": "F06-001",
  "tags": ["html", "semantics", "a11y"]
}
```

### 12.4 标签命名规范

- 使用小写英文或中文，优先使用英文
- 多个单词用连字符连接，如 `design-system`
- 避免过于宽泛的标签，如 `frontend`
- 每道题目标签 1-3 个，每篇文档标签 5-10 个

## 十三、开放题与 AI 评分规范

### 13.1 开放题类型

| 类型 | 示例 |
|------|------|
| 方案设计题 | "设计一个支持百万级并发的实时消息系统" |
| 案例分析题 | "分析某电商首页性能瓶颈并给出优化方案" |
| 选型决策题 | "为团队选择构建工具，输出 ADR" |
| 故障复盘题 | "某项目上线后出现大量白屏，如何排查和复盘" |

### 13.2 开放题结构

```markdown
### 设计题 1：设计一个 XXX 系统

**场景**：...
**要求**：...
**参考答案要点**：
1. ...
2. ...
3. ...

**AI 评分维度**：
- 架构完整性（25%）
- 技术选型合理性（25%）
- 风险与可扩展性（25%）
- 表达清晰度（25%）
```

### 13.3 AI 评分提示

开放题不设置唯一标准答案，由 AI 根据评分维度对学习者回答进行点评。评分时重点关注：
- 是否覆盖关键架构决策点
- 是否有明确的 trade-off 分析
- 是否考虑可维护性和可观测性
- 是否能用业务语言解释技术方案

## 十四、网站组件无障碍（a11y）规范

本知识库的网站面向广泛的前端学习者，自定义 Vue 组件应遵循基本的无障碍实践：

### 12.1 交互元素

- 所有可点击元素使用原生 `<button>` 或附带 `role="button"`、`tabindex="0"` 及键盘事件处理。
- 按钮需具备清晰的 `aria-label` 或可见文本说明。
- 切换类按钮使用 `aria-pressed` 表示状态。
- 展开/折叠内容使用 `aria-expanded` 和 `aria-controls` 关联控制目标。

### 12.2 焦点管理

- 自定义组件需支持键盘聚焦，焦点顺序符合视觉顺序。
- 为交互元素添加 `:focus-visible` 样式，确保键盘用户可见。

### 12.3 语义化

- 使用语义化 HTML 标签（如 `table`、`nav`、`main`、`section`）。
- 为复杂组件（如面试卡片）添加 `role` 和 `aria-label`，便于屏幕阅读器识别。

### 12.4 颜色对比

- 文字与背景对比度至少满足 WCAG 2.1 AA 标准（4.5:1）。
- 不要仅用颜色传达状态，应配合图标或文字。
