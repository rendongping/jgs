# L08 技术治理与变革管理

> 目标：掌握技术治理、标准制定、技术债管理和组织变革推动的方法，确保技术战略有效落地。

---

## 核心要点（TL;DR）

- 技术治理是在创新和规范之间找到平衡，让团队高效协作。
- 技术标准是治理的载体：代码规范、架构原则、技术选型、安全基线。
- 技术债不可避免，但必须可视、可控、可偿还。
- 变革管理需要理解人、建立共识、小步快跑、持续反馈。
- 技术治理不是控制，而是赋能。

---

## 1. 技术治理体系

### 1.1 治理范围

| 领域 | 内容 |
|------|------|
| 架构治理 | 架构评审、技术选型、技术雷达 |
| 代码治理 | 规范、Review、质量标准 |
| 安全治理 | 安全基线、审计、SDL |
| 数据治理 | 埋点规范、数据质量、隐私合规 |
| 运维治理 | SLO、发布规范、值班制度 |

### 1.2 治理组织

- **技术委员会**：决策重大技术方向和标准。
- **架构评审委员会（ARB）**：评审重大架构变更。
- **SIG（Special Interest Group）**：按领域深入治理。
- **治理 Owner**：每项标准都有明确负责人。

## 2. 技术标准制定

### 2.1 标准生命周期

```
提案 → 评审 → 试行 → 正式发布 → 监督执行 → 定期复审
```

### 2.2 标准文档示例

```markdown
# React 组件开发规范

## 目的
统一组件开发方式，提升可维护性。

## 适用范围
所有业务线 React 组件。

## 规范内容
1. 组件文件组织
2. Props 命名
3. 状态管理
4. 测试要求

## 例外处理
... 

## Owner
前端架构组
```

## 3. 技术债管理

### 3.1 技术债分类

| 类型 | 说明 |
|------|------|
| 代码债 | 坏味道、重复、遗留代码 |
| 架构债 | 耦合、扩展性差 |
| 测试债 | 覆盖率低 |
| 文档债 | 文档缺失或过时 |
| 工具债 | 老旧工具链 |

### 3.2 技术债登记册

- 记录每项债的影响、成本、优先级、Owner。
- 定期 Review，分配偿还时间。

### 3.3 偿还策略

- 重构周/技术债日。
- 新功能开发时顺带偿还。
- 大版本升级时集中清理。

## 4. 变革管理

### 4.1 变革阻力来源

- 习惯改变。
- 对不确定性的恐惧。
- 利益受损。
- 信息不透明。

### 4.2 变革管理模型

- **Kotter 八步法**：建立紧迫感、组建联盟、愿景、沟通、授权、短期胜利、巩固、制度化。
- **ADKAR**：认知、渴望、知识、能力、强化。

### 4.3 推动技术变革

1. 找到痛点，建立共识。
2. 从小范围试点开始。
3. 展示早期成果。
4. 培养内部拥护者。
5. 建立激励机制。
6. 将新做法融入流程。

---

## 5. 技术治理度量

- 标准 adoption 率。
- 代码规范违规趋势。
- 技术债偿还比例。
- 架构评审覆盖率。
- 安全漏洞修复时效。

## 常见误区

| 误区 | 正确理解 |
|------|---------|
| 治理就是制定规则 | 治理是平衡创新和规范 |
| 技术债必须清零 | 技术债需要可控，不可能清零 |
| 变革靠强制推行 | 变革需要理解和共识 |
| 标准越多越好 | 标准要精简、可执行 |
