# 项目管理与交付面试题

> 模拟真实面试场景，训练对项目规划、敏捷、风险、质量、复盘的思考与表达。

---

## 一、基础题

### 1. 项目管理的五大过程组是什么？

**参考答案**：

启动、规划、执行、监控、收尾。

**评分维度**：
- 完整列出五大过程组（60%）
- 简要说明每个过程组核心任务（40%）

**常见错误**：
- 把过程组错记为"需求、设计、开发、测试、上线"（这些是阶段，不是过程组）。
- 缺少"监控"过程组。

**扩展追问**：
- 哪个过程组在项目中花费的时间最多？
- 启动阶段最重要的输出是什么？

---

### 2. Scrum 中的三个角色是什么？各自职责是什么？

**参考答案**：

Product Owner（负责产品价值和需求优先级）、Scrum Master（负责流程执行和团队守护）、Development Team（负责交付产品增量）。

**评分维度**：
- 角色名称准确（30%）
- 说明职责（40%）
- 角色之间关系（30%）

**常见错误**：
- 把 Scrum Master 当成"团队经理"或"项目经理"。
- 认为 PO 可以命令开发团队必须做什么。
- 认为团队内部分工后不需要全员为 Sprint 目标负责。

**扩展追问**：
- Scrum Master 可以由开发人员兼任吗？利弊是什么？
- 如果团队没有全职 PO，你会怎么弥补？

---

### 3. 什么是 OKR？请举例说明。

**参考答案**：

OKR 是 Objectives and Key Results 的缩写，即目标与关键结果。O 是有野心的目标，KR 是可衡量的关键结果。例如：O: 打造行业领先的前端性能体验；KR1: FCP 降低到 1.5s 以内；KR2: 核心页面 Lighthouse 评分达到 90+。

**评分维度**：
- 定义准确（40%）
- 说明 O 和 KR 的区别（30%）
- 举例说明（30%）

**常见错误**：
- O 写成可衡量的指标（"提升 FCP 到 1.5s"其实是 KR，不是 O）。
- KR 写成不可衡量的模糊描述。
- 混淆 OKR 和 KPI。

**扩展追问**：
- OKR 和 KPI 的核心区别是什么？
- OKR 完成率 100% 是好是坏？为什么？

---

### 4. 什么是 Kanban？和 Scrum 有什么主要区别？

**参考答案**：

Kanban 是一种可视化工作流管理方法，核心是看板和限制在制品。主要区别：Kanban 没有固定迭代周期，可随时添加任务；Scrum 有固定 Sprint，迭代内不新增任务。Kanban 没有规定角色和会议；Scrum 有固定的角色和五类会议。

**评分维度**：
- Kanban 概念（25%）
- 区别说明（50%）
- 适用场景（25%）

**常见错误**：
- 认为 Kanban 就是"贴便利贴的板子"。
- 忽略 WIP Limit 是 Kanban 的核心机制。
- 说 Kanban 比 Scrum 好或不好（应该根据场景选择）。

**扩展追问**：
- 你觉得什么类型的项目适合 Kanban？
- 一个团队可以同时用 Scrum 和 Kanban 吗？

---

### 5. 什么是 WBS？分解的原则是什么？

**参考答案**：

WBS（Work Breakdown Structure）是工作分解结构，将项目分解为可管理的工作包。原则包括：80 小时法则（每个工作包不超过 80 小时）、100% 法则（父级工作等于子级之和）、可交付物导向。

**评分维度**：
- 概念准确（30%）
- 分解原则（40%）
- 举例说明（30%）

**常见错误**：
- 按组织架构（按团队分工）分解，而非按可交付物分解。
- 分解粒度过粗或过细。

**扩展追问**：
- 如果发现一个工作包需要 120 小时，你会怎么处理？
- 前端项目 WBS 和传统软件项目的 WBS 有什么不同？

## 二、进阶题

### 6. 项目进度落后时，你会怎么做？

**参考答案**：

- 分析落后的真实原因。
- 重新评估关键路径和剩余工作。
- 识别可以加资源、并行或砍范围的地方。
- 与干系人沟通调整预期。
- 加强每日跟踪，避免进一步偏离。

**评分维度**：
- 原因分析（25%）
- 关键路径分析（25%）
- 调整方案（25%）
- 沟通与跟踪（25%）

**常见错误**：
- 第一反应是加班赶工，而不是先分析原因。
- 只通知干系人进度落后了，不提供解决方案。
- 面对压力轻易承诺压缩工期，不考虑质量影响。

**扩展追问**：
- 如果管理层坚持原定上线时间，不接受延期，你会怎么处理？
- 进度落后和范围蔓延经常同时发生，你怎么识别和应对？

---

### 7. 如何管理需求变更？

**参考答案**：

- 建立统一的变更申请和评审机制。
- 评估变更对范围、进度、成本、质量的影响。
- 由相关方共同决策是否接受。
- 接受后更新计划和风险登记册。
- 定期同步变更状态。

**评分维度**：
- 流程设计（40%）
- 影响评估（30%）
- 决策与更新（30%）

**常见错误**：
- 把变更是为"天塌下来"，对所有变更都拒绝。
- 口头接受变更但不更新计划，导致后期混乱。
- 变更评审成为形式主义，流于签字。

**扩展追问**：
- 如果是一个非常小但紧急的变更（比如改一个按钮文案），也需要走完整流程吗？
- 你怎么避免因为需求变更导致的团队士气下降？

---

### 8. 如何做项目估算？你觉得哪种方法最靠谱？

**参考答案**：

常见方法：故事点（敏捷团队相对估算）、人天/人时（直接对应成本）、三点估算（乐观+最可能+悲观加权）、类比估算（参考历史项目）、T-Shirt Sizing（早期粗粒度）。没有最"准"的方法，应该根据项目阶段和可用信息组合使用。团队的历史数据是估算准确性的基础。

**评分维度**：
- 方法掌握（30%）
- 方法选择（30%）
- 准确度认知（25%）
- 经验分享（15%）

**常见错误**：
- 认为估算就是"承诺"，把估算和承诺混为一谈。
- 只用一个方法，忽略了不同阶段的适用性。
- 不反思历史估算偏差，每次都重新拍脑袋。

**扩展追问**：
- 估算偏差超过 100% 时怎么办？如何向干系人解释？
- 前端项目和后端项目在估算上的主要差异是什么？

---

### 9. 如何进行风险管理？请用一个实际项目举例。

**参考答案**：

风险管理流程：识别 -> 评估（可能性 x 影响）-> 优先级排序 -> 制定应对策略 -> 监控与更新。应对策略有规避、转移、减轻、接受四种。举例（略）。

**评分维度**：
- 流程完整（30%）
- 评估方法（25%）
- 应对策略（25%）
- 实际案例（20%）

**常见错误**：
- 把风险登记完成当成风险管理结束了（缺少持续监控）。
- 只识别高概率风险，忽略低概率高影响风险。
- 没有指定风险负责人。

**扩展追问**：
- 你如何判断一个风险发生的概率？
- 如果风险管理计划本身变成了"纸上谈兵"，团队不执行，怎么办？

---

### 10. 如何保证前端项目的交付质量？

**参考答案要点**：

1. 明确 DoD（Definition of Done）：包括代码评审、测试、性能基准等。
2. 建立质量门禁：CI 检查、自动化测试、Code Review。
3. 设定性能预算：关键指标的超时告警。
4. 灰度发布：小范围验证后全量。
5. 数据反馈：通过埋点和监控持续观察质量。

**评分维度**：
- 流程保障（30%）
- 自动化建设（25%）
- 质量标准（25%）
- 持续改进（20%）

**常见错误**：
- 把质量等同于"测试"，认为 QA 应该对所有质量问题负责。
- 只看上线前的质量，不看线上运行后的质量。
- 质量门禁形同虚设（比如 CI 失败也能合并代码）。

**扩展追问**：
- 当业务压力要求降低质量标准时，你能做的最低的妥协底线是什么？
- 什么是"好"的 Code Review？如何让 Code Review 不流于形式？

## 三、高级题

### 11. 如何在资源有限的情况下保证项目质量？

**参考答案要点**：

1. 优先保障关键路径和核心功能质量。
2. 建立自动化测试和质量门禁。
3. 用风险驱动测试，重点覆盖高风险场景。
4. 引入代码评审和结对编程。
5. 设定性能预算和安全基线。
6. 灰度发布，逐步放量。

**评分维度**：
- 优先级判断（25%）
- 自动化与门禁（25%）
- 风险驱动（20%）
- 过程改进（15%）
- 发布策略（15%）

**常见错误**：
- 在压力下完全放弃质量控制，承诺"上线后再补"。
- 平均分配测试资源，没有根据风险区分优先级。
- 削减自动化投入，认为"手动测试也能凑合"。

**扩展追问**：
- 你如何在项目初期就让团队关注质量，而不是等到测试阶段才发现问题？
- "上线后再补"这个承诺在你的经验中兑现率如何？

---

### 12. 你如何进行项目复盘？有什么要求？

**参考答案要点**：

1. 回顾目标和关键结果。
2. 对比实际结果，找出差异。
3. 分析成功和失败的原因，用 5 Whys 深入追根。
4. 提取可复用的经验和教训。
5. 制定改进行动项并跟进。
6. 沉淀为团队知识库或 SOP。

**评分维度**：
- 结构完整（20%）
- 根因分析（25%）
- 经验沉淀（25%）
- 行动跟进（20%）
- 安全氛围（10%）

**常见错误**：
- 复盘变成了追责大会，大家不敢说真话。
- 改进措施"一定要加强XX"之类的空话，没有可执行性。
- 复盘记录写完了但从不回头跟踪。
- 复盘频率太低（一年一次）或太高（每个故事都复盘）。

**扩展追问**：
- 你有过"问题反复出现"的经历吗？说明复盘出了什么问题？
- 如果团队拒绝参加复盘，说"浪费时间"，你怎么处理？

---

### 13. 如何协调前/后端之间的依赖关系？

**参考答案要点**：

1. **契约优先**：前后端先约定接口规范（OpenAPI / GraphQL Schema），并行开发。
2. **Mock 先行**：前端使用 Mock Server，不依赖后端开发完成才联调。
3. **尽早联调**：不要等到双方都开发完，尽早开始端到端集成。
4. **契约测试**：用 Contract Test 确保双方对接口理解一致。
5. **定期同步站会**：暴露依赖阻塞。
6. **RACI 清晰**：接口规范谁负责定义、谁负责实现、谁负责验收。

**评分维度**：
- 流程机制（30%）
- 工具方法（25%）
- 沟通协同（25%）
- 风险防范（20%）

**常见错误**：
- 后端的 API 设计完全由后端决定，前端被动接收。
- 前端等待后端开发完成才开始联调（串行依赖）。
- 接口文档写完了不更新，联调时才发现对不上。

**扩展追问**：
- 如果你的团队用 GraphQL，前后端依赖管理有什么不同？
- 当后端说"这部分会改"你们先不要联调，你怎么应对？

### 14. 前端项目如何做 Sprint Planning？

**参考答案要点**：

1. **容量确认**：统计团队可用人天（扣除假期、会议）。
2. **Backlog 梳理**：PO 讲解故事，团队澄清细节。
3. **估算**：用 Planning Poker 或团队熟悉的方式估算故事点。
4. **承诺**：根据历史 Velocity 选择本次 Sprint 可以完成的故事。
5. **依赖对齐**：确认前后端依赖、设计资源是否到位。
6. **DoD 确认**：明确本次 Sprint 的完成标准。

**评分维度**：
- 流程完整（25%）
- 容量管理（20%）
- 依赖处理（20%）
- 承诺机制（20%）
- 前端特殊性（15%）

**常见错误**：
- Planning 会议上才开始看需求（应该先 Backlog Refinement）。
- 靠"感觉"而不是 Velocity 做承诺。
- 前端专属依赖（设计稿、接口、浏览器兼容）未考虑。

**扩展追问**：
- 如果你的团队 Velocity 波动很大，你怎么处理 Sprint Planning？
- 前端开发中如何处理"设计师还在改稿"这种情况？

### 15. 如何将 OKR 落地到前端团队的日常工作中？

**参考答案要点**：

1. **季度 OKR 设定**：从公司/部门 OKR 分解出团队 OKR，全员参与讨论。
2. **OKR 公开透明**：挂在团队看板上，每次站会引用。
3. **月度 OKR 检查**：每月检查 KR 进度，必要时调整。
4. **Sprint 层面对齐**：每个 Sprint 的 Backlog 中至少有一个任务是驱动某条 KR 的。
5. **OKR 不是绩效考核**：明确 OKR 是方向指引，不是评价工具。
6. **足够激进**：O 要有挑战性，70% 完成率是好的。

**评分维度**：
- 层级分解（25%）
- 日常融入（25%）
- 与 Sprint 结合（20%）
- OKR 文化理解（15%）
- 实际案例（15%）

**常见错误**：
- OKR 变成"目标管理"的下发工具，团队被动接受。
- OKR 设定后没有日常跟踪，到季度末才发现偏了。
- 把 OKR 达成率直接与绩效挂钩，导致团队不敢设激进的 O。
- OKR 和日常任务脱节，两个世界互不干扰。

**扩展追问**：
- 如果 OKR 和紧急的线上问题冲突了，你怎么取舍？
- 你怎么判断团队是在"做 OKR"还是在"写 OKR"？
